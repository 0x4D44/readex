## M5 stage plan

The shape of M5 is **iterative**: we don't know how many divergences exist until we run the diff once. So the stage plan is *adaptive*:

- **Stage 1 (scoping)** — Agent surveys Python `trafilatura.extract` default behaviour, designs the harness skeleton, returns a one-pager plan
- **Stage 2 (harness + first run)** — Build the Python expectations generator + Rust diff harness; run on all 51 snapshots; produce divergence triage
- **Stage 3 (triage report)** — Bucket divergences (whitespace, escape, ordering, semantic gap); decide which buckets to fix vs allowlist
- **Stages 4..N (fix loop)** — One agent per bucket, each landing a focused fix; re-run harness after each
- **Stage Final-1** — Promote the harness to a BLOCKER gate (`trafilatura_corpus_diff_gate.rs`); record final pass rate + allowlist
- **Stage Final** — Wrap-up commit + journal closure

Anti-inversion doctrine (carried from M3/M4): **Python source always wins**; agents flag any apparent Python-side bug for supervisor verdict rather than silently aligning Rust to a wrong-looking output.

## Stage 4 — Bold-marker drop fix (landed)

**Files touched:**
- `D:\language\mdrcel\src\lib.rs:652-665` — `extract_to_markdown` now sets `cleaning::Options.formatting = true` when constructing the cascade options. Added a 10-line `// Python's settings.py:133 ...` doc-comment block at the call site explaining the parity claim and citing the Python line of truth.
- `D:\language\mdrcel\Cargo.toml` — version 0.15.2 → 0.15.3 PATCH; CHANGELOG block added.

**Python rend mapping confirmed (xml.py:48):**

```python
HI_FORMATTING = {'#b': '**', '#i': '*', '#u': '__', '#t': '`'}
```

Plus the non-`<hi>` formatting cases (xml.py:258-274):
- `<head rend="hN">` → `# ... ##... ###...` (N hashes)
- `<del>` → `~~text~~`
- `<code>` (inline, no newline) → `` `text` ``
- `<code>` (multiline) → fenced ``` ``` ```text``` ``` ```

All of these were ALREADY correctly implemented in `src/trafilatura/output.rs::replace_element_text` (lines 468-563) — the `hi_formatting` table at lines 86-93 matches Python byte-for-byte. The bug was NOT in the formatter; the bug was upstream: `cleaning::convert_tags` (`src/trafilatura/cleaning.rs:548-572`) is gated on `options.formatting`. When false (the default), it strips `<b>/<strong>/<i>/<em>/<u>/<tt>` via lxml-style `strip_tags`; when true, it rewrites them to `<hi rend="..."/>`. By the time `xmltotxt` ran, the `<hi>` elements had been stripped and there was nothing for the formatter to wrap.

Python hides this branch because `settings.py:133` auto-derives `formatting = formatting or self.format == "markdown"` — a markdown-format caller never has to pass `formatting=True` explicitly. mdrcel's `extract_to_markdown` was constructing a vanilla `cleaning::Options::default()` without applying the markdown-format override. **One-line fix:** `formatting: true,` in the struct literal.

**Suggested Stage 5 first target — whitespace bucket:**

The whitespace bucket has TWO dominant patterns visible in the Stage 4 re-run output:

1. **Trailing-newline emission (~8 fixtures: 8badbcb95530e9c2, 8d5cc5247b273722, 9a1590d0917107a7, a604eb8a03efa82d, aa562fed8195cd92, d71ec714e950bddf, eceb960849e96838, 8638632aa27b2f45-style suffixes).** Rust emits `...text.\n` where Python emits `...text` (no trailing newline). The off-by-one is invariably ONE `\n` at the very end of the document. Likely source: `xmltotxt`'s final `sanitize` (`output.rs:780+` ish) or the join semantics differing at the last-element boundary. Python's `sanitize` doesn't `rstrip('\n')`; ours may be adding one.

2. **Cell-pipe newline collapse (~3 fixtures: 9ec7aaf8edb71ac1, d159708a94e68ab6, e6037cf1c861d089).** Pattern: Rust emits ` | ` mid-cell where Python emits ` |\n`. These are all UK-government-style table rows where a single `<td>` contains multiple paragraphs; Python's `process_element` cell branch (xml.py:344-345 — `returnlist.append(" | ")`) appears to be interacting with the inner-`<p>` newline in a way Rust's port collapses.

**Recommended Stage 5 entry point: trailing-newline fix first.** It's ~8 fixtures vs ~3, and the fix is almost certainly a single `rstrip('\n')` (or its absence) in `xmltotxt`'s post-`sanitize` step. Pattern 2 is more invasive (`process_element` cell branch ordering) and should be Stage 5b or Stage 6.
