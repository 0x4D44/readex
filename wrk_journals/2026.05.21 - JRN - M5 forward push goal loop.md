## Stage 5a — Trailing-newline divergence in `extract_to_markdown`

**Goal:** Fix the ~8 whitespace-only fixtures where Rust emits one stray trailing `\n` Python omits.

**Anti-inversion source-of-truth pass:**

Python's `xmltotxt` (`xml.py:354-363`) does NOT `rstrip` — it returns `unescape(sanitize("".join(returnlist)) or "")`. `sanitize` (`utils.py:303-312`) line-by-line filters but doesn't strip trailing newlines either. So `xmltotxt` itself emits a string that DOES end with `\n` for typical bodies — this matches Rust's output.

The actual stripping happens in the CALLER, `determine_returnstring` at `core.py:95-96`:

```python
if document.commentsbody is not None:
    returnstring = f"{returnstring}\n{xmltotxt(document.commentsbody, options.formatting)}".strip()
```

Crucially, `bare_extraction` at `core.py:287-292` ALWAYS sets `commentsbody = Element("body")` (possibly empty) when `options.comments` is true — which is the default for `extract(output_format='markdown')`. So the strip branch fires unconditionally in the default markdown path. For an empty commentsbody, `xmltotxt` returns `""`, so the formatted string is just `f"{returnstring}\n".strip()`.

## Stage 5b — Cell-pipe newline collapse in `handle_table`

**Goal:** Fix the 4 whitespace-only fixtures where Rust emits ` | ` mid-row where Python emits ` |\n`.

**Anti-inversion source-of-truth pass — root cause was NOT in `process_element`:**

Initial hypothesis (per Stage 5a journal entry) was that the bug lived in `process_element`'s cell branch ordering. Investigation showed `process_element` for a `<cell>` produces ` | ` end-separator both in Python and Rust. The `\n` in Python's output comes from the `<cell>`'s `.tail` whitespace — `\n      ` for the apprentice cell in the HMRC fixture — which Python emits AFTER the closing ` | ` (xml.py:350-351). After `sanitize` (utils.py:303-312) collapses whitespace within each line, the embedded `\n` survives and breaks the row across multiple output lines.

Traced through the pipeline:
1. html5ever preserves the `<td>`'s tail whitespace (verified: raw parse → `td tail=Some("\n      ")`).
2. `tree_cleaning` / `strip_tags_multi(tbody)` preserve the tail (verified post-cleaning).
3. `handle_table`'s descendant walk reaches the `<td>` with tail still intact.
4. `handle_table` calls `set_tail(&new_child_elem, tail(&subelement).as_deref())` — and this is a no-op.

**Why it's a no-op:** `set_tail` (`src/readability/dom.rs:512`) stores the tail as a following-Text-sibling-run, which requires a parent slot. The function early-returns on detached elements (`let Some((parent, idx)) = parent_and_index(elem) else { return; };`). The freshly-minted `<cell>` from `define_cell_type` is orphan until `append_child(&newrow, &new_child_elem)` at line 1213 runs. Python's lxml stores `.tail` on the element itself so the assignment sticks on detached nodes — masking the divergence.

This is the SAME architectural divergence as Stage 3-B Cluster B (`handle_paragraphs` tail-on-orphan). Same fix shape.

**Rust fix (`src/trafilatura/main_extractor.rs:1126-1234`):**

```rust
let new_child_elem = define_cell_type(is_header);
let mut pending_tail: Option<String> = None;

if element_child_count(&subelement) == 0 {
    if let Some(processed_cell) = process_node(&subelement, options) {
        set_element_text(&new_child_elem, element_text(&processed_cell).as_deref());
        pending_tail = tail(&processed_cell);
    }
} else {
    set_element_text(&new_child_elem, element_text(&subelement).as_deref());
    pending_tail = tail(&subelement);
    // ... inner descendants walk ...
}
// ... appended_child appendage gate ...
if has_text || element_child_count(&new_child_elem) > 0 {
    append_child(&newrow, &new_child_elem);
    if let Some(t) = pending_tail.as_deref() {
        set_tail(&new_child_elem, Some(t));
    }
}
```

The set_tail call is deferred until after `append_child` so the cell has a parent slot for the Text-node-sibling to land in.

**Note on leaf-cell tails:** the leaf branch uses `processed_cell.tail` (line 400), where `process_node` (htmlprocessing.py:274) trims the tail. `trim("\n      ")` → empty → None. So leaf cells lose their tail in BOTH languages — the divergence only mattered on non-leaf cells (the ones with inner `<p>` blocks). The regression test asserts the non-leaf invariant on a programmatic two-cell row.

## Stage 6a (2026-05-21) — Wikipedia `[edit]` / `[Bearbeiten]` heading-tail recovery

### Source-truth check up-front

The Stage 5b "next sub-target" note framed the divergence as "Python emits `[edit]`, Rust omits it" — and conjectured a missing `mw-editsection` strip rule. **Premise inverted on inspection.** Python's `xpaths.py` has NO `mw-editsection` rule; `MANUALLY_CLEANED` / `MANUALLY_STRIPPED` (`settings.py:349-429`) don't mention it either. Python is NOT stripping the edit links; it is FAITHFULLY EMITTING them.

A direct Python probe on `9c8f49f04f792f81.html` (Wm Morrisons) confirms:
```
edit count: 32
... "## History\n\n[edit]The company was founded as ..."
```

So the question was: why does mdrcel `extract_to_markdown` SUPPRESS the `[edit]` text that Python keeps?

### Root cause — `_extract` loses element tails on append

Python's `lxml._Element` stores `.tail` as an intrinsic attribute. `result_body.append(elem)` (`main_extractor.py:608`) moves the tail along with the element — by definition.

mdrcel models tails as **sibling Text nodes in the OLD parent** (`dom.rs:404` `tail()`, `dom.rs:512` `set_tail`). `append_child(&result_body, &new_elem)` (`main_extractor.rs:1747` pre-fix) detaches and re-parents the element only; the sibling Text node is left in the source parent's child list. **Result: the outer tail vanishes on every move.**

On Wikipedia headings the structure (after `convert_tags`) is:

```
<head rend="h2">History</head>     ← tail = "[edit]" (after span/ref stripping)
<p>The company was founded ...</p>
```

mdrcel was appending `<head>History</head>` into `result_body` and silently dropping the `[edit]` tail. Markdown output became `## History\n\nThe company ...` — no `[edit]`. The same bug bit every element-with-tail across the corpus, but only became visible on heading-adjacent inline text (Wikipedia editsections are the only fixture where the tail is *content* rather than whitespace).

### Fix — re-attach tail on append, gated by same-tag heuristic

**Locations:**
- `main_extractor.rs:1742-1768` — `_extract`'s `for e in subelems` loop.
- `main_extractor.rs:1580-1592` — `recover_wild_text`'s `for e in subelems` loop.

**Shape:**

```rust
let src_tag = local_name(&e);
if let Some(new_elem) = handle_textelem(&e, &potential_tags, options) {
    let new_tag = local_name(&new_elem);
    let saved_tail = if src_tag == new_tag { tail(&e) } else { None };
    append_child(&result_body, &new_elem);
    if saved_tail.is_some() {
        set_tail(&new_elem, saved_tail.as_deref());
    }
}
```

The **same-tag guard** is the load-bearing detail. Three `handle_textelem` arms WRAP the source element into a freshly-built `<p>` and FOLD the source's tail INTO the wrapper's text:
- `<lb>` arm (`main_extractor.rs:1326-1336`) — `lb`-with-tail → new `<p>` with `text = tail`. Original tail still sitting on the source `<lb>`; re-attaching would double-emit.
- `<ref>` / `<hi>` / `<span>` arm via `handle_formatting` (`main_extractor.rs:2179`) — wraps in `<p>` when parent isn't `FORMATTING_PROTECTED`; `process_node` already swapped tail-to-text.
- `<div>` arm via `handle_other_elements` (`main_extractor.rs:650-660`) — renames `<div>` to `<p>` after `handle_textnode` swap.

In all three, `new_elem`'s local-name differs from `e`'s. The `src_tag == new_tag` guard rejects re-attachment for these. For `<head>` (handle_titles returns the same tag) and `<p>` (handle_paragraphs returns same tag) the tail IS the inline-flow content that should follow the element — re-attach.

This was discovered the hard way: the first version of the patch unconditionally re-attached `tail(&e)` AFTER `handle_textelem`, regressing the Rust 1.82/1.83 blog (heading text duplicated because the anchor-tail inside `<head>` had been folded into the wrapper but the original tail was still on the source `<a>`) and the Companies House address (each `<br>`/<lb> tail-as-text wrapped into `<p>`, then re-emitted as tail). Adding the same-tag guard fixed both.

### Root cause: `deep_clone` loses `.tail`

`handle_table`'s non-leaf cell loop (`main_extractor.rs:1166-1216`) dispatches each `<td>`-descendant through `handle_textelem` and appends via `define_newelem`. For `<code>` children, `handle_textelem` routes through `handle_quotes`'s early-return `handle_code_blocks(elem)` path which calls `deep_clone(elem)`. lxml's `deepcopy` retains `elem.tail` because tail is intrinsic to the element node; rcdom models tail as a sibling Text node in the OLD parent, so `deep_clone` (`dom.rs:1460`) — which clones only `node.children` — loses it. By the time `define_newelem` (`main_extractor.rs:2457`) reads `tail(processed)`, it returns None, and the `, ` separator vanishes from the rendered cell.

### Fix landed (Stage 6d-a)

**Commit `8293978`** — `M5 Stage 6d-a: recover cell-child tails lost by deep_clone in handle_table`. Cargo 0.15.8 → 0.15.9 PATCH.

- Capture `tail(&child)` at the TOP of the inner-descendants loop, BEFORE `handle_textelem` (which detaches `child` via `iter("*")` renames in `handle_code_blocks` / `handle_quotes` / `handle_paragraphs`).
- AFTER `define_newelem`, if the just-appended cell-child has tail=None AND its tag matches the source child's tag AND `trim(saved_tail) != ""`, set the saved tail on it. The `trim`-empty guard mirrors Python's `process_node` tail-trim at `htmlprocessing.py:274` — whitespace-only tails (e.g. the `\n` between adjacent `<p>` siblings in pretty-printed Wikipedia POTD) intentionally do NOT trip the recovery (a first iteration without the trim guard regressed `d71ec714` into `whitespace-only` by emitting a stray newline mid-cell).

#### Attempt that failed (kept for the post-mortem)

First approach: attach the `handle_code_blocks` deep_clone result to a sentinel parent and `set_tail` there, so `tail(processed)` would return Some(", ") in `define_newelem`. Failed because:

1. The sentinel parent goes out of scope when `handle_code_blocks` returns.
2. rcdom's `impl Drop for Node` (`markup5ever_rcdom-0.39.0/lib.rs:268-284`) iteratively `mem::take`s every DESCENDANT's children Vec — so dropping the sentinel drains `renamed.children` (the `.rs` Text node) even though `processed_subchild` still holds a strong Rc to `renamed`. By the time `define_newelem` reads `element_text(renamed)`, the leading-text run is empty.

The cell-level fix sidesteps the rcdom Drop quirk entirely by never relying on a sentinel-parent invariant; it just reads tail off the source (which is still attached at capture time) and writes tail to the new cell-child (which is attached to a permanent parent).

## Stage 6g — ptest XPath absolute-vs-relative fix (Verge + Guardian Spotlight)

Cargo 0.15.13 -> 0.15.14 (PATCH).

### Original Stage 6f nominations vs reality

Stage 6f nominated two fixtures:

1. **Primary FRED `e339ce76`** — hypothesis: `<noscript>` must be cleaned before BODY_XPATH selection.
2. **Secondary Verge `d153da3363ba7cf1`** — hypothesis: byline `<span class="k8dtcj6">` byline tighten / prune-set.

Both hypotheses turned out to be **wrong** under source-truthing. The real defect was a single XPath porting bug that surfaced on **Verge** AND on **Guardian Spotlight `3d00ac8e`**.

### Source-truthing

`main_extractor.py:589` reads
```python
ptest = subtree.xpath('//p//text()')
```
In **lxml** the `//` is always **absolute** (rooted at the owning document) regardless of the calling context node — confirmed by direct experiment:
```python
article.xpath('//p//text()')   # returns every <p>//text() in the whole tree
article.xpath('.//p//text()')  # descendant-only (relative)
```

The original Stage 2 Rust port at `main_extractor.rs:1746-1747` used `".//p//text()"` (relative) under the comment "Use `.//p//text()` (descendant-relative) instead of `//p//text()` (absolute, root-relative) to match the document's `<p>` elements anchored at the subtree's root." — i.e. the porter assumed XPath semantics that lxml does **not** follow.

### Consequence (Verge fixture)

The Verge article body has only ~1 substantive article `<p>` inside the matched BODY_XPATH subtree, so the **relative** ptest sum was below `min_extracted_size * 3 = 750` chars → `potential_tags.insert("div")` fired → `handle_other_elements` no longer discarded the byline div (`<span class="k8dtcj6">Terrence O'Brien</span>`) and the headline div, so both leaked into the output.

Python's absolute `//p//text()` scans the whole document (nav + teasers + footer + article), so the ptest sum is huge → div is NOT added to potential_tags → byline div is discarded as expected. Python's debug log confirms: `discarding element: div Terrence O'Brien` / `discarding element: div Paying influencers...`.

### Rust fix

`main_extractor.rs:1742-1766`: walk parents up from `subtree` to the topmost ancestor (Document node), then evaluate `.//p//text()` from there. Equivalent observable behaviour to Python's `//p//text()` absolute semantics.

### Stage 6i-a — orphan tail-Text on append to result_body

**Commit `<pending>`** — `M5 Stage 6i-a: clear old-parent tail when moving element to result_body`. Cargo 0.15.16 -> 0.15.17 PATCH.

#### Python source-truth

lxml's `.tail` is intrinsic to the element node (stored on `_Element` itself, not as a sibling). `result_body.append(el)` therefore moves the tail along atomically with the element. No special handling needed in Python.

#### mdrcel issue

rcdom models tail as following-sibling `Text` nodes in the parent's `children` `Vec`. `append_child` calls `remove(child)`, which only removes the element node from the parent's children — the following tail-Text siblings stay behind. The existing Stage 6a code at `main_extractor.rs:1873-1881` already detected the same-tag case (`<head>` / `<p>` returning unchanged from handle_textelem) and re-attached the tail at the NEW location via `set_tail(&new_elem, ...)`. But it never CLEARED the tail at the OLD location. Result: the original tail-Text node remained orphaned inside whatever ancestor was still in result_body (typically a div that later got `strip_tags(result_body, 'div')`-flattened). The flatten then bubbled the orphan up into an unrelated ancestor's child list, fusing two pieces of content.

Surface manifestation on `3d00ac8e.html`: `<a>Learn more about Guardian Labs content</a>` lives inside `<details>` → `<div>` → `<div>` → next to the inner `<p>Paid content is paid for...</p>`. After `_extract`'s `strip_tags(subtree, 'a')` / convert_tags(a→ref) / `strip_tags(subtree, 'ref')` chain, "Learn more..." became a Text sibling of the inner `<p>` inside `<div class="dcr-9go7jt">`. When `_extract` moved that inner `<p>` to result_body, the tail-Text stayed in `dcr-9go7jt`. But `dcr-9go7jt` is a DESCENDANT of the outer `<div class="dcr-w58v3c">`, which the same loop had already renamed to `<p>` and appended to result_body (bringing the full nested div chain along). After `strip_tags(result_body, 'div')` flattened every descendant div in result_body — including the chain inside the outer renamed `<p>` — the orphan "Learn more..." Text bubbled up into the outer `<p>Paid content</p>` directly, producing `<p>Paid contentLearn more about Guardian Labs content</p>`.

#### Fix

Two-line change in `_extract`'s dispatch loop (`main_extractor.rs:~1876`): when `saved_tail.is_some()` (we captured the tail because src_tag == new_tag), call `set_tail(&e, None)` BEFORE `append_child` to drain the orphan tail-Text from the old parent. Then `append_child` + `set_tail(&new_elem, saved_tail.as_deref())` re-attach it at the new location as before.

```rust
let saved_tail = if src_tag == new_tag { tail(&e) } else { None };
if saved_tail.is_some() {
    set_tail(&e, None);  // <-- the fix: drain orphan tail from old parent
}
append_child(&result_body, &new_elem);
if saved_tail.is_some() {
    set_tail(&new_elem, saved_tail.as_deref());
}
```

The brief hypothesised that Rust's XPath evaluator was missing the `contains(@class, "noprint")` predicate in OVERALL_DISCARD_XPATH. Empirical check:

<p>...coming close</p>
<p><hi rend="#sup">[</hi>. The main difference...combine.</p>
<hi rend="#i">citation needed</hi>]
<p>An important type of web agent...</p>
```

i.e. the leading `[` ends up as the text of a `<hi rend="#sup">` inside a fresh `<p>`, the rest of the original `<p>`'s text becomes that `<hi>`'s tail (`.`), and the `<i>citation needed</i>` survives as a BARE `<hi rend="#i">` sibling at body level (no `<p>` parent) with the closing `]` as ITS tail. Reproducing Python's fracture requires understanding which pipeline stage performs the split (likely `handle_paragraphs` interaction with `handle_formatting` calls on the `<i>` descendant, where `handle_formatting` returns a NEW `<p>` wrapper for orphan formatting elements). This is multi-function, anti-inversion-violating work — replicating Python's structural mangling is conceptually a bug-for-bug replication of an extraction quirk.
