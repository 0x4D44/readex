### Cluster A — RESOLVED, 5/10 fixtures green (commit `8bf2e15`)

**Root cause:** rcdom's `replace_element_tag` (`dom.rs:1361`) DRAINS the
original element's children into a fresh replacement node — opposite to
Python's `subelement.tag = "done"` which is an in-place tag mutation.
The pre-fix code renamed the cell to "done" BEFORE walking its
descendants, so the OLD subelement became childless and the inner walk
yielded nothing.

**Diagnostic methodology (worth repeating):**
1. Build a minimal Wikipedia-infobox shape (`<th>Type</th><td><list><item>...</item></list></td>`).
2. Verify Python's output: `<cell role="head">Type</cell><cell><list/></cell>`.
3. Run the same through Rust: lost the cell-with-list entirely.
4. Instrument by calling `handle_lists` / `handle_textelem` / `handle_table`
   directly with `eprintln!` traces — quickly localised the bug to
   `handle_table`'s outer-rename-before-inner-walk.

**Fix:** defer the rename to the unconditional cleanup at the loop
bottom (`line 1128`, mirroring `main_extractor.py:432`). The inner walk
now sees the original children, and `define_newelem` correctly appends
placeholder markers.

**Pinned regression test:** `handle_table_cell_with_list_preserves_placeholder_marker`
in main_extractor.rs (lib test 580).

**Fixtures unblocked:** Wikipedia Morrison, Apple 10-K, HMRC. Plus BBC
(was already passing, now pinned). example.com pin retained. Total
gate corpus: 5/10.

**Apple_Inc Wiki near-miss (2 tokens diff):** at token 2407, Rust opens
`<list rend="ul">` INSIDE an `<item>`; Python closes the item first then
makes the list a sibling-item-containing-list. Different bug — nested-
list-in-list-item ordering in `handle_lists` or `process_nested_elements`.
Separate cluster; not in tonight's plan.

### Cluster B — RESOLVED, 7/10 fixtures green (commit `6462060`)

**Root cause:** `set_tail` (`dom.rs:512`) stores tails as a following-
Text-sibling-run, which requires the element to have a PARENT — it early-
returns on orphan nodes. `handle_paragraphs` was setting the tail on
`newsub` BEFORE appending to the `processed_element`, so every inline
child's tail was silently dropped.

Python's lxml `newsub.tail = ...` works on detached nodes because lxml
stores tail directly on the element (not as a sibling).

**Diagnosis methodology:** dumped both Python and Rust outputs for the
gov.uk fixture, found the first `<p>` containing `<lb>` markers. The
Python had multi-line address text separated by `<lb/>`; Rust collapsed
every line's text. Made it obvious that `<lb>` tails were being dropped.

**Fix:** capture tail BEFORE the graphic-swap, append, THEN set tail.

**Pinned regression test:** `handle_paragraphs_preserves_lb_tail_text`
(lib test 581) on a 3-line address fragment.

**Fixtures unblocked:** gov.uk PAYE + Fed reserve. Gate corpus 5/10 → 7/10.

**Side-effect insight:** this bug was probably affecting OTHER inline-
element tails too (hi, ref, span). Wherever an inline child's tail
carried text content in a paragraph, that text was lost. The Cluster B
gate-pass count rise suggests the only places we cared structurally were
the `<lb>` cases, but other text content was likely also being silently
lost across all fixtures (text byte-equivalence will surface this if we
add a Stage 3-C text gate later).

### Cluster C — RESOLVED, 9/10 fixtures green (commit `b54e02d`)

**Root cause:** SAME structural problem as A/B — Python's
`elem.tag = "done"` is an in-place tag mutation; our `replace_element_tag`
detaches the OLD node instead. `_extract`'s outer loop was re-processing
orphan elements drained by prior handlers, emitting extra empty `<code>`
placeholders.

Visible on Rust 1.83 release blog: every inline `<code>rustup</code>`
inside a `<p>` was being re-emitted as a top-level empty orphan after
`handle_paragraphs` drained it.

**Fix:** `_extract`'s outer loop now skips detached elements
(parent == None) before dispatch. This is the Rust analogue of Python's
"tag == 'done'" skip via `handle_other_elements`.

**Pinned regression test:** `_extract_skips_detached_orphan_elements_from_cluster_c`
(lib test 582).

**Fixtures unblocked:** de.wikipedia Rust + Rust 1.83 release blog.
Gate corpus 7/10 → 9/10.

**Pattern across all 3 clusters:** `replace_element_tag`'s drain+detach
semantics diverge from Python's in-place rename in subtly different ways.
Each cluster surfaced a different effect of the same root mismatch:
  - Cluster A: drain breaks the inner walk that depends on the OLD
    node's children.
  - Cluster B: orphan node's tail-setter early-returns.
  - Cluster C: detached node re-processed by outer walk.

The right long-term solution might be a true in-place tag mutator (would
need `unsafe` against rcdom's `NodeData::Element::name`). For now the
defensive patches in each handler cover the observed cases.
