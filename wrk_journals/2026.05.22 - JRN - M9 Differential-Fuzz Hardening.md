### 2026-05-22 — Stage-0 follow-up: fix the htmldate char-boundary panic (highest-severity find)
Fixed `custom_parse` Block 1 (`extractors.rs:304`): replaced `string[..4]`/`string[4..8]` byte-slices
with a code-point char-prefix (`string.chars().take(8)`), mirroring Python's `string[:4]` (code-point
slice). For ASCII input it is byte-identical to the prior form (so the 51 gates are unaffected); for
non-ASCII it falls through exactly as Python's `False` `.isdigit()` does, instead of panicking. Block 2
(regex-matched ASCII digits) and Block 3 (named groups) were already safe. Added 2 regression unit
tests (CJK date prefix; short multi-byte string). All 7 `custom_parse` lib tests pass.

Rationale for fixing now (vs deferring to Stage 5): it is a confirmed, root-caused crash that aborts
ALL extraction (every entry point calls `extract_metadata` → `custom_parse`), so leaving it would force
`catch_unwind` plumbing through the Stage-4 diff harness and lose ~9% of pages. A crash is the
highest-severity bug class; clearing it first de-risks every later stage. M8 has landed, so the §10
metadata-coordination carve-out (don't fight a live M8 over date semantics) is moot.

Validation in flight: full markdown + txt corpus gates (51 fixtures each) running to confirm no ASCII
regression. Next: commit; then Stage 1 (shared-helpers refactor).
