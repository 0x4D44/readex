## Three divergence clusters identified

### Cluster A — table cell content stripped (4 fixtures)

**Symptom:** Rust emits `<cell></cell>` immediately closing the row; Python
emits the cell **with placeholder children** (`<list/>`, `<p/>`) inside.

**Concrete example (Wikipedia Apple_Inc, first table):**

```
Python:  <row span="2"><cell role="head"/><cell><list/></cell></row>
Rust:    <row span="2"><cell role="head"/></row>
         <row span="2"><cell/></row>
```

**Root-cause hypothesis (HIGH CONFIDENCE):** Python's `define_newelem`
(`main_extractor.py:154-158`) creates a SubElement with only the
`processed_elem.tag` (plus text/tail) — explicitly NOT the children. So a
`<list>` with `<item>` children returned by `handle_lists` becomes an
EMPTY `<list/>` when appended via `define_newelem`. Python intentionally
keeps these as "this had a list/p here" placeholder markers.

Our Rust `define_newelem` (`main_extractor.rs:2074-2083`) DOES the same —
creates an empty element with just tag/text/tail. So `define_newelem`
itself is faithful.

The bug must be earlier in the chain: the cell-internal walk in
`handle_table` (`main_extractor.rs:1068-1111`) is either (a) producing
None from `handle_lists`/`handle_paragraphs` when Python produces Some,
or (b) not reaching the inner descendants where Python does.

**Next probe:** run a focused diagnostic on a single Wikipedia infobox
row to capture which branch of the inner-walk dispatch fires for the
`<list>` child, and what `handle_lists` returns. The `is_text_element`
gate at `handle_lists` line 311 is the likely difference — its
implementation needs cross-checking.

### Cluster B — `<lb>` line breaks truncated in `<p>` (2 fixtures)

**Symptom:** Rust closes `<p>` after fewer `<lb>` elements than Python.

**Concrete example (gov.uk income tax):**

```
Python:  </lb><lb></lb><lb></lb></p>
Rust:    </lb><lb></lb></p><p></p>
```

**Root-cause hypothesis (MEDIUM CONFIDENCE):** `handle_paragraphs`
(`main_extractor.rs:?`) probably has a stricter "promote text after lb to
new paragraph" rule than Python's. Python keeps `<lb>` elements
appended to the same `<p>`; Rust appears to be cutting off and starting
a new `<p>`. Cross-reference `main_extractor.py:272-360` (Python
`handle_paragraphs`).

### Cluster C — `<code>` block emission (2 fixtures)

**Symptom:** Rust emits `<code>` blocks where Python emits `<p>`, or
vice-versa; the boundary differs.

**Concrete example (Rust 1.83 release blog):**

```
Python:  </code></p><code class="...">k</code><code>...</code>
Rust:    </code></p><code></code><code class="...">k</code><code>...</code>
```

**Root-cause hypothesis (LOW CONFIDENCE — needs investigation):**
`is_code_block_element` / `handle_quotes` / `handle_other_elements`
interaction. Could be a `handle_textelem` dispatch ordering issue or a
`is_code_block_element` false-positive.

## Discovery: `define_newelem` is intentionally lossy

Python's `define_newelem` (line 154-158) is **not** a tree-copy — it's a
PLACEHOLDER marker creator. It captures only the tag, text, and tail of
the processed element. Children are dropped. This means the cells in
the extracted table contain breadcrumbs (`<list/>`, `<p/>`) that mark
"this cell had structure here" but without the actual structured content.

This explains why Trafilatura's "table extraction" output looks sparse
on Wikipedia infoboxes: the cells become marker-only after the
extraction pass. Downstream consumers (e.g., `xmltotxt`) presumably read
the cell as "had-a-list" rather than rendering the list contents.

Our port matches this in `define_newelem` already (was right from
Stage 2c-iii). The bug is in WHEN we call it — the upstream chain in
`handle_table`'s inner walk is dropping items earlier than Python does.
