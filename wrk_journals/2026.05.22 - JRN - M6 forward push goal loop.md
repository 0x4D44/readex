### Stage 1 — pivot to allowlist (anti-inversion finding)

**Status:** Closed — bucket-move only, no code change.
**Outcome:** `507b9cdb` moved from `DEFERRED_KNOWN_DEFECT` → `PYTHON_UNDER_EXTRACT_ALLOWLIST`. Gate accounting: **41 substantive + 8 allowlist + 2 deferred = 51**. Lib tests: 1155 passing (unchanged).

**Diagnosis (M6 Stage 1 investigation agent).** The M5 deferred ADR misdiagnosed the mechanism. It claimed Python's `xmltotxt` has a special `<sup class="reference">` paragraph-break branch and that mdrcel should mirror it (a ~30 LOC port in `output.rs`). Direct verification of Python source disproves this:

- `htmlprocessing.py:402-407` — `convert_tags` walks the tree and renames `<sup>` → `<hi rend="#sup">`, calling `elem.attrib.clear()` FIRST. The `class="reference"` discriminant is destroyed before `xmltotxt` ever sees the element.
- `xml.py:253-351` — `process_element` (the `xmltotxt` body renderer) has NO `<sup>`-class branch. There is nothing in the rendering path that could distinguish `<sup class="reference">` from any other `<sup>`, even if the attribute had survived.
- `main_extractor.py:108-115, 272-351` — the paragraph break is actually a side-effect of `handle_paragraphs` / `handle_formatting` running on inline `<hi rend="#sup">` markers when `formatting=True`. `strip_tags` mutates the live `iter("*")` during `handle_paragraphs`, leaving some `<hi>` descendants un-marked-"done" with their parent renamed. Subsequent `_extract` iteration dispatches them to `handle_formatting`, which wraps each in a fresh `<p>` whose tail becomes the new paragraph's body.

**Smoking gun.** `trafilatura.extract(output_format='txt')` (`formatting=False`) on the same fixture produces inline references matching mdrcel exactly; the paragraph break appears ONLY in `output_format='markdown'`. Python's XML/txt and markdown paths disagree on the same body. This is the textbook anti-inversion case: the break is a `formatting=True` side-effect, not a class-aware rendering choice.

**Prior art.** Same class as the already-allowlisted `86df4d2e` (M5 Stage 6i, `wrk_docs/m5-allowlist/86df4d2e.md`) — the `xmltotxt(body, formatting=True)` re-fracture pattern. `86df4d2e` was triaged correctly at M5 close; `507b9cdb` was misdiagnosed and deferred. M6 Stage 1 corrects the bucket.

### Stage 2 — PBS fix (readability retry-loop in-place mutation; outcome A)

**Status:** Closed — fix landed, fixture green.
**Outcome:** `e1106c5e26712078.html` moved from `DEFERRED_KNOWN_DEFECT` → substantive. Gate accounting: **42 substantive + 8 allowlist + 1 deferred = 51**. Cargo `0.16.0` → `0.16.1` PATCH. Lib tests: 1155 → 1156 (+1 net new). All 5 BLOCKER gates remain green.

**Diagnosis (M6 Stage 2 investigation agent — second consecutive anti-inversion catch).** The M5 deferred ADR claimed PBS was a `BODY_XPATH` content-selection divergence (Python lands on "Latest Stories" sidebar, Rust on article body). Runtime verification disproved this: BOTH engines select the same `<ul>` of Latest Stories. Python's own arm extracts only a 14-char `<p>Latest Stories</p>` heading; Python's **readability arm** is what surfaces the substantive sidebar list.

The real bug was in mdrcel's `readability_fork::Document::summary` retry-loop, which re-parsed `self.html` on every attempt (an M2 Mozilla-Readability flag-sieve port pattern, HLD §m-3) instead of mutating `self.doc` in place across attempts like Python's `readability_lxml.Document.summary` (`readability_lxml.py:124-166`). Re-parsing per attempt meant ruthless flag-strip work on one attempt was discarded before the next attempt's lenient pass — the cascade lost cumulative cleaning effort.

**Fixes (`src/trafilatura/readability_fork.rs`).**
1. `Document::summary`: dropped script/style ONCE outside the while loop; removed per-attempt re-parse so mutations accumulate.
2. Body-fallback path: detach into a fresh `<body>` wrapper (reparent children) so the returned NodeRef survives the rcdom Drop quirk after `try_readability` returns.

**Test changes.** One existing test (`document_summary_falls_back_to_lenient_when_ruthless_fails`) was rewritten to pin Python-faithful semantics — it was previously pinning the WRONG M2 re-parse behaviour. Differential proof against Python recorded in the new docstring. Added one new pin test (`document_summary_mutates_dom_in_place_across_attempts`) to lock the in-place contract.

**Anti-inversion findings.**
- The M5 deferred ADR's claim ("`BODY_XPATH` selector divergence") was structurally wrong — Python and Rust agree on the body selector. The bug was downstream in the readability cascade's retry semantics, not upstream in body selection.
- The test that was rewritten was actively *protecting* the bug. Per CLAUDE.md doctrine ("Tests are scaffolding... A test that costs more to maintain than the confidence it provides should be torn down"), rewriting a test to pin Python-faithful semantics after fixing the bug it was masking is the correct move — but it deserves a doctrinal note: **a passing test can hide a bug if it was written to the wrong contract.** Going forward, any test that pins behaviour without a Python differential proof in its docstring is suspect.

**rcdom shared-ownership note.** This is the 3rd M5/M6 occurrence of the same systemic class: a NodeRef nominally "the tree" that is mutated in place by one consumer while another consumer expects pristine input. (Stage 3 below makes it the 4th.)

**Verification.** Gate at GREEN 51 = 42 substantive + 8 allowlisted + 1 deferred / 51 — PBS now substantive. 1156 lib tests passing.

**Bucket move.** Removed `"e1106c5e26712078.html"` from `DEFERRED_KNOWN_DEFECT`. Deleted `wrk_docs/m5-deferred/e1106c5e.md` (diagnosis was wrong).

### Stage 3 — FRED fix (two-surface bug; outcome A)

**Status:** Closed — fixture green, deferred bucket retired.
**Outcome:** `e339ce76eb1cba73.html` moved from `DEFERRED_KNOWN_DEFECT` → substantive. Gate accounting: **43 substantive + 8 allowlist + 0 deferred = 51**. Cargo `0.16.1` → `0.16.2` PATCH. Lib tests: 1156 (unchanged). All 5 BLOCKER gates remain green.

**Diagnosis (M6 Stage 3 investigation agent).** The M5 deferred ADR diagnosed the divergence as "mdrcel's `justext.core.classify_paragraphs` returns 487 chars where Python's returns 2907" — a classifier-internal defect. M6 Stage 3 anti-inversion verification revealed the diagnosis was *partially* correct but the root cause is **two independent surface bugs upstream of `classify_paragraphs`**, neither of which is the classifier itself.

**Surface bug (1) — threshold mismatch (`src/trafilatura/justext_core.rs`).** Python's `trafilatura.external.try_justext` (`external.py:129-150`) does NOT call jusText with library defaults — it calls `custom_justext` (`external.py:121-126`), a thin wrapper that passes **trafilatura-specific overrides**:

```python
def custom_justext(tree, stoplist):
    paragraphs = ParagraphMaker.make_paragraphs(tree)
    classify_paragraphs(paragraphs, stoplist, 50, 150, 0.1, 0.2, 0.25, True)
    revise_paragraph_classification(paragraphs, 150)
    return paragraphs
```

vs jusText library defaults at `justext/core.py:28-36`: `(70, 200, 0.30, 0.32, 0.2, False)`. mdrcel's `try_justext` was wired to `classify_and_revise` (a convenience wrapper that uses jusText's library defaults — `justext_core.rs:1140`). The override thresholds are dramatically more permissive on stopword density (0.1/0.2 vs 0.30/0.32) — English-narrative paragraphs at 0.10–0.30 stopword density (typical for news / data exposition) were classified as `bad` instead of `neargood`/`good`. **Fix:** new private `custom_justext` helper inside `justext_core.rs` (line 1294-1361) that invokes `classify_paragraphs_with`/`revise_paragraph_classification_with` with the override thresholds verbatim, line-cited to `external.py:121-126`.

**Surface bug (2) — missing `cleaned_tree_backup` (`src/trafilatura/readability_fork.rs::bare_extraction_with_cascade`).** Python (`core.py:280-297`) deep-copies the post-`tree_cleaning` body BEFORE `convert_tags` / `extract_content` mutate it, then hands the copy to `compare_extraction` as its `tree` arg so `justext_rescue(tree, ...)` (`external.py:97`) sees the full un-extracted tree. mdrcel passed the same `body` NodeRef to both `extract_content` and `compare_extraction` — `extract_content` mutates the body in place (strips elements via `strip_tags_multi`, marks `<done>` etc.), so jusText was running on a body already pruned to ~9 paragraphs out of the original 19. **Fix:** `deep_clone(&body)` between `tree_cleaning` and `convert_tags`, capture into `cleaned_body_backup`, pass that to `compare_extraction` as the `tree` arg. This is the **canonical `rcdom` shared-ownership trap** the M5 journal flagged as systemic (3+ M5 occurrences).

**Anti-inversion findings.**
- The M5 deferred ADR's "487 vs 2907 chars" measurement is correct **as a symptom** but misattributes the cause. The classifier in `justext_core.rs` is structurally faithful to `justext/core.py:243-275`; the divergence was entirely in the call shape upstream (wrong thresholds) AND the tree fed into it (wrong tree).
- **Both fixes are required.** Fix (1) alone with the un-backed-up tree gives 1299 chars of partial content (some narrative paragraphs missing because `extract_content` stripped them). Fix (2) alone with the wrong thresholds keeps most narrative paragraphs classified as `bad` and leaves the noscript stub in the output. Together they produce byte-equivalent output to Python (2915 chars).
- **Anti-inversion doctrine reinforced:** the brief said "the classifier itself looks structurally faithful — the divergence is more likely upstream." That hunch was correct. The brief's enumerated possible causes (stoplist staleness, ParagraphMaker boundaries, link-density accounting) were ALL wrong — the actual causes were neither in the classifier nor in its inputs, but in the **call-shape configuration** (thresholds) and **caller's tree-sharing model** (no backup). The M5 deferred ADR was written from inside the classifier's perspective; the real bugs were one layer up.

**Empirical reproduction.**
- Python oracle: `python benchmark/oracles/trafilatura/run.py --markdown benchmark/corpus/snapshots/e339ce76eb1cba73.html` → 2915 chars (M5 ADR cites 2907; the ~8-char drift is between trafilatura point releases, immaterial).
- mdrcel before fix: 5186 chars beginning with "Please enable JavaScript..." (noscript stub from readability arm; jusText override gate kept readability winner because `jt_len=487` was too small).
- mdrcel after fix (1) only: 1299 chars beginning with "An official website..." (right starting paragraph, missing OMO narrative paragraphs that `extract_content` stripped before jusText saw them).
- mdrcel after fixes (1) + (2): 2915 chars, byte-identical to Python.
