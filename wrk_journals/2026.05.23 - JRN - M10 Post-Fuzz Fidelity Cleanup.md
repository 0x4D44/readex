# M10 — Post-Fuzz Fidelity Cleanup

**Started:** 2026-05-23
**Lead:** Claude (team lead) + Arthur (supervisor)

## Why this milestone

M9 closed with the held-out KPI at md 69.80% / txt 75.00% / xml 71.40% — substantially
below what the 51-fixture gates had been implying. The wild-distribution divergence
breakdown (per M9 Stage-5 Round-3 triage):

- **~60%** selection-phase (different paragraph or table chosen as main content;
  100% diff-shape diversity — each one essentially a different page).
- **~10%** known-deferred control chars (the unported `remove_control_characters`).
- **~30%** scattered structural patterns, 1–5 pages each.

M10 attacks the **bounded, known-yield** work first before deciding whether the
selection-phase pile is worth taking on — partly because that decision depends on
what's left after the cheap base hits land, and partly because the metric question
("is byte-for-byte selection equivalence the right success criterion?") is
unresolved.

## Phase plan

1. **Phase 1 — Control-char port** (bounded; known target; ~10% working-slice yield):
   port `remove_control_characters` from vendored Python faithfully; remove any
   now-stale mask from the diff harness; re-baseline.

2. **Phase 2 — Long-tail structural triage** (post-Phase-1; targeted):
   triage post-Phase-1 working-slice divergences; identify patterns with ≥5
   records that are HIGH-confidence one-liners; fix and re-baseline.

3. **Phase 3 — Selection-phase go/no-go decision** (gated on Phase 2 results):
   only if remaining divergences are ≥50% selection-phase AND a specific
   algorithmic difference is localized (not "the algorithm differs somehow") do
   we proceed. Otherwise surface the metric question to Arthur before sinking a
   multi-session refactor.

### 2026-05-23 — Phase 2B tracer outcome: MEDIUM on B, HIGH on a neighbour

Tracer agent (`a74f20b6638a4aac8`) ran empirical traces on 8 G5 / G14
representatives (rep numbers 0, 2, 3, 4, 5, 10, 14, 18). Outcome:

**Hypothesis B (serialiser-level blank-line accounting) FALSIFIED.**
`process_element`, `sanitize`, and `line_processing` are all algorithmically
faithful to Python. The divergence is *upstream* of the serialiser: the DOM
tree shape going INTO process_element differs (Python has more `<p>` siblings;
mdrcel has flatter `<div>` + tail-text-on-inline structure). G5 dissolves into
at least two sub-loci: ~8/19 records where mdrcel has FEWER `<p>` boundaries
than Python; ~8/19 records where mdrcel has MORE blank lines (text-vs-tail
apportionment / markdown-only divergences with byte-equal XML). The original
hypothesis was the wrong framing.

**Sub-finding, HIGH-confidence on a DIFFERENT hypothesis: rescue pass missing
from `transform_misused_divs_into_paragraphs`.**

- **Python source:** `benchmark/oracles/trafilatura/.venv/Lib/site-packages/
  trafilatura/readability_lxml.py:297-325` — the function has TWO passes. The
  first retags `<div>` to `<p>` when no block-tag children present; the
  second (lines 305-325) wraps any `<div>` leading text in a fresh `<p>`
  child, and any child's tail text in a fresh `<p>` sibling, and drops `<br>`
  children.

- **mdrcel state:** `src/trafilatura/readability_fork.rs:801-819` implements
  ONLY the retag pass. Comment at lines 784-790 explicitly omits the rescue
  pass on the justification that it "does not affect the scoring decisions
  the rest of summary() consumes (paragraph scoring already concatenates
  descendant text via text_content)."

- **Justification verifiably wrong.** Python's `score_paragraphs`
  (readability_lxml.py:225-259) iterates `<p>`/`<pre>`/`<td>` descendants.
  The rescue pass CREATES new `<p>` elements. Therefore those new `<p>`s
  become scoring candidates, change `select_best_candidate` inputs, and
  influence which body element is picked. The `text_content` concatenation
  argument is about a different invariant.

- **Caller-order verification:** `readability_lxml.py:138-139` runs
  `transform_misused_divs_into_paragraphs()` then `score_paragraphs()` —
  rescue pass is unambiguously upstream of scoring.

- **Empirical evidence (rep 5 sha `781765a839`):** mdrcel keeps the
  Vasilyev/sup pattern as flat `<div>.text="А.И. Васильев…"`; Python's tree
  shows `<div><p>А.И. Васильев</p><sup> 1</sup><a>…</a><p>, М.М. Березенцев</p>…</div>`
  with rescue-pass `<p>` wrappers visible. Final mdrcel xml has the flat
  layout; final Python xml has the wrapped layout.

- **Estimated yield:** ~5-8 of 19 G5 records + unknown number elsewhere in
  the working slice (probably +10-15 records total ≈ 2-3% working slice,
  ≈ +0.5-1% KPI on md/txt).

- **Blast-radius risk:** non-trivial. Adding `<p>` elements during readability
  CHANGES selection decisions, which could break 51-fixture-corpus gates that
  currently pass because mdrcel makes a particular (different from Python)
  selection. Measurable empirically by running all 6 gates after the fix
  attempt — but a real risk.

**Tracer scaffolding:** `examples/m10_phase2b_trace.rs` (uncommitted; per the
brief, scratch).

**Allowlist candidates this round: NONE.** Mirror finding (no Python idiosyncrasy).
