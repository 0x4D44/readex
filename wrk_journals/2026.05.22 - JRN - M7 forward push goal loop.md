## Charter

**M7 = output-format differential-gate expansion.**

Recon (Plan agent, 2026-05-22) reframed the premise. mdrcel **already implements** all six
trafilatura output formats — `extract_to_markdown / json / csv / xml / tei` in `src/lib.rs`,
serialized by `src/trafilatura/output.rs` (port of Python `xml.py`). txt is the
formatting-off variant of the markdown/`xmltotxt` path.

What is **missing** is corpus-wide BLOCKER **gates** for the non-markdown formats. Today only
`trafilatura_markdown_gate.rs` (markdown) and `trafilatura_extract_content_gate.rs`
(intermediate XML) exist. M7 adds gates for txt / json / csv / xml / xmltei, mirroring the
markdown gate's structure.

So each stage = (1) add a Python oracle mode + (2) clone the gate harness for that format +
(3) run the 51-fixture corpus byte-diff + (4) triage divergences using the established
three-bucket doctrine (substantive / `PYTHON_UNDER_EXTRACT_ALLOWLIST` / `DEFERRED_KNOWN_DEFECT`),
each non-substantive bucket entry backed by an ADR.

### Anti-inversion doctrine (carried from M3-M6 — earned, non-negotiable)
- Python source of truth is **vendored**: `benchmark/oracles/trafilatura/.venv/Lib/site-packages/trafilatura/*.py` (v2.0.0).
- ADRs are starting points, not recipes. Verify every root-cause claim against Python source.
- Allowlist = Python is wrong (mdrcel is right). Deferred = mdrcel is wrong (known defect).
  Fix cheaply when the fix is small and safe; otherwise bucket with an ADR and move on.
- Scope-reduce rather than block. If a format stalls, bucket the divergences and proceed.

### Reserve work (fill-in if a format stalls)
- rcdom shared-ownership audit (tier-1 systemic, 4 M5/M6 occurrences).
- jusText port hardening (generalise FRED's `custom_justext` fix as a differential test).

## Harness anatomy (from recon — what each gate must replicate)
- Corpus: 51 `benchmark/corpus/snapshots/*.html`, hardcoded `FIXTURES` array.
- Oracle: **live** `python run.py --<fmt> <path>` subprocess per fixture (no stored goldens).
- Compare: NFC-normalise both sides, strict byte equality.
- Buckets: untriaged enum + `PYTHON_UNDER_EXTRACT_ALLOWLIST` + `DEFERRED_KNOWN_DEFECT`;
  accounting invariant asserts pass+allowlist+deferred+buckets == total; GREEN when buckets empty.
- New oracle modes go in `benchmark/oracles/trafilatura/run.py` (mirror the `--markdown` branch).

## Known divergence risks flagged by recon
- **json**: `fingerprint`/`id`/`filedate` are `null` in mdrcel (output.rs:1216-1220) → likely allowlist.
- **csv**: header-row delta (mdrcel prepends a header Python's `xmltocsv` omits, lib.rs:830-834) + `\r\n`/quoting dialect.
- **xmltei**: TEI header/namespace/`check_tei` attribute whitelist; mdrcel explicitly does not validate (lib.rs:1064) — most format-specific surface.

### Stage 1 — run.py oracle modes (5 formats) + txt gate (2026-05-22)

**What I did**
- **run.py** — added all five M7 oracle modes (`--txt/--json/--csv/--xml/--xmltei`) up
  front, as the brief mandated (front-loaded so Stages 2-5 don't contend on the shared
  file). They share ONE branch: `_parse_args` now returns a 7th element `extra_format`
  (the `output_format=` string), and a single `if extra_format is not None:` block mirrors
  the `--markdown` branch exactly, changing only the format string. Verified the accepted
  format strings against the vendored `core.py:168`
  (`{"csv","html","json","markdown","txt","xml","xmltei"}`) — `--xmltei` maps to the
  literal `"xmltei"` (NOT `"tei"`). Mutual-exclusion + env sentinel + CLI conventions
  preserved.
- **lib.rs** — added public `extract_to_txt(html, base_url, opts)`. There was NO existing
  txt entry point. The pre-existing doc on `extract_to_markdown` claimed
  `extract_with(..).text` IS the txt path — **that is false**: `extract_with` derives text
  via `text_content + trim` (flat itertext join), which is NOT byte-equivalent to Python's
  `xmltotxt(document.body, formatting=False)` (the latter preserves table `|` separators,
  list items, paragraph breaks). `extract_to_txt` is `extract_to_markdown` with
  `formatting=false` in BOTH places (cleaning `convert_tags` AND `xmltotxt(.., false)`),
  matching `settings.py:133` (`self.formatting = formatting or self.format == "markdown"`
  → false for txt) + `core.py:71-98`.
- **tests/trafilatura_txt_gate.rs** — cloned the markdown gate verbatim (same 51 FIXTURES,
  same three-bucket accounting + invariant assert), swapping the subprocess to `--txt` and
  the mdrcel call to `extract_to_txt`.

**txt gate result: GREEN — 45 substantive + 5 allowlisted + 1 deferred = 51.**

**Divergences + triage (6 total)** — strikingly, all 6 are fixtures already on the
markdown allowlist; the 2 markdown-allowlist entries that are formatting-specific
(86df4d2e `<sup>` fracture, 39ca4af9 empty code fence) correctly PASS on txt. This is the
expected signal that txt ≈ markdown.
- **5 → allowlist** (`wrk_docs/m7-allowlist/`), each cross-referencing its m5 markdown ADR
  because the root cause is **format-independent** (selection / parser / decoding, NOT
  formatting), verified against Python source:
  - `41d2afac` — Python `bare_extraction` returns empty (upstream of the txt/markdown
    branch); mdrcel extracts ~75KB. (empty-vs-non)
  - `0f63a2a5` — HN: Python over-extracts the `pagetop` nav + flat-cell render; mdrcel
    tables + omits nav. (selection/table-walk)
  - `683d5643` — DFIN XBRL: one extra empty table cell from html5ever-vs-lxml tree
    construction (>99.95% identical). (parser)
  - `9c64e8e3` — Rust blog index: Python's `link_density_test_tables` rejects the post-list
    table that IS the content (161 chars vs ~17KB). (selection)
  - `dc8ba3c0` — `&#153;` HTML5 §13.2.5 CP-1252 remap (0x99→U+2122 ™); html5ever
    spec-faithful, lxml strips. (character decoding)
- **1 → deferred** (`wrk_docs/m7-deferred/507b9cdb.md`): **NEW finding, mdrcel is the buggy
  side.** mdrcel leaks **U+2063 INVISIBLE SEPARATOR** (Unicode cat `Cf`) that the source
  HTML literally contains around link text; Python strips it via
  `remove_control_characters` (`utils.py:272-300`: `char.isprintable() or
  char.isspace()`). mdrcel's `output::line_processing` (`output.rs:787`) deliberately
  omitted that step "until a future test surfaces a control-character leak" — THIS gate is
  that test. Confirmed empirically: Python's txt output has ZERO non-printable-non-space
  chars. Deferred (not fixed) because a faithful `isprintable` port needs a Unicode
  general-category facility (`char::is_control()` only covers `Cc`, not `Cf`) — a new
  dependency / vendored table that is a DEC-3 architectural decision warranting supervisor
  sign-off; a hand-rolled approximate `isprintable` would be a fresh divergence surface.
  NOTE: this is INDEPENDENT of the markdown-gate `507b9cdb` allowlist (that one is the
  `<sup>`-fracture under `formatting=True`); same fixture, two distinct findings, no
  conflict.
