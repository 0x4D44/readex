# M12 — Broad Sweep Confidence Test

**Started:** 2026-05-24
**Lead:** Claude (team lead) + Arthur (supervisor)
**Predecessor:** M11 Metric Pivot + Parser Pre-Processor (closed 2026-05-24, commit `a7ed9c5`)

## Why this milestone

M10/M11 brought byte-equivalence to 85.8% and effective parity (Dice ≥ 0.80) to
94.8% on the 1500-page M9 corpus. But the entire quality story rests on 1500
Common Crawl pages. Before mdrcel gains real consumers, we need confidence that
there aren't nasty corner cases lurking at scale — panics, content extraction
failures, or genuinely broken output on the diverse HTML the real web produces.

M12 is a one-off confidence sweep: run mdrcel and Python trafilatura on 50K+
diverse pages, compare every pair, surface surprises.

## Process discipline

- **HLD-first** for each work package; team lead reviews and iterates with
  design agents before signing off implementation.
- **Code-review discipline.** `/code-reviewer` and `/code-review-apply-feedback`
  skills on every implementation diff.
- **Commit policy.** Commits authorised only by team lead.
- **Journal cadence.** Updated after each significant step.

## Work packages

Two implementation packages (parallelisable):

1. **WP1 — Harvester additions** (`benchmark/fuzz/harvest.py`): `--append`,
   `--max-per-domain`, `--loose-filter` flags. ~100 lines Python.
2. **WP2 — Sweep harness** (`tests/fuzz_diff.rs`): `ExtractOutcome::Timeout`
   + 30s deadline in `safe_extract`, `ShapeClass::Timeout`, `broad_sweep()`
   test function. ~80 lines Rust.

Then three sequential pipeline steps (unattended):

3. **Harvest** — 50K+ pages from Common Crawl WARCs (~1-2 hours)
4. **Oracle** — Python trafilatura batch on all pages (~2-3 hours)
5. **Sweep** — `broad_sweep()` on full corpus (~30-60 min)

Finally:

6. **Triage** — analyse outlier report, surface findings.

### 2026-05-24 — Oracle complete, sweep started, BUG FOUND

Oracle batch completed: 52,001 lines (1 header + 52,000 entries), 571 MB.
522 per-page errors caught by try/except (expected — some pages crash
trafilatura's lxml C extension).

Broad sweep started but hit tool 10-minute timeout after processing 2500
of 52K pages. However, during those 2500 pages, the sweep surfaced a
**real bug**:

**Bug: char-boundary panic in `python_html_unescape` on CJK metadata.**

- Location: `src/trafilatura/metadata.rs:1206`
- Root cause: `scan_charref()` named-entity branch (line 1269) uses a
  32-BYTE cap (`max_end = body_start + 32`) but CJK characters are 3 bytes
  each. The cap can land inside a multi-byte char, causing
  `&s[body_start..p]` to panic.
- Trigger: Chinese metadata containing a bare `&` followed by CJK text
  (e.g. `&中神通信息技术...`). The `&` enters the named-entity scan,
  which advances 32 bytes into CJK text and stops mid-character.
- Fix: 3-line insertion after the while loop — back `p` up to the
  previous char boundary when it lands on a UTF-8 continuation byte.
- Regression test added: `unescape_cjk_with_ampersand_does_not_panic`.
- Verified: 1231 lib tests pass (1230 existing + 1 new), build clean.

This is EXACTLY what the broad sweep was designed to find — a real panic
on real-world CJK HTML that our 1500-page (mostly English/European) M9
corpus never triggered.

**Remaining:** full 52K sweep needs ~2-3 hours.

#### Bugs found and fixed

1. ✅ **CJK char-boundary panic** — `metadata.rs:1206`, found in first
   2,500-doc partial run, fixed in commit `49b14bc` (pre-sweep). 32-byte
   cap landed inside multi-byte CJK chars.

2. ✅ **Japanese URL char-boundary panic** — `metadata_url.rs:361`,
   found in full sweep at ~50K mark. `url[..scheme.len()]` slicing
   landed inside `ペ` (bytes 6..9) when metadata contained
   `このページのURL`. Fix: switch to byte-slice comparison
   (`<[u8]>::eq_ignore_ascii_case`); schemes are pure ASCII so the
   byte cap is guaranteed at a char boundary on successful match.
   3 panic events, all on the same shape. Regression test:
   `strip_scheme_handles_non_ascii_url_prefix`. Version bump 0.18.7 →
   0.18.8.

#### Outlier breakdown

- **6,556 outliers** written to `benchmark/broad/report.jsonl`
- **911 empty-vs-nonempty** pages (one engine produced content,
  the other didn't)
- **2 timeouts** — SHAs `0c31d5b7` (22 chars from mdrcel, 192 from

Many of the worst-50 cases are XML envelope `<doc>...</doc>` (~36

(mdrcel emits envelope; oracle returns "" for failed extraction).
Not necessarily bugs.

- **93.9% effective parity** — *higher* than the M9 held-out KPI
- Both bugs found were of the same family (byte-slicing without

- **mdrcel is ready for real consumers.** The broad sweep delivered

  metadata that the M9 English-heavy corpus would never have found.
