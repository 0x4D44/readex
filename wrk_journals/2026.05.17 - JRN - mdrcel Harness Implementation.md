The commit message's "repairs pre-committed Cargo.lock byte-stability" is **factually false** — the outcome (byte-stable) is correct, but not for the stated reason. The commit is **not** amended (CLAUDE.md). **Accurate rationale, recorded here as the authoritative record:** committing `.gitattributes` before any `-text` artifact is **prophylactic for *future* byte-stable artifacts** — the HTML snapshots (Stage 3) and the oracle `contract.schema.json` (Stage 4), which MUST have the `-text` rule in effect *at `git add` time* or `text=auto` will mangle them irreversibly. Cargo.lock was never at risk. The ordering discipline stands; only the Cargo.lock-specific justification was fiction. Future sessions/oracle team: do not cargo-cult "renormalize repaired the lock."

## Open decisions for Arthur (non-blocking; build continues meanwhile)

- **DEC-1 (HLD §8 tokenizer spec — casefold).** Devil's advocate PROVED false-negatives: lowercase+NFC (the literal §8 spec) is NOT Unicode caseless matching — `tokens("ﬁle") != tokens("file")`, MICRO SIGN U+00B5 ≠ Greek mu, ﬀ/ﬃ ligatures split. These occur in our SEC/regulator corpus, so the "English/Western-European v1" scope only partially excuses it; Coverage/Precision get artefactually depressed (measuring-instrument bias). **Mitigation in place:** documented as a known limitation + pinning test `known_limitation_ligature_not_casefolded` (so a future fix is intentional/reviewed). **Decision needed:** change HLD §8 "lowercase, NFC" → **`NFKC_Casefold`** for v1? This touches the shared tokenizer-comparability contract, so it is a spec change, not a Stage-2 patch. **Supervisor recommendation: yes, adopt NFKC_Casefold for v1.** Deferred to Arthur; build proceeds on documented current behaviour.
