# M7 TXT Allowlist ADR — `683d5643b173c7fd.html` (DFIN XBRL 10-K, single empty-cell drift)

**Gate:** `tests/trafilatura_txt_gate.rs` (`--txt` / `output_format="txt"`).
**Bucket pre-allowlist:** `content-mismatch` (Rust = 374,635 chars, Python = 374,473 chars; first-diff at byte 32,324).
**Verdict:** anti-inversion-clean — identical root cause to the markdown gate, format-independent.

## Why this is the same divergence as markdown

At byte ~32,324 in this 8 MB DFIN ActiveDisclosure XBRL 10-K, mdrcel emits one
extra empty table-cell row in the cover-page checkbox area:

```
rust:   …FORM 10-K\n|\n|\n|\n☒ |\nANNUAL REPORT…
python: …FORM 10-K\n|\n|\n☒ |\nANNUAL REPORT…
```

The extra `|\n` is a single empty cell. The cause is html5ever vs lxml tree
construction on XBRL-namespaced (`xbrldi`/`xbrldt`/`xbrli`) input: the two
parsers retain XBRL hidden/non-display cells slightly differently, so the
table-walk sees one more empty cell. This is a **parser/table-walk** effect that
runs identically on the markdown and txt paths — `output_format` only governs
serialization — so the txt output drifts by the same one empty cell.

162-char delta in a 374 KB document (0.043%); >99.95% byte-identical.

## Cross-reference

Full diagnosis + anti-inversion analysis: `wrk_docs/m5-allowlist/683d5643.md`
(markdown gate). This M7 ADR records that the SAME trivial empty-cell drift
manifests on the txt path; no new mechanism.

## Anti-inversion check

Closing this would require either re-implementing html5ever's XBRL tree
construction to match lxml (faithfulness-violating) or post-trimming empty
cells (risking real empty-cell loss on other fixtures). Neither is a clean win;
allowlisting documents the trade-off honestly.
