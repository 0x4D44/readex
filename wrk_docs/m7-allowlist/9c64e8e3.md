# M7 TXT Allowlist ADR — `9c64e8e3fcd844d4.html` (blog.rust-lang.org index, link-density filter)

**Gate:** `tests/trafilatura_txt_gate.rs` (`--txt` / `output_format="txt"`).
**Bucket pre-allowlist:** `content-mismatch` (Rust = 16,979 chars, Python = 161 chars).
**Verdict:** anti-inversion-clean — identical root cause to the markdown gate, format-independent.

## Why this is the same divergence as markdown

Python's `htmlprocessing.link_density_test_tables` rejects the
76.8%-link-density `<table class="post-list">` that IS the page's substantive
content (the post listing). Result: Python emits 161 chars (the page
description only). mdrcel preserves the post listing (~17 KB of titles + URLs +
dates).

This rejection happens in the **content-selection** stage (table filtering),
which runs identically on the markdown and txt paths — the `output_format`
string only governs serialization of the already-selected body. So the txt
output diverges for exactly the same reason the markdown output did.

## Cross-reference

Full diagnosis + anti-inversion analysis: `wrk_docs/m5-allowlist/9c64e8e3.md`
(markdown gate). This M7 ADR records that the SAME Python under-extraction
manifests on the txt path; no new mechanism.

## Anti-inversion check

Matching Python would mean deliberately dropping the post listing that is the
page's content to emit the description alone — inversion (out-cleaning to match
a buggy oracle). Allowlisting is correct.
