# M7 XML Allowlist ADR — `39ca4af9befa0524.html` (`<code>` leading-newline)

**Gate:** `tests/trafilatura_xml_gate.rs` (`--xml` / `output_format="xml"`).
**Bucket pre-allowlist:** `content-mismatch` (Rust = 24,808 chars, Python =
24,826 chars; first-diff at byte ~1,269).
**Verdict:** anti-inversion-clean — HTML5 §13.2.5 leading-newline rule;
html5ever follows the spec, lxml (XML mode) does not. Format-independent.

## Diagnosis

A `<code>`/`<pre>` block in this page begins with a literal newline + space:

```
rust:   <code>cc #build-dependencies\nA build-time dependency…
python: <code>\n cc #build-dependencies\nA build-time dependency…
```

The HTML5 parsing spec (§13.2.5, "A start tag whose tag name is one of: pre,
listing, textarea") mandates that a newline immediately following the start tag
be dropped. mdrcel's html5ever implements this; lxml, parsing the document in
lenient-XML mode, does not, so it retains the leading `\n `.

`<code>` is in `SPACING_PROTECTED` (utils.py:80), so `sanitize_tree` preserves
*internal* whitespace verbatim on both sides — note the internal `\n` after
`dependencies` survives identically in mdrcel's output, proving mdrcel's
`preserve_space` path works. The ONLY difference is the single spec-mandated
leading-newline drop, which happens at parse time, upstream of any output
format. The txt, json, csv, and markdown gates are GREEN on this fixture
(`process_element`'s code handling collapses the leading whitespace on the text
paths, so it never surfaces there).

This is the same family as the allowlisted `dc8ba3c086153274.html`: html5ever
follows the HTML5 spec, lxml diverges, and the spec-compliant side is mdrcel.

## Anti-inversion check

Closing this would require *un*-doing a correct HTML5-spec behaviour to match
lxml's XML-mode laxity — actively wrong. Allowlisting documents that mdrcel is
the spec-conformant side. Cross-reference: `wrk_docs/m7-allowlist/dc8ba3c0.md`.
