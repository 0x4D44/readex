## The measurement correction (why the number jumped)

Midway through Wave 3 we discovered the metric was **measuring test code**.
`#[cfg(test)] mod tests` blocks live in the same `src/*.rs` files as production
code, so `cargo-llvm-cov` was instrumenting them. Test bodies emit their own
conditional branches:

- `assert!`/`assert_eq!` panic-sides (never taken in a passing test);
- table-driven `for tag in ["a","b",...] { assert!(...) }` loops;
- membership probes like `let in_either = SET_A.contains(x) || SET_B.contains(x);`
  — where the `||` short-circuit leaves one side unobserved.

The last one is perverse: a test *written to cover a production `||`* adds its
**own** half-uncovered `||` to the missed count. The four "missed" branches in
`settings_constants.rs` were **all** of this form — entirely inside the test
functions.

**Fix (standard practice):** opt `#[cfg(test)] mod tests` out of coverage
instrumentation via `#[cfg_attr(coverage_nightly, coverage(off))]` (33 modules)
+ `#![cfg_attr(coverage_nightly, feature(coverage_attribute))]` in the crate
root + a `[lints.rust] check-cfg` declaration. `cargo-llvm-cov` sets
`--cfg coverage_nightly`; no other build does, so stable/ordinary-nightly
builds are byte-unaffected (verified: clean `cargo build`, clean
`cargo build --tests`, 2333 tests pass).

Effect on the metric (full `--workspace --tests` run):
- test-INCLUSIVE: 85.47 % branches, 623 missed of 4288;
- production-ONLY: **88.20 % branches, 470 missed of 3982**.

So ~306 counted branches and ~153 "missed" were test-code artifacts.

## Honest assessment of the gap to 98 %

98 % of 3982 production branches ⇒ ≤ ~80 missed ⇒ close ~390 more.

The remaining branches fall into three buckets, and they are **not** evenly
tractable:

1. **Reachable with crafted input (~55–65 %).** Most of `metadata_url`,
   `metadata`, `metadata_jsonld`, `cleaning`, `xpath_engine`, `prep`,
   `extractors`, and the easy tail. These are ordinary "write a bespoke
   negative-shape test" work — exactly what the agent fleet does well. Each
   test now closes ~0.5–1 branch (the cheap targets are gone).

2. **Genuinely unreachable, faithful-port dead arms (~15–20 %).** The ports of
   Mozilla Readability / Trafilatura / htmldate preserve defensive guards
   verbatim (`Readability.js:2527` object-tag check after an upper-case
   tagName test; htmldate's `deferred_url_extractor` block pinned `false`;
   `external_date_parser` faithful stub; `&&` second-operands where the first
   operand is structurally always-true). The agents have annotated ~50 of
   these with `// llvm-cov:branch-not-reachable:` + rationale; a careful sweep
   would justifiably annotate ~50–80 more. These should be **annotated, not
   tested** — forcing a test would mean tampering with a pattern that cannot
   occur, which the plan forbids.

3. **Reachable but expensive (~15–20 %).** `grab_article`'s `_grabArticle`
   score-walk (≥3 alternative candidates with specific ratio/link-density/
   heading-density topologies) and `main_extractor`'s nested table-cell
   tail-recovery. Each needs a hand-crafted synthetic DOM that puts the scorer
   on a precise boundary. Feasible, but slow — each test is a small arithmetic
   puzzle.

**Realistic ceiling without production refactors:** ~96–98 %, reached by
(a) the agent fleet doing one more bucket-1 pass, (b) a disciplined bucket-2
annotation sweep, (c) a focused bucket-3 effort on grab_article/main_extractor.
The last 1–2 points may require either a small number of `branch-not-reachable`
annotations on the deepest `&&` short-circuits or accepting them as the
irreducible faithful-port residual.

### Why 98 % is not reachable here

The 136 unreachable annotations are **comments** — llvm-cov still counts those
branch sides in the denominator. So the **mathematical ceiling**, achieved only
if *every remaining reachable branch* were covered, is:

```
(3982 − 136) / 3982 = 96.58 %
```

98 % branch coverage is therefore **structurally impossible** on this codebase
under `--branch` **without one of**:

1. **Removing the faithful-port dead arms** from production code. These are
   deliberately preserved verbatim from Mozilla Readability / Trafilatura /
   htmldate (e.g. `Readability.js:2527` object-tag check after an upper-case
   tagName test; htmldate's `deferred_url_extractor` block pinned `false`;
   `external_date_parser` faithful stub; `&&` second-operands where the first
   operand is structurally always-true; html5ever-guaranteed `body()`/`head()`
   None guards). Removing them violates the project's faithful-port /
   anti-inversion doctrine and is an **Arthur-level architectural decision**.
2. **`#[coverage(off)]` on the enclosing production functions** — which would
   discard the *real* coverage of those functions too, and is exactly the
   Goodhart move the plan forbids on production code.

Neither is acceptable without an explicit decision. The honest, principled
ceiling is **~96.6 %**, and reaching it requires closing the remaining ~130
*reachable* branches.

### The ~130 reachable residual (why the last leg is so expensive)

After four passes the remaining reachable branches are the irreducibly-hard
tail, concentrated in:

- `readability/grab_article.rs` (32) — `_grabArticle` score-walk operand sides
  needing synthetic multi-candidate DOMs with precise finalized-score
  topologies (ratio ≥ 0.75 at a specific depth). Each test is a score-
  arithmetic puzzle that needs llvm-cov-in-the-loop to confirm the side moved.
- `trafilatura/main_extractor.rs` (31) — `handle_table` nested-cell tail
  recovery on `<code>`/`<quote>` cells; `handle_lists` tail-merge wide arms.
- `trafilatura/output.rs` (31) — TEI helper arms that fire only through the
  full integration emission path, plus a large named-charref decode table.
- `trafilatura/readability_fork.rs` (26), `readability/prep.rs` (17), and a
  long single-digit tail across the metadata / cleaning / clean / dom files.

Each remaining branch now costs a bespoke, often-puzzle-like test for ≈ 1
branch of movement — strongly diminishing returns versus the early stages.

### Recommendation

- **Accept ~93.3 % production branch coverage as the delivered result** (with
  97 % line / 96 % region / 97 % function), or
- **Authorise a final grind to the ~96.6 % ceiling** — feasible but expensive
  (one more agent wave with llvm-cov-in-the-loop on grab_article /
  main_extractor / output), or
- **Decide on the faithful-port dead arms** — if (and only if) the project is
  willing to either delete the preserved-verbatim dead branches or accept
  function-level `#[coverage(off)]` on the pure-faithful-port functions, the
  reported number can rise further, but at a cost to the faithful-port
  doctrine and/or the integrity of the metric.

98 % exactly is not honestly attainable under `--branch` on this codebase. This
is surfaced as the judgement call it is, rather than gamed.
