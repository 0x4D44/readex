# M10 Phase 2F — Parser Probe (html5ever vs lxml) on SEO-Spam Pages

**Date:** 2026-05-23
**Author:** Phase 2F probe agent
**Scope:** Hypothesis C tractability gate — determine whether the
post-Phase-2C SEO-spam tag-leakage residue (~25 records / ~5.7% of
working slice) is a bounded-fix mdrcel-side issue, or a parser-layer
divergence that requires multi-session work.
**Discipline:** HLD §7 — asymmetric trust + anti-inversion (every
Python claim cites file:line + verbatim quote; every probed bytes-on-disk
artifact lives under `D:\tmp\m10_phase2f\`).
**Outcome (TL;DR):** **MIXED, with the dominant signal being parser-layer
divergence.** html5ever and lxml DISAGREE on the SEO-spam subtree shape
(html5ever silently drops `<td>`/`<tr>`/`<th>`/`<tbody>`/`<tfoot>`/`<thead>`
tokens encountered outside a `<table>` insertion-mode context, and parses
`<rt>`/`<source>`/etc. inside `<xmp>`/`<noframes>` as RAW TEXT). The disagreement
is at the HTML5 tree-construction layer; html5ever exposes NO knob to change
this behaviour. **HOWEVER**, the secondary finding is that mdrcel's
`OVERALL_DISCARD_XPATH` plumbing (which contains a `contains(@style, "display:
none")` clause that matches every spam container) is **already correct** —
when invoked directly on `<body>` it removes 100% of spam divs across all 5
reps, with `backup_used=false`. The remaining leakage is therefore a
sub-tree-scoping issue (BODY_XPATH picks a sub-tree that may not include the
spam div, or a downstream arm like readability re-parses raw HTML without the
discard pass), NOT an absent or broken filter. **Recommendation: MEDIUM-
confidence tractable as a small follow-up probe to confirm pipeline locus**
(NOT HIGH-confidence: the leaked-tag-soup TEXT content inside raw-text
elements would still leak after any post-parse fix, because in mdrcel's tree
that text is no longer recoverable as elements). **Estimated yield range:
3-18 records** (the larger figure assumes both parser-layer text-leakage AND
sub-tree-scoping issues are addressed; the smaller figure assumes only the
sub-tree-scoping piece is bounded). At ~6 KPI points-per-50-records (the
Phase-2C ratio), this maps to **+0.4 to +2.1 KPI** for the bounded part.

### 3a. Exact disagreement shape (cited)

Three distinct parser-layer disagreements observed:

**(i) `<td>`/`<tr>`/`<th>`/`<tbody>`/`<tfoot>`/`<thead>` outside `<table>` context.** Per
`whatwg/HTML` spec §13.2.6.4.7 ("in body" insertion mode), encountering a
start tag whose name is in this set is a parse error and **the token is
ignored** — meaning the element is silently dropped. html5ever follows
the spec literally; lxml's HTML parser is permissive and keeps the
element as a child of whatever element is currently open. The
divergences.jsonl `mdrcel_window` evidence of leaked literal text like
`<td id="6fatj"></td>` in the OUTPUT comes from a DIFFERENT mechanism
(see (ii)), NOT from these tags surviving parse.

Verbatim from `D:\tmp\m10_phase2f\f66a4bfb73ef_python_pre_clean.txt:137-138`:

```
  <td> [id="woq6h"]
  <th> [id="woq6h"]
```

— inside the `pl_css_ganrao` container, direct children of the spam div.
And from `D:\tmp\m10_phase2f\f66a4bfb73ef_mdrcel_pre_clean.txt:520-528`
(stray-tag counts) — **`<td>`, `<th>`, `<tbody>`, `<tfoot>`, `<thead>`,
`<tr>` are missing from the entire mdrcel document tree**.

**(ii) `<xmp>` / `<noframes>` / `<style>` raw-text content.** These
elements are spec-defined raw-text / raw-script elements. html5ever
treats their contents as TEXT (the inner `<rt id="…">` becomes part of
the text node, not a nested element). lxml is permissive and parses inner
content as actual element children. Verbatim from
`D:\tmp\m10_phase2f\f66a4bfb73ef_mdrcel_pre_clean.txt:76-77`:

```
  <xmp> [id="woq6h"]
    #text "<rt id=\"woq6h\"><legend id=\"woq6h\"></legend></rt>"
```

vs lxml's parse (`D:\tmp\m10_phase2f\f66a4bfb73ef_python_pre_clean.txt:90-92`):

```
  <xmp> [id="woq6h"]
    <rt> [id="woq6h"]
      <legend> [id="woq6h"]
```

In mdrcel's tree this text content is no longer addressable as elements;
in lxml's tree it is. **This is the ONLY mechanism by which the literal
string `<rt id="…">` appears in mdrcel's final output** — it's the
serialised text content of `<xmp>` etc., not a foster-parented element.

**(iii) Content after `</body>`.** html5ever (per spec) keeps stray
content after `</body>` as still part of the body (the body close tag is
treated as advisory). lxml may drop or move it. In rep 5 `6f3fe19c6751`,
mdrcel finds **2** spam divs (one inside body, one after `</body>`); lxml
finds only **1** (the inside-body one). This is the **reverse direction**
of (i)/(ii) — mdrcel keeps MORE than lxml here. Not a leak source in
practice (the extra spam div is removed by OVERALL_DISCARD_XPATH on the
body level the same way), just an observation.

### 3b. Is there an html5ever knob to disable (i)/(ii)?

**No.** `html5ever::tree_builder::TreeBuilderOpts` exposes only:
`exact_errors`, `scripting_enabled`, `iframe_srcdoc`, `drop_doctype`,
`ignore_missing_rules`, `quirks_mode` (verified at
`C:\Users\marti\.cargo\registry\src\index.crates.io-1949cf8c6b5b557f\html5ever-0.27.0\src\tree_builder\mod.rs`).
None of these affects the "in body" insertion-mode rules for stray table
tags or raw-text element parsing — those behaviours are hardcoded in the
HTML5 tree-construction algorithm. `quirks_mode` is a CSS/box-model
emulation flag, not a parser-behaviour switch.

There is one alternative on the table: **pre-process the HTML bytes
before html5ever sees them** — strip the spam containers via regex or
detect raw-text `<xmp>` blocks and rewrite. But: (a) regex-stripping
HTML is fragile, (b) it diverges from Python's tree-cleaning model
(asymmetric-trust §7 concern), and (c) the Phase 2 triage already
considered this and noted it would be a multi-session work item.

## 5. Why does the leakage persist in the pipeline output then?

This is the critical follow-up: **mdrcel's filter is correct, the input
tree shape difference is identified — but the actual extraction
pipeline still emits the leaked text in the working-slice divergences.**
Three plausible reasons (not all verified in this probe):

1. **`prune_unwanted_sections` is called on a SUB-TREE picked by
   `BODY_XPATH`**, not on the full body. If `BODY_XPATH` selects a
   container that does NOT include the spam div (common when the spam
   div is a sibling of the article container, not a descendant), then
   the spam-div discard never runs on it. The spam div remains in the
   ORIGINAL tree and gets pulled in later (e.g. via the readability
   fallback arm or by an unscoped text-walk). Verified at
   `src/trafilatura/main_extractor.rs:1786`:
   `let subtree = prune_unwanted_sections(&subtree, &potential_tags, options);`
   — operates on `subtree`, not `body`.

2. **`compare_extraction` runs the readability arm by re-parsing the
   ORIGINAL raw HTML** (`src/trafilatura/readability_fork.rs:1973-1985`),
   which only applies OVERALL_DISCARD_XPATH conditionally
   (`if options.focus == Focus::Precision`). In the default
   `focus=Balanced` config — the corpus-baseline — that branch is
   skipped, so the readability arm operates on a tree that still
   contains the spam div. If the readability arm wins via the
   `compare_extraction` arbiter, the spam content leaks into the
   serialised output.

3. **The raw-text leakage from `<xmp>`/`<noframes>` is not addressable
   by OVERALL_DISCARD_XPATH at all.** Even if the spam div is removed
   entirely, IF the leaked text is inside a `<xmp>` that's nested inside
   the spam div, removing the spam div removes both. BUT — if `<xmp>`s
   containing the spam-tag-soup text appear elsewhere in the document
   (outside the spam div), they survive. This is a smaller leak source
   but still real.

A **secondary bounded probe** (option (c) of Phase 2D §10) could pin
which of (1)/(2)/(3) is responsible per record. **This probe does NOT
do that** (out of scope per the brief — "information only, do NOT
write a fix").

## 6. Confidence-rated recommendation

### Headline: **MEDIUM-confidence tractable** — with a precise caveat.

**The tractable part** (likely 1-2 sessions of bounded work, no parser
changes, no html5ever modifications):

- Pipeline-locus fix in `prune_unwanted_sections` call site —
  ensure OVERALL_DISCARD_XPATH runs on the body-level tree BEFORE
  BODY_XPATH narrows. The Python source does this implicitly because
  `tree_cleaning` is applied on the WHOLE tree at `core.py:280` before
  `trafilatura_sequence` picks sub-trees, and OVERALL_DISCARD_XPATH
  runs on whatever sub-tree the cascade lands on. If mdrcel's
  `BODY_XPATH` is picking a tighter sub-tree than Python's, the fix is
  one of: (a) add a body-level OVERALL_DISCARD pre-pass at
  `lib.rs`/`main_extractor.rs` (asymmetric trust §7: review whether
  Python does this), or (b) ensure `compare_extraction`'s readability
  arm applies OVERALL_DISCARD even in `focus=Balanced`.

**The intractable part** (multi-session, parser-layer):

- The text content of `<xmp>`/`<noframes>`/`<style>` raw-text elements
  containing `<rt id="…">` etc. will leak no matter how aggressively
  post-parse filtering runs, because in mdrcel's tree it's text not
  elements. A bounded fix here would require **pre-processing the HTML
  bytes** to strip these blocks before html5ever sees them — possible
  but risk-of-fragility per asymmetric-trust §7.

**One-sentence fix description (if pursued):** "Run
`prune_unwanted_nodes(body, OVERALL_DISCARD_XPATH, with_backup=true)` on
the full `<body>` tree before BODY_XPATH narrows" (covers ~80% of the
yield estimate; doesn't cover the raw-text-element residue).

### Why not HIGH-confidence?

Three reasons:

1. The probe identified that mdrcel's filter PLUMBING works correctly on
   the full body — but did NOT trace through the actual pipeline to
   verify which sub-tree the spam-tag-text actually rides into the
   output through. The fix locus is therefore inferred, not pinned.
2. The raw-text-element leakage (mechanism (ii)) is unaddressable
   without parser-layer or HTML pre-processing work — so a bounded fix
   yields PARTIAL improvement, not full elimination.
3. asymmetric-trust §7 requires that any change track Python's
   behaviour. The fix described in §6 may DIVERGE from Python (Python's
   filter runs on the same tree mdrcel's does, just one level wider).
   Verification needed before implementation.

### Why not HIGH-confidence parser-layer (OUT-OF-SCOPE)?

Because the probe showed `prune_unwanted_nodes` + OVERALL_DISCARD_XPATH
**does work** on the spam containers. The previously-suspected
parser-layer issue (Phase 2 §3 Hypothesis C: "html5ever foster-parents
or drops `<td>`") is half-correct — it does drop those tags — but
**that's not the leakage source**. The leakage source is text content
inside surviving raw-text elements (mechanism (ii)) and pipeline
sub-tree-scoping (§5 (1)/(2)). The first is parser-layer; the second is
not.

## 7. Estimated yield

Per the quantification done by the probe agent:

- **18 records** (across 11 unique pages) show an asymmetric stray-tag
  pattern (mdrcel has different `{td, tr, th, …}` token counts than the
  oracle window). This is the **strict** count from
  `divergences.jsonl` (with the cavet that windows are 80-char clipped,
  so true count is likely higher — Phase 2D estimated ~25).

- Of these 18, **all 14 markdown/txt** divergences are pure spam-class
  records (`extra-element` shape, `near_tag=''`); **4 xml** divergences
  are the N1/N2 sub-patterns (different mechanism per rep 4 d4ae —
  legitimate tables).

- **A bounded body-level OVERALL_DISCARD pre-pass fix would resolve**
  approximately 14 of the markdown/txt records (the ones whose leakage
  comes from the spam div, not from raw-text-element residue). At the
  Phase-2C ratio of ~50 records → +3.47 KPI (≈ 0.069 KPI / record), 14
  records → **+0.4 to +1.0 KPI** in the held-out slice. This is the
  **bounded-fix expected yield**.

- If the raw-text-element residue (mechanism (ii)) is ALSO addressed
  (via HTML pre-processing — out-of-scope per asymmetric trust §7), the
  remaining ~4 markdown/txt records and possibly several xml records
  drop, for a **total yield ceiling of ~18-25 records = +1.2 to +1.7
  KPI**.

- **Floor estimate (no fix attempted):** 0 records, 0 KPI — close M10
  here per Phase 2D §7 option (a).

## 8. Open questions for the team lead

1. **Where exactly does the leaked text enter the output stream?** The
   probe identified the parser-layer disagreement but did NOT pin
   through-line. Before any HLD on a fix, a secondary probe should:
   (i) run the full pipeline on rep 1 (f66a4bfb73ef), (ii) intercept
   the tree at each pipeline boundary, (iii) confirm which boundary
   the leaked `<rt id="woq6h">` text first appears at in the output.
   One session of work.

2. **Asymmetric-trust gate.** If the fix is "run OVERALL_DISCARD at the
   body level before BODY_XPATH", does Python do the same thing? Python
   calls `tree_cleaning(tree)` once on the whole tree (`core.py:280`)
   before `trafilatura_sequence`. But `OVERALL_DISCARD_XPATH` only
   runs inside `prune_unwanted_sections` (called by `_extract` /
   `extract_content`), on the per-BODY_XPATH-match sub-tree. So **Python
   has the same sub-tree-scoping issue mdrcel has** — but Python's lxml
   tree didn't have the leakage problem in the first place because it
   preserved the spam tags as elements that downstream filters
   eventually scored as boilerplate (link-density / empty-element
   prune). The asymmetric-trust answer is: **mdrcel's fix must not
   diverge from Python's filter set**; it can only re-order/re-scope
   existing filters. A body-level pre-pass with OVERALL_DISCARD is **NOT
   in Python's pipeline** and would be a §7 violation.

3. **Is the raw-text-leakage from `<xmp>` worth a separate small fix?**
   `<xmp>` is deprecated since HTML 3.2; it appears almost exclusively
   in SEO-spam pages. A targeted strip of `<xmp>`/`<noframes>` before
   html5ever sees them (or as a tree_cleaning addition) would close
   mechanism (ii). Total yield: ~4-7 records.

4. **N1 sub-pattern (rep 4 d4ae) — same root cause?** This probe shows
   d4ae is NOT a SEO-spam page (no `pl_css_ganrao` container). Its
   leaked-`<td>` shape is a different mechanism (rescue-pass `<p>`
   wrappers + legitimate `<td>` content). The Phase 2D triage's
   attribution of N1 to Hypothesis C should be revisited if a fix is
   pursued.
