## Pre-HLD note: one brief claim is verifiably wrong

The brief states:

> Python's `child.drop_tree()` removes the element AND its tail; rcdom's
> `dom::remove` removes ONLY the element — same trap as M9 Stage 5 R1. The
> rescue pass must use the M9 `delete_element` helper or equivalent.

This is **wrong about lxml**, though the conclusion about which helper to use
is still mostly right (see below). lxml's `drop_tree()` removes the element
**and re-anchors the tail onto the previous sibling (or parent.text if no
previous sibling)** — i.e. tail-PRESERVE, not tail-drop. Empirically confirmed:

```
$ python -c "
import lxml.html
t = lxml.html.fromstring('<div>A<br/>B<br/>C</div>')
for br in t.findall('.//br'):
    br.drop_tree()
print(lxml.html.tostring(t).decode())
"
<div>ABC</div>
```

The tails `B` and `C` were preserved (concatenated into the div's text run).
This matches mdrcel's existing `delete_with_tail_preserve_free` (`dom.rs:1307`),
which is already documented at `readability_fork.rs:758-762`:

> "lxml `drop_tree()` removes the element AND its subtree but re-anchors its
> tail text on the previous sibling (or parent.text if no previous sibling).
> Our `delete_with_tail_preserve_free` is exactly that semantic (`dom.rs:1191`)."

**Practical consequence for the rescue pass:** for the specific `<br>` rescue,
the tail has already been **swallowed into a freshly-created `<p>` sibling**
BEFORE the `child.drop_tree()` runs (because the algorithm runs the tail-rescue
clause first, which sets `child.tail = None`). So by the time `drop_tree` fires,
the `<br>` has NO tail to preserve, and `delete_with_tail_preserve_free` is the
correct primitive. The §3 algorithm spec preserves Python's order exactly so
this falls out for free.

(If a `<br>` had a whitespace-only tail (`.strip()` empty), the tail-rescue
clause does NOT fire, so the tail would survive into `drop_tree`. lxml then
re-anchors that whitespace tail to the previous sibling or parent.text —
which is the right thing. mdrcel's `delete_with_tail_preserve_free` matches.)

**Annotation:**

- **Lines 298-310 — RETAG PASS (already ported).** First iteration over
  `findall(".//div")`. For each `<div>` whose serialised children contain no
  block-tag opener (`divToPElementsRe` miss), retag to `<p>` via `elem.tag = "p"`.
  mdrcel's mirror is `readability_fork.rs:801-819` (current implementation).
- **Lines 312-324 — RESCUE PASS (to be ported).** Second iteration over
  `findall(".//div")` — a fresh `findall` after the retag pass. For each `<div>`
  remaining in the tree:
  - **Lines 313-316:** if leading text exists and is non-whitespace, wrap it in
    a fresh `<p>` inserted as child 0.
  - **Lines 318-324:** iterate child-element positions in REVERSE; if a child's
    tail is non-whitespace, wrap it in a fresh `<p>` inserted at `pos+1`; and
    if the child is a `<br>`, drop it.

**Key observations on the rescue pass:**

1. **Reverse enumeration is load-bearing.** `sorted(enumerate(elem),
   reverse=True)` materialises the index/child pairs ONCE, then iterates
   highest-index first. This means an `elem.insert(pos+1, p_elem)` at position
   `pos+1` shifts only ALREADY-PROCESSED indices (i.e. positions > pos), not
   yet-to-be-processed ones. The Rust port MUST preserve this.

2. **`enumerate(elem)` enumerates child ELEMENTS** (not text nodes). lxml's
   iteration over an Element yields child Elements only — text is `.text`
   / `.tail`, not a separate Text node. mdrcel's `children(node)` (filtered
   to elements, `dom.rs:338-345`) matches this exactly.

3. **`elem.insert(pos, p_elem)` is element-index-based**, not all-children-index-
   based. lxml's `insert(idx, child)` indexes into the child-element list.
   mdrcel's `insert_with_tail(new_parent, child, idx)` (`dom.rs:813`) indexes
   into the *all-children* `Vec` (Text + Element). The port must convert the
   element-index `pos+1` to the corresponding all-children index when calling
   `insert_with_tail`. See §3 + §4 for the conversion.

4. **`fragment_fromstring("<p/>")` creates a fresh, detached, empty `<p>`.**
   mdrcel's analogue is `create_element("p")` (`dom.rs:678`), which already
   produces an empty, namespace-tagged HTML element with no attrs/children.

5. **The two clauses inside the for-loop fire INDEPENDENTLY.** Both the tail
   rescue AND the `<br>` drop can fire on the same iteration. The tail clause
   runs first (line 319-322), the `<br>` clause runs second (line 323-324).
   For a `<br>` with a non-empty tail: tail becomes a fresh `<p>` sibling,
   then the `<br>` is dropped.

6. **Whitespace-only text/tail does NOT trigger wrapping.** The `if … and
   …strip()` guard. Empirically: `<div>   <a></a></div>` is unchanged by the
   rescue pass; the three-space text run survives because `.strip()` returns
   empty (falsy).

7. **The second `findall(".//div")` re-scans the tree after retag.** Divs
   that were retagged to `<p>` in the first pass are NOT visited by the second
   pass — they are `<p>` now, no longer matching `.//div`. So the rescue pass
   only sees divs that survived the retag.

        candidates = {}

        for elem in self.doc.iter("p", "pre", "td"):
            parent_node = elem.getparent()
            if parent_node is None:
                continue
            grand_parent_node = parent_node.getparent()

            elem_text = trim(elem.text_content())
            elem_text_len = len(elem_text)

            # discard too short paragraphs
            if elem_text_len < self.min_text_length:
                continue

            for node in (parent_node, grand_parent_node):
                if node is not None and node not in candidates:
                    candidates[node] = self.score_node(node)

            score = 1 + len(elem_text.split(",")) + min((elem_text_len / 100), 3)
            # if elem not in candidates:
            #    candidates[elem] = self.score_node(elem)

            candidates[parent_node].score += score
            if grand_parent_node is not None:
                candidates[grand_parent_node].score += score / 2

        # Scale the final candidates score based on link density. Good content
        # should have a relatively small link density (5% or less) and be
        # mostly unaffected by this operation.
        for elem, candidate in candidates.items():
            candidate.score *= 1 - self.get_link_density(elem)

**rescue affects scoring decisions.** mdrcel's current comment at

### 2d. Why the omission justification is wrong

The comment claims the rescue pass "does not affect the scoring decisions"
because "paragraph scoring already concatenates descendant text via
`text_content`". This is a confusion between two different invariants:

- **What the comment is right about:** if you were scoring a `<div>` directly,
  its `text_content` would include the leading text regardless of whether that
  text is in `.text` or in a `<p>` child. So the SCORE of the `<div>` itself
  would not change.
- **What the comment misses:** `score_paragraphs` (`readability_lxml.py:228`)
  iterates `self.doc.iter("p", "pre", "td")` — i.e. it scores `<p>`, `<pre>`,
  `<td>` elements, **not `<div>` elements**. The rescue pass CREATES new `<p>`
  elements. Those new `<p>` elements become scoring CANDIDATES that
  contribute `1 + len(text.split(",")) + min(text_len/100, 3)` to their
  parent's score (line 249) and half to the grandparent's score (line 251).
  The parent (the `<div>` that owns the new `<p>`) therefore gets MORE score
  than it would have without the rescue pass.

**Concrete consequence:** a `<div>` with leading text "Lorem ipsum dolor sit
amet, consectetur" (≥ `min_text_length` of 25) and no other `<p>`/`<pre>`/`<td>`
descendants:

- **Without rescue:** `score_paragraphs` finds zero qualifying `<p>` in this
  div. The div is never added as a parent candidate (the `for node in
  (parent_node, grand_parent_node)` branch at line 241 only runs if there IS
  a `<p>` whose parent is this div). The div has score zero, never appears in
  `candidates`, cannot be selected as best.
- **With rescue:** the leading text is wrapped in a new `<p>`. That `<p>` is
  iterated by `score_paragraphs`, its text length (45) is ≥ 25, the parent
  div is added to `candidates` via `score_node`, and the div's score becomes
  `class_weight + DIV_SCORES(+5) + (1 + 3 + 0.45) = base + 9.45`.

The rescue pass therefore upgrades div-only-text content from "invisible to
scoring" to "scoring candidate". This is exactly the divergence Phase 2B's
tracer observed on rep 5 (`781765a839ab`): the Vasilyev/sup pattern's `<div>`
text leaks into mdrcel's `text_content` but is never made into a `<p>`, so the
`<div>` doesn't score — and a different `<div>` wins selection.

**Verdict:** the comment must be removed and the rescue pass ported.

## 3. Algorithm specification

The pseudocode below mirrors Python's algorithm. It runs INSIDE
`transform_misused_divs_into_paragraphs`, AFTER the retag loop, BEFORE
returning the pinned Vec. It uses mdrcel's existing dom-facade primitives.

```
// Already done above this point:
//   for div in get_elements_by_tag_name(&doc, "div"):
//       (retag pass — unchanged)
//   This step retains a `pinned: Vec<NodeRef>` of post-retag handles.

// RESCUE PASS — readability_lxml.py:312-324
let doc = self.dom.document()
let divs_remaining = get_elements_by_tag_name(&doc, "div")   // findall(".//div")
                                                              // (post-retag — only
                                                              // divs that survived)

for div in divs_remaining:
    // 3a. Leading-text rescue (lines 313-316).
    let leading = element_text(&div)
    if leading is Some(s) and s.trim().is_non_empty():
        let p = create_element("p")
        set_element_text(&p, Some(s))    // p.text = leading
        set_element_text(&div, None)     // div.text = None (drains the leading
                                          // Text-node run)
        // div.insert(0, p) — insert as FIRST child element.
        // In mdrcel's all-children Vec, "first element" means index 0 if no
        // leading Text run; or index <leading-Text-run-len> if there is one
        // (but we just drained it, so index 0 is correct).
        // Use insert_with_tail to be tail-trap-safe.
        insert_with_tail(&div, &p, 0)
        pinned.push(p.clone())

    // 3b. Per-child tail/br rescue (lines 318-324).
    // Snapshot child elements BEFORE the loop (so insertions don't reshuffle
    // what we iterate).
    let child_elems: Vec<NodeRef> = children(&div)     // ELEMENT children only
                                                       // (filters Text/Comment)
                                                       // — matches enumerate(elem)
                                                       // semantics in lxml
    // Enumerate gives (element-index, child). Reverse-iterate.
    for (pos, child) in child_elems.iter().enumerate().rev():
        // 3b-i. Tail rescue (lines 319-322).
        let t = tail(&child)
        if t is Some(s) and s.trim().is_non_empty():
            let p = create_element("p")
            set_element_text(&p, Some(s))    // p.text = tail
            set_tail(&child, None)            // child.tail = None
            // div.insert(pos + 1, p) — insert AFTER `child` in the element list.
            // Need to convert element-index (pos+1) to all-children-index.
            let target_all_idx = nth_element_to_all_index(&div, pos + 1)
            insert_with_tail(&div, &p, target_all_idx)
            pinned.push(p.clone())

        // 3b-ii. <br> drop (lines 323-324).
        // Note: we operate on the snapshotted `child`, NOT re-fetched by index.
        // Identity-stable across insertions.
        if local_name(&child) == Some("br"):
            // lxml drop_tree() removes the element and re-anchors its tail
            // onto the previous sibling / parent.text. At this point the
            // tail has ALREADY been drained by 3b-i (if it was non-whitespace);
            // whitespace-only tails survived 3b-i and will be re-anchored by
            // delete_with_tail_preserve_free, matching Python.
            delete_with_tail_preserve_free(&child)
```

### 3c. The element-index → all-children-index conversion

Python's `elem.insert(pos+1, p_elem)` indexes into the child-element list.
mdrcel's `insert_with_tail(parent, child, idx)` indexes into the all-children
list (which includes Text nodes between elements). To insert "after the `pos`-th
element", we need the all-children-Vec index of the first slot at-or-after the
`(pos+1)`-th element. Pseudo-helper:

```
fn nth_element_to_all_index(parent: &NodeRef, n: usize) -> usize {
    // Return the all-children-Vec index where the n-th element child would
    // live. If n equals the element count, return all_children_len (append).
    let all = child_nodes(parent)      // includes Text nodes
    let mut elem_count = 0
    for (i, c) in all.iter().enumerate():
        if is_element(&c):
            if elem_count == n:
                return i
            elem_count += 1
    return all.len()    // n == elem_count → append at end
}
```

**Equivalent simpler formulation:** if we just-drained the leading Text run
via `set_element_text(div, None)`, the children list starts with elements
(no leading Text). For the tail rescue, the tail of `child` is the Text run
immediately after `child`; we drained it via `set_tail(child, None)`. So:

- the `pos`-th element child is now at all-children-Vec index `i`, where `i`
  is the position of `child` in `parent.children` (we can find it with
  `parent_and_index` indirectly).
- The slot at `pos+1` in the element-index sense is the slot IMMEDIATELY
  AFTER `child` in the all-children Vec — i.e. `i + 1` (because `child`'s
  tail Text-run was just drained).

So the conversion is: find `child`'s all-children index `i`, target is `i + 1`.
This is the cleaner implementation in §4.

### 3d. Iteration-order verification

The brief asks us to clarify Python's intent regarding the second `findall`.
Reading `readability_lxml.py:298` vs `:312`: both lines call
`self.doc.findall(".//div")` — a fresh `findall` each time. Between the two
calls, the retag pass mutates the tree (some `<div>`s become `<p>` via the
`elem.tag = "p"` assignment, which is in-place; the element identity is
preserved but the tag changes). After the retag pass, `findall(".//div")`
returns only `<div>` elements that still have tag `"div"`. **The rescue pass
does NOT process the retagged-to-`<p>` ones.**

Note also: lxml's `elem.tag = "p"` is an in-place tag change; mdrcel's
`replace_element_tag` (used at `readability_fork.rs:815`) DETACHES the old
node and returns a fresh node. The retag-pass `pinned` Vec already holds
those new `<p>` nodes (which are reparented in place of the old `<div>`s).
The rescue-pass's `get_elements_by_tag_name(&doc, "div")` will not include
them (they're `<p>` now). So the post-retag, pre-rescue tree has the right
shape: only un-retagged `<div>`s remain as `<div>`, and `get_elements_by_tag_name`
sees them. Crucially, `pinned` (from the retag pass) must still hold those
new `<p>`s — and it does — to defeat the rcdom Drop quirk.

## 4. mdrcel implementation proposal

### 4a. Function shape

Keep the existing function signature; extend it:

```rust
fn transform_misused_divs_into_paragraphs(&mut self) -> Vec<NodeRef> {
    let mut pinned: Vec<NodeRef> = Vec::new();
    let doc = self.dom.document();

    // === Pass 1 — retag (unchanged) ===
    for div in get_elements_by_tag_name(&doc, "div") {
        let joined: String = children(&div)
            .iter()
            .map(serialize_converted_tree)
            .collect();
        if !div_to_p_elements_re().is_match(&joined) {
            pinned.push(replace_element_tag(&div, "p"));
        }
    }

    // === Pass 2 — rescue (NEW; readability_lxml.py:312-324) ===
    //
    // A fresh descendant-only `<div>` snapshot picks up the post-retag
    // survivors. New <p> elements and detached <br>s are pushed onto
    // `pinned` so the rcdom Drop quirk (M3 Stage 3-B follow-on) cannot
    // strand the new graph.
    for div in get_elements_by_tag_name(&doc, "div") {
        // 4a-i. Leading-text rescue (lines 313-316).
        let leading = element_text(&div);
        if let Some(s) = leading.as_deref()
            && !s.trim().is_empty()
        {
            let p = create_element("p");
            set_element_text(&p, Some(s));    // p.text = leading
            set_element_text(&div, None);     // div.text = None
            insert_with_tail(&div, &p, 0);    // div.insert(0, p)
            pinned.push(p);
        }

        // 4a-ii. Per-child tail/br rescue (lines 318-324).
        //
        // Snapshot ELEMENT children only (matches Python's enumerate(elem)
        // over lxml's element iterator). REVERSE order so a later
        // insertion at "after this child" doesn't shift earlier indices.
        //
        // Important: we hold each `child` as an owned NodeRef so the
        // identity is stable across in-between mutations (Rc reference
        // equality is the test for sibling membership).
        let child_elems: Vec<NodeRef> = children(&div);
        for child in child_elems.iter().rev() {
            // 4a-ii-α. Tail rescue (lines 319-322).
            let t = tail(child);
            if let Some(s) = t.as_deref()
                && !s.trim().is_empty()
            {
                let p = create_element("p");
                set_element_text(&p, Some(s));
                set_tail(child, None);
                // div.insert(pos + 1, p): insert immediately AFTER `child`
                // in the all-children Vec. Since we just drained `child`'s
                // tail Text-run, the slot immediately after `child` is the
                // next element (or end-of-list). Compute `child`'s all-Vec
                // index and target `idx + 1`.
                let child_idx_in_div = div
                    .children
                    .borrow()
                    .iter()
                    .position(|c| Rc::ptr_eq(c, child))
                    .expect("child is still a member of div");
                insert_with_tail(&div, &p, child_idx_in_div + 1);
                pinned.push(p);
            }

            // 4a-ii-β. <br> drop (lines 323-324).
            // lxml drop_tree() preserves the tail by merging into the
            // previous sibling / parent.text — mdrcel's
            // delete_with_tail_preserve_free does the same (dom.rs:1307).
            // At this point any non-whitespace tail was already drained
            // by 4a-ii-α; whitespace-only tails (if any) are merged in
            // by delete_with_tail_preserve_free, matching Python.
            if local_name(child).as_deref() == Some("br") {
                // Push the detached br onto pinned BEFORE removing it,
                // so its Rc count stays > 0 across the remove call.
                // delete_with_tail_preserve_free does not return the
                // removed node, so we clone the handle into pinned.
                pinned.push(child.clone());
                delete_with_tail_preserve_free(child);
            }
        }
    }

    pinned
}
```

### 4b. Why `pinned` matters here (rcdom Drop quirk)

The retag pass's `replace_element_tag` motivation is documented at length
in `readability_fork.rs:792-800`. The rescue pass introduces two NEW classes
of node-handle that need the same protection:

1. **Newly-created `<p>` elements** from `create_element("p")`. These are
   freshly Rc-owned, with no parent at construction. After
   `insert_with_tail(&div, &p, …)`, the `<p>` is parented into the div's
   children Vec. The local binding `p` is dropped at end of statement, but
   the Rc count is now 1 (held by the div's `children: RefCell<Vec<NodeRef>>`).
   That's enough to keep `<p>` alive while the div lives. So strictly we do
   not NEED to push `p` onto `pinned` — but doing so makes the lifetime
   guarantee local-and-obvious, costs O(1) memory per div, and matches the
   defensive pattern used elsewhere. **Recommend pushing.**

2. **Detached `<br>` elements** from `delete_with_tail_preserve_free`. After
   removal, the `<br>`'s parent is `None` and the only remaining Rc is the
   one we held in the `child_elems` snapshot — which goes out of scope when
   the loop ends. If a `<br>` had children (unusual for `<br>` but possible
   for tampered HTML or invalid input), Dropping the last Rc would trigger
   the rcdom iterative-drain quirk. **Recommend pushing the `<br>` onto
   `pinned` before the remove call** to make the lifetime spelled out.

Both pushes are defensive; the M3 Stage 3-B `dones_alive` precedent
(`main_extractor.rs`, commit `a10dfa5`) makes this idiomatic.

### 4c. Why we iterate `get_elements_by_tag_name(&doc, "div")` a second time

The brief asks us to clarify Python's intent. Per §3d above: Python's second
`findall(".//div")` re-scans after the retag pass; only divs that survived
retag are visited. mdrcel mirrors this by calling `get_elements_by_tag_name`
again (after the retag pass mutated some divs into `<p>`s via
`replace_element_tag`, which physically REPLACES the node, so the freshly
re-scanned `<div>` list is exactly the survivors).

We could also walk the pinned Vec to find divs that were NOT retagged, but
that requires tracking the un-retagged path separately. The re-scan is
cleaner and matches Python textually — recommended.

The current imports at `readability_fork.rs:72-78` cover:
- `children`, `create_element`, `delete_with_tail_preserve_free`,
  `element_text`, `set_element_text`, `local_name`, `get_elements_by_tag_name`.

The rescue pass additionally needs:
- `tail` (already imported? — `use crate::readability::dom::{..., tail, ...}`).
  Not in the current imports — add it.
- `set_tail` — add.

So the use-statement gains `tail, set_tail, insert_with_tail`. No new dom-facade

## 7. Risk assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| **R1. 51-fixture corpus regression.** The rescue pass changes scoring inputs; some fixture currently byte-equal because of compensating errors may flip. | MEDIUM | The gate machinery surfaces any byte-equality regression. Decision tree: (a) if Python's behaviour is what mdrcel now matches → keep the port; the previous "passing" was a coincidence. (b) If a fixture diverges from BOTH old and new → genuine regression, revert. (c) If implementation finds a Python-faithful behaviour that breaks a fixture, allowlist with a §7 ADR citing the rescue-pass cause. The implementation phase MUST report any flip, not silently absorb. |
| **R2. rcdom Drop quirk.** New `<p>` elements / detached `<br>` elements could be Dropped before the `summary()` loop body finishes, draining child Vecs iteratively (the M3 Stage 3-B bug class). | LOW | Every newly-created `<p>` and detached `<br>` is pushed onto the existing `pinned: Vec<NodeRef>` whose lifetime already covers the loop body. Matches the M3 Stage 3-B precedent (`dones_alive` in `main_extractor.rs`). The `_dones_alive` binding in the caller (`summary()` line 615) is already in place. |
| **R3. Iteration-order ambiguity.** If the implementation iterates children in forward order, an `insert(pos+1, …)` shifts subsequent (yet-to-be-processed) indices, breaking the algorithm. | LOW | §3 + §4 spec is explicit about reverse iteration on a snapshotted `Vec` of element children. The unit tests in §6a directly cover the reverse-iteration invariant (`rescue_wraps_both_leading_and_tail` exercises three insertions in one div). |
| **R4. `<br>` tail-preserve semantics.** lxml's `drop_tree()` re-anchors the tail onto the previous sibling. mdrcel's `delete_with_tail_preserve_free` does the same — but if the implementation uses bare `dom::remove` by mistake, whitespace-only tails are silently dropped instead of merged. | LOW (catches in test) | The unit test `rescue_handles_br_with_whitespace_tail` specifies exact post-shape; an incorrect `remove` call would produce different output. The implementation MUST use `delete_with_tail_preserve_free`, not `remove`. |
| **R5. Empty post-rescue divs.** A `<div>` whose only content was `<br>` elements becomes an empty `<div>` after rescue. | LOW | The retag pass runs FIRST (per current code, unchanged). Such an empty `<div>` would have been retagged to `<p>` in the retag pass already (because empty children join to "" which doesn't match `divToPElementsRe`, so retag fires). By the time the rescue pass scans for `<div>`, that element is a `<p>` and won't be visited. Verified by reading the existing code at `readability_fork.rs:801-819`. No downstream concern. |
| **R6. Element-index vs all-children-index conversion bug.** Python's `insert(pos+1, p_elem)` indexes the element list; mdrcel's `insert_with_tail` indexes the all-children Vec (which includes Text nodes). Wrong conversion misplaces the new `<p>`. | LOW | §4a's chosen formulation finds `child`'s all-children index via `position(|c| Rc::ptr_eq(c, child))` and targets `idx + 1`. This is correct because: (i) `set_tail(child, None)` drained the Text-run between `child` and the next element; (ii) after draining, the slot at `child_idx + 1` is the NEXT element (or end-of-list), which is exactly where Python's `insert(pos+1, …)` would put the new `<p>`. The `rescue_wraps_both_leading_and_tail` test catches a wrong conversion (three `<p>` insertions interleaved with two `<a>` originals — any off-by-one breaks the expected shape). |
| **R7. `element_text` / `set_element_text` semantics on multi-Text-run elements.** mdrcel models lxml's `.text` as a *run* of leading Text-node children (`dom.rs:454`); `set_element_text(elem, None)` drains all of them. Python's `elem.text = None` is conceptually the same single-slot drain. | LOW | mdrcel's helpers are explicitly designed to mirror lxml `.text` / `.tail` semantics (see the M3 Stage 2b' commentary at `dom.rs:454-510` and `:530-558`). Same primitives are used elsewhere in the codebase without trouble. |
| **R8. Tracer rep 5 doesn't actually flip.** The HLD predicts rep 5 will byte-equal Python after the port, but selection cascade may still pick a different body element. | LOW-MEDIUM | If rep 5 doesn't flip, that's a signal that the rescue pass alone isn't sufficient — there's another root cause downstream. Implementation phase records the rep-5 status and surfaces to the team lead. Not a blocker for landing the port (the unit tests + 51-fixture green is the merit bar). |

   the existing M3 Stage 3-B precedent and costs O(1).*

   - (c) Test via the public `Document::summary()` and assert on the final
     extracted output — less direct but exercises the integrated path.
   *Design agent's recommendation: (a). The function is already a workhorse
