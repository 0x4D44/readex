## 3. Root-cause hypotheses

### Hypothesis A — "selection-phase" (≈ 286 records, ≈ 58% of working slice)

**Manifest.** mdrcel and Python choose different elements as the main content
block. Diff appears at or near `first_diff_byte = 0` (median first_diff_byte
across G1+G2+G6+G8 = 16; 53/146 G1 records and 45/122 G2 records start at byte 0).

**Quantification.** A record is classified as selection-phase if
`first_diff_byte / oracle_output_len < 0.05`. Counts per group:

| Group                          | Sel-phase | Structural | Total |
|--------------------------------|----------:|-----------:|------:|
| `(md, text-content, '')`        |        90 |         56 |   146 |
| `(txt, text-content, '')`       |        74 |         48 |   122 |
| `(xml, text-content, 'doc')`    |        25 |          7 |    32 |
| `(xml, extra-element, 'doc')`   |        39 |          5 |    44 |
| `(xml, missing-element, 'doc')` |        17 |          2 |    19 |
| `(xml, empty-vs-nonempty, '')`  |         7 |          0 |     7 |
| (other groups, summed)          |        34 |         86 |   120 |
| **TOTAL**                      |   **286** |    **204** | **490** |

**Why this is selection-phase, not serialisation:** the cross-format same-sha
correlation. 121/146 G1 shas also have a G2 divergence; 122/146 G1 shas also have
some xml divergence. If the bug were in `xmltotxt` or `process_element` it would
show up on only ONE format (because Python's `xmltotxt` is also what produces
the markdown/txt diff). The fact that the same page yields divergence on all
three formats means the underlying CONTENT (post-extraction body tree) differs.

**Vendored Python source.** The selection logic that diverges is the
`main_extractor.extract_content` / readability fallback chain, specifically:

- `core.py:30-95` — `extract` orchestrator; calls `extract_content` then falls
  back to `baseline()` if content is too short.
- `main_extractor.py:558-720` — `extract_content`; iterates over BODY_XPATH
  expressions and picks the first sub-tree with sufficient content.
- `main_extractor.py:30-50` — XPaths `BODY_XPATH`, `OVERALL_DISCARD_XPATH`,
  `TEASER_DISCARD_XPATH`, `PAYWALL_DISCARD_XPATH`, `COMMENTS_XPATH`,
  `COMMENTS_DISCARD_XPATH`.

**mdrcel mirror locations.** `src/trafilatura/main_extractor.rs` (the
extract_content / handle_paragraphs / handle_lists / handle_table pipeline) and
`src/trafilatura/xpaths.rs` (the XPath catalog).

**Verdict.** ALGORITHMICALLY equivalent (M9 audit confirmed). The difference is
in *what HTML reaches the XPath engine* and *how the XPath engine resolves
predicates*. This is the Phase 3 escape-valve question: a fix needs either an
empirical-tracer round per failing page (expensive — 286 unique snowflakes) or
a metric reframe (selection-equivalence vs byte-equivalence).

**Confidence:** N/A — this is the deferred / Phase-3 bucket. No fix is proposed.

### Hypothesis B — "blank-line accounting" near content transitions (G5 + G14, ~19 records)

**Manifest.** mdrcel's markdown / txt output has +1 or −1 blank lines compared
to oracle, at points where text content transitions across element boundaries
(div → div, p → list, etc.).

**Vendored Python source citation.**

- `xml.py:300-352` — `process_element`. The textless-element branch
  (`if not element.text and not element.tail: …`) at lines 309-336 emits `\n`
  for `NEWLINE_ELEMS`; the end-tag branch at lines 340-347 emits another `\n`
  for elements in `NEWLINE_ELEMS` not under a `cell`. So a `<p>` element with
  text emits text + `\n`; a `<p>` with `.tail` continues with the tail; an
  empty `<p/>` emits `\n`.
- `utils.py:303-313` — `sanitize`. Splits on `\n`, filters `None` from
  `line_processing`, rejoins with `\n`. Consecutive blank lines BETWEEN
  surviving lines collapse to **exactly one `\n`** between them; blank lines
  AT BOUNDARIES are dropped.

**mdrcel mirror.**

- `src/trafilatura/output.rs:899-915` — `sanitize`. Reads faithful — split on
  `'\n'`, filter `None` from `line_processing`, rejoin with `\n`.
- `src/trafilatura/output.rs:957-1003` — `line_processing`. Reads faithful per
  M10 Phase 1 (control-char strip + entity sub + trim + `None` for all-blank).
- `src/trafilatura/output.rs` (process_element analogue) — text-and-tail
  branches mirror Python.

**Verdict.** Algorithms match. The divergence is in WHAT process_element sees
— specifically, whether elements like `<div>` carry `.tail` text that wraps
into the next text run, or whether their tail terminates with `\n`-equivalent
boundary. This is the same DOM-text-storage layer concern as M9 root cause A
(rcdom tail-trap, `Cargo.toml:440-549`).

**Confidence:** **MEDIUM**. The pattern is real (19 markdown + 5 txt members)
but the fix locus is NOT confirmed. Needs an empirical-tracer round (dump
mdrcel's `<p>.text`/`.tail`/getsibling() values vs Python at the moment of
process_element for one representative page). Per-fix yield = ≈ 24 markdown/txt
records, but xml-shape downstream sensitivity is unknown.

### Hypothesis C — "SEO-spam tag leakage from `display:none` containers" (≈ 47 records, ≈ 10% of working slice)

**Manifest.** Pages with `<div class="pl_css_ganrao" style="display: none;">`
containers (or similar) holding malformed nested-tag soup such as
`<source id="X"><i id="X"><strike id="X">…</strike></i></source>`. Tags like
`<rt>`, `<source>`, `<fieldset>`, `<tfoot>`, `<td>`, `<ul>`, `<li>` leak into
mdrcel's output as ESCAPED text (e.g. `&lt;td id="X"&gt;&lt;/td&gt;`).

**Quantification.** Cross-group spam-pattern (matches the regex
`<(td|source|fieldset|rt|tfoot|ul|li|dfn|cite|acronym|tbody) id="…">`):

| Group                          | Spam | Total | %   |
|--------------------------------|-----:|------:|----:|
| `(txt, extra-element, '')`      |    6 |     8 | 75% |
| `(md, extra-element, '')`       |    6 |     8 | 75% |
| `(xml, extra-element, 'doc')`   |   10 |    44 | 23% |
| `(xml, text-content, 'doc')`    |    6 |    32 | 19% |
| `(xml, text-content, 'p')`      |    5 |    12 | 42% |
| `(md, text-content, '')`        |   24 |   146 | 16% |
| `(txt, text-content, '')`       |   22 |   122 | 18% |
| (others, summed)                |  ~15 |  ~118 |     |
| **TOTAL (unique pages, ~)**    | **~47** | **490** | **~10%** |

**Vendored Python source.**

- `settings.py:349-404` — `MANUALLY_CLEANED` list includes `iframe`,
  `fieldset`, `rp`, `rt`, `rtc`, `source` (the tags that leak).
- `settings.py:407-429` — `MANUALLY_STRIPPED` list includes `tbody`, `tfoot`,
  `thead`, `ruby`, `cite`, `dfn`, `acronym` (also leaking).
- `htmlprocessing.py:48-80` — `tree_cleaning`. Calls `strip_tags(tree,
  stripping_list)` at line 64, then iterates `cleaning_list` calling
  `delete_element` at lines 75-78. Order: stripped first, then cleaned.

**Empirical probe** (Python `lxml` + trafilatura on a fragment from `f66a4bfb…`):

```python
html = '<html><body><div class="pl_css_ganrao" style="display: none;">'
       '<source id="X"></source><fieldset id="X"><tbody id="X"></tbody></fieldset>'
       '<td id="X"></td><ul id="X"><li id="X"></li></ul></div></body></html>'
tree = lxml.html.fromstring(html)
# After parse: <td> is preserved verbatim INSIDE the <div> (no foster parenting,
# because lxml's HTML parser is permissive). After tree_cleaning: <source>,
# <fieldset> are removed (MANUALLY_CLEANED); <tbody> is unwrapped (its empty
# child set means nothing survives); <td>, <ul>, <li> SURVIVE.
```

So even Python's tree_cleaning leaves `<td>`, `<ul>`, `<li>` inside the spam
`<div>`. The reason **Python's final output excludes them entirely** must be
a *later* filter — likely `prune_html` (empty-element removal) finishes the job,
or the spam `<div>` itself is excluded by `link_density_test` /
`OVERALL_DISCARD_XPATH` / `TEASER_DISCARD_XPATH` because its descendant text is
near-zero.

**mdrcel mirror.**

- `src/trafilatura/cleaning.rs:220-272` — `tree_cleaning`. Mirrors Python
  semantics (strip_tags_multi for stripping_list, delete_elements_by_tag for
  cleaning_list, then prune_html).
- `src/trafilatura/cleaning.rs:416+` — `prune_html`. Walks descendants; deletes
  empty elements whose tag is in CUT_EMPTY_ELEMS. CUT_EMPTY_ELEMS does NOT
  include `td`, `ul`, `li`, `tr`, `cell` (settings.py:320-343 — the catalog
  is `article, b, blockquote, dd, div, dt, em, h1-h6, i, li, main, p, pre, q,
  section, span, strong`). Wait — `li` IS in CUT_EMPTY_ELEMS.

**The key divergence is most likely upstream of `tree_cleaning`: html5ever's
parsing of `<td>` / `<tr>` outside a `<table>` context.** HTML5 spec says
`<td>` in "in body" insertion mode is a parse error and the token is ignored
(spec section 13.2.6.4.7). lxml's HTML parser is permissive and keeps `<td>`
as a child of whatever element is open. **html5ever follows HTML5 spec more
strictly** and may either drop the token or foster-parent it into a synthetic
`<table>` context. Either way, mdrcel's tree differs from Python's lxml tree
at the start, and downstream `tree_cleaning` operates on different inputs.

If `<td id="X">…</td>` parses into html5ever's tree as TEXT inside the parent
div (rather than as an element), then mdrcel's serialiser would emit the
literal characters as escaped text — exactly matching the observed
`&lt;td id="6fatj"&gt;` pattern.

**Confidence:** **MEDIUM**. The signature is clear and the root mechanism
(rcdom-vs-lxml parsing semantic gap) is consistent with the M9 finding. But the
fix locus is NOT obviously a one-liner — it's either:

- (a) A pre-processing pass that strips out `<td>`/`<tr>` outside table context
  before tree_cleaning runs, OR
- (b) Replicate lxml's permissive parsing more faithfully (large change to
  rcdom integration), OR
- (c) Add a `display: none` / known-spam-class XPath to `OVERALL_DISCARD_XPATH`
  equivalent in mdrcel (BUT — Python doesn't have this either, so adding it
  diverges from upstream and breaks asymmetric-trust §7).

Option (a) is closest to a HIGH-confidence patch but requires careful design.

**Allowlist candidacy:** marginal. Python's own code does not have a
display:none strip, and `pl_css_ganrao` is a Chinese SEO-spam pattern — neither
side has explicit handling. Python's downstream filters happen to clean these
pages better because of cascading effects (boilerplate density filters); not
because of an *intentional* difference. **Not an allowlist candidate**.

### Hypothesis D — "double-escaped entity decoding" (≤ 5 records)

**Manifest.** Source HTML contains `&amp;uumlck` (double-escape: literally
`&amp;` then `uumlck`). After lxml's text extraction, the `.text` attribute
is `&uumlck`. Python's `xml.py` does NOT pass this through `html.unescape`
in the default path, so it remains as `&uumlck` in the serialised output. But
mdrcel's `process_element` calls `unescape` (the html5 entity-decoder mirror)
on the text, decoding `&uumlck` to `ü`.

Rep evidence: G13 rep 3 `sha=8f783a9b2712`:
- oracle: `…Probe …St&amp;uumlck …`  (rendered: `St&uumlck`)
- mdrcel: `…Probe …Stück…`  (decoded)

**Vendored Python source.**

- `xml.py:363` — `xmltotxt` ends with `unescape(sanitize(…) or "")` — so
  `unescape` IS called on the joined output.
- `utils.py` — `from html import unescape`.

**But:** the divergence record shows oracle WITH the entity (`&amp;uumlck`)
and mdrcel WITHOUT. So Python's `unescape("&amp;uumlck")` returned
`&uumlck` (single unescape step — `&amp;` → `&`, leaving `&uumlck` as an
unrecognised entity). mdrcel's `unescape` call is decoding `&uumlck` as a
two-step `&amp;` + `&uumlck` → `&uumlck` → `ü`. **This is a divergence in
the `unescape` implementation specifically.**

**mdrcel mirror.** `src/trafilatura/output.rs:1010+` — `unescape` helper.

**Confidence:** **LOW**. Only 1-2 records exhibit this pattern; the fix is
either deferral (not worth one-record yield) or a specific known-entity carve-
out. **Deferred bucket.**

### Hypothesis E — "smart-quote preservation" (≤ 2 records)

G13 rep 1 `sha=1d3fc86310f7`: oracle has `'thing'` (straight) vs mdrcel
`'thing'` (smart). Python's `trim` likely passes smart quotes through but
some upstream layer maps them to straight in oracle output. Single record;
deferred.

**The 58% selection-phase ratio CLEARS the HLD's Phase 3 ≥50% gate.**

However, the HLD's Phase 3 gate also requires "**a specific algorithmic

a localized PARSER difference.**

**Recommendation for Phase 3:** the gate's literal text is not satisfied. But
the SPIRIT of the gate is "do we know enough about WHY selection differs to
justify a refactor?" The answer is "yes — html5ever vs lxml". Whether that
justifies the multi-session refactor is the metric-question the milestone
brief flagged.

Conservative interpretation: surface to Arthur for go/no-go before sinking
