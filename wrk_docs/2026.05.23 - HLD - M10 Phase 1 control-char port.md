## 1. Vendored Python source

`benchmark/oracles/trafilatura/.venv/Lib/site-packages/trafilatura/utils.py:266-274`:

```python
@lru_cache(maxsize=2**14)  # sys.maxunicode = 1114111
def return_printables_and_spaces(char: str) -> str:
    'Return a character if it belongs to certain classes'
    return char if char.isprintable() or char.isspace() else ''


def remove_control_characters(string: str) -> str:
    '''Prevent non-printable and XML invalid character errors'''
    return ''.join(map(return_printables_and_spaces, string))
```

**Type signature:** `str -> str`. It operates on a Python string only — never
on an lxml element or tree. (The DOM-walking happens in `sanitize_tree`
upstream, which feeds each text/tail slot through `sanitize` →
`line_processing` → `remove_control_characters`.)

**Codepoint-iteration semantics, not regex.** Iterates one Python codepoint at
a time and keeps the codepoint iff `char.isprintable() or char.isspace()`. This
is the only correctness-load-bearing detail of the function. The `@lru_cache`
on `return_printables_and_spaces` is a per-codepoint memoisation tactic; it
makes the function effectively a 1.1M-entry lookup table after warm-up. The
mdrcel port should NOT replicate the `@lru_cache` literally — see §4.

### Which Unicode categories does it strip vs keep?

Verified empirically against the vendored Python interpreter
(`benchmark/oracles/trafilatura/.venv/Scripts/python.exe`, run by this design
agent). Results across the full Unicode codepoint range (0..0x110000):

| Category | Total codepoints | Kept | Stripped |
|----------|---:|---:|---:|
| Cc (Control)     | 65         | **10**    | 55         |
| Cf (Format)      | 170        | 0         | **170 (all)** |
| Cn (Unassigned)  | 819,533    | 0         | **all**    |
| Co (Private Use) | 137,468    | 0         | **all**    |
| Cs (Surrogate)   | 2,048      | 0         | **all**    |
| Ll, Lu, Lt, Lm, Lo (Letters)  | …  | all       | 0          |
| Nd, Nl, No (Numbers)          | …  | all       | 0          |
| Mn, Mc, Me (Marks)            | …  | all       | 0          |
| Pc/Pd/Pe/Pf/Pi/Po/Ps (Punct.) | …  | all       | 0          |
| Sc, Sk, Sm, So (Symbols)      | …  | all       | 0          |
| Zs (Space Separator)          | 17 | all       | 0          |
| Zl (Line Separator)           | 1  | all (U+2028) | 0       |
| Zp (Paragraph Separator)      | 1  | all (U+2029) | 0       |

**The exact set of Cc kept** — this is the load-bearing list:

| Codepoint | Name | Why kept |
|---|---|---|
| U+0009 | TAB                | `isspace()` |
| U+000A | LF                 | `isspace()` |
| U+000B | VT                 | `isspace()` |
| U+000C | FF                 | `isspace()` |
| U+000D | CR                 | `isspace()` |
| U+001C | INFORMATION SEPARATOR FOUR  | `isspace()` |
| U+001D | INFORMATION SEPARATOR THREE | `isspace()` |
| U+001E | INFORMATION SEPARATOR TWO   | `isspace()` |
| U+001F | INFORMATION SEPARATOR ONE   | `isspace()` |
| U+0085 | NEL (Next Line)             | `isspace()` |

**Cc surprise (call it out):** 0x1C–0x1F (the FS/GS/RS/US ASCII separators) and
0x85 (C1 NEL) are `isspace()` in Python. They do **not** show up in Rust's
`char::is_whitespace()` (which follows Unicode `White_Space` property,
excluding the C0 separators and C1 NEL). This is the single most important
trap in the port: a naive `c.is_whitespace() || !c.is_control()` filter will
strip the wrong set. See §4.

**Cf surprise: NO Cf is kept.** This includes:
- U+00AD SOFT HYPHEN (the discretionary hyphen);
- U+200B / U+200C / U+200D (ZWSP, ZWNJ, ZWJ);
- U+200E / U+200F (LRM, RLM directional marks);
- U+2060 / U+2063 (WORD JOINER, INVISIBLE SEPARATOR — the M7 fixture's leak);
- U+FEFF (BOM).

This is counter-intuitive — directional marks and the BOM are semantically
meaningful in many contexts. But Python's `str.isprintable()` returns False
for the entire `C` category (Cc/Cf/Cs/Co/Cn), and they are not in `isspace()`,
so `remove_control_characters` drops them. **mdrcel must do the same** for
byte parity.

**Co (Private Use Area) is stripped.** All of U+E000–U+F8FF, plus the SPUA
planes 15–16. This is unlikely to appear in real crawler output, but the port
must respect it for fidelity.

**Cs (Surrogates) is stripped.** Not reachable from a well-formed Rust `&str`
(it is UTF-8-validated), so this is a no-op in practice but the implementation
must not assume so — see Risk §7.

### Regex vs codepoint iteration

Python iterates codepoints one at a time. The mdrcel port can use the
`regex` crate's `\p{C}`-subset class with explicit Cc carve-outs (see §4)
— this gives a one-pass `replace_all` with no per-codepoint allocation,
and the kept set is identical to Python's. A full-range empirical sweep
(`notes/m10-strip-probe/`) confirms zero disagreements between the
proposed regex and Python's `isprintable() or isspace()` predicate across
all 1,112,064 non-surrogate codepoints. See §4 for the chosen regex.

## 2. All Python call sites for `remove_control_characters` (direct or via wrappers)

`remove_control_characters` itself is called from exactly **one** site, but
its wrapper `line_processing` is reached via `sanitize` and `sanitize_tree`
from many. The transitive set is what mdrcel must match.

### Direct call site

**`utils.py:288`** — inside `line_processing(line, preserve_space=False, trailing_space=False)`:

```python
def line_processing(line: str, preserve_space: bool = False, trailing_space: bool = False) -> Optional[str]:
    '''Remove HTML space entities, then discard incompatible unicode
       and invalid XML characters on line level'''
    # spacing HTML entities: ...
    new_line = remove_control_characters(line.replace('&#13;', '\r').replace('&#10;', '\n').replace('&nbsp;', ' '))
    if not preserve_space:
        ...
```

The control-character strip runs *immediately after* the three HTML-space-entity
substitutions and *before* the `LINES_TRIMMING` regex + `trim`. Order matters:
the `&nbsp;` -> U+00A0 substitution happens first, so U+00A0 (which IS kept by
`isprintable or isspace` — category Zs) survives the strip; numeric-entity
replacements `&#13;` and `&#10;` likewise produce `\r` / `\n` which survive
(both are `isspace()` Cc-kept).

### 3c. NFC + control-char ordering

**Python order** (`core.py:98`): strip first (during sanitize), NFC last.

**mdrcel current order** (`src/lib.rs:702-705`, :812-814, :1132-1135, :1257-1259):
NFC last in every formatter. There is no strip step today, but when one is
added it MUST run *before* NFC, inside `sanitize` / `line_processing` /
`sanitize_tree`, matching Python's call-site location.

**Does NFC ever introduce a Cc/Cf char from non-control input?**
Empirically verified across BMP: **no**. For every BMP codepoint whose
category does not start with `C`, NFC/NFD/NFKC/NFKD produces no Cc or Cf
codepoints. This is consistent with the Unicode Standard: canonical and
compatibility decompositions are defined only for letters, marks, and a few
symbols — never for control or format characters. NFC is safe to run after
the strip.

**Could NFC introduce a Cn (unassigned) codepoint?** No — NFC only emits
already-assigned characters; `Cn` is by definition unassigned, and the
mapping table contains no Cn outputs.

**Conclusion:** order is Python's order. Strip in `line_processing` /
`sanitize_tree`, NFC at the end of each `extract_to_*`. No reordering.

## 4. Proposed mdrcel implementation

### Location

The shared helper `fn strip_control_chars(s: &str) -> String` lives in
`src/trafilatura/output.rs`, immediately above the existing
`line_processing` at line 925 (within the same module / `pub(crate)`
scope, no new public surface). Three callers consume it:

1. `src/trafilatura/output.rs:line_processing` — the `xmltotxt` /
   `sanitize_tree` cascade (text/tail of every element of every
   format).
2. `src/trafilatura/metadata_url.rs:876:line_processing` — the
   `extract_catstags` per-element path (categories/tags). Re-export
   `pub(crate) use crate::trafilatura::output::strip_control_chars;`
   or duplicate the small helper into `metadata_url.rs`.
   **Recommendation:** re-export, single source of truth.
3. `src/trafilatura/metadata.rs:clean_and_trim_metadata` — the new
   centralised metadata-slot strip introduced by the O1 resolution
   (§5). Called once at the end of `extract_metadata`. Same
   re-export pattern.

The two `line_processing` functions remain separate — they have
intentionally different call surfaces (one runs full `LINES_TRIMMING` +
trim, one just does entity-sub + trim); conflating is its own
milestone. Just share the `strip_control_chars` helper.

### Signature

```rust
/// Faithful port of Python's `remove_control_characters` (utils.py:266-274).
///
/// Strips every codepoint for which `char.isprintable() == False
/// && char.isspace() == False` in Python — empirically equivalent to:
///   - Strip all of categories Cf, Co, Cs, Cn.
///   - Strip Cc EXCEPT the 10 codepoints Python's str.isspace() keeps:
///     \t \n \v \f \r (U+0009..U+000D), U+001C..U+001F (FS/GS/RS/US), U+0085 (NEL).
///   - Keep everything else (L, M, N, P, S, Zs/Zl/Zp).
///
/// Returns a new String (matching Python's str -> str signature). No DOM
/// mutation; callers (line_processing / sanitize_tree mirror) thread the
/// result back into the text/tail slot they own.
pub(crate) fn strip_control_chars(s: &str) -> String { ... }
```

`&str -> String` is correct (Python: `str -> str`). It does NOT take a
`&mut NodeRef` — DOM walking happens at the existing `sanitize_tree` call
site (`output.rs:827`), which already knows how to read/write text and tail.

### Dependency choice

**Locked: option A3 — use the existing `regex` crate (`=1.12.3`), no new
crates, no vendored data table, no per-codepoint scalar loop.** Review
comment 2 (team lead) explicitly chose A3; A1 / A2 / B / C are out of
scope unless a follow-up perf milestone reopens them.

The strip is one compiled-once `OnceLock<Regex>` and one
`Regex::replace_all` per call. The kept-Cc set becomes a comment about
which ranges are carved OUT of `\p{Cc}` inside the character class —
NOT a runtime data structure.

### The strip regex

```
[\p{Cf}\p{Co}\p{Cn}\x00-\x08\x0E-\x1B\x7F-\x84\x86-\x9F]
```

Construction:
- `\p{Cf}` — Format (all 170 codepoints strip).
- `\p{Co}` — Private Use (all 137,468 codepoints strip).
- `\p{Cn}` — Unassigned (all ~820k codepoints strip).
- `\x00-\x08` — Cc minus 0x09 TAB.
- `\x0E-\x1B` — Cc between 0x0D CR and 0x1C FS.
- `\x7F-\x84` — Cc between 0x7E and 0x85 NEL.
- `\x86-\x9F` — Cc above 0x85 NEL.
- (Kept Cc: 0x09-0x0D + 0x1C-0x1F + 0x85 — i.e. the gaps between the
  four hex ranges above.)
- Cs (surrogates) is unreachable from a Rust `&str` and is therefore
  omitted from the regex; lone surrogates cannot enter the function.

**Probe-compile + full-Unicode equivalence test (this design agent ran it):**

- The regex compiled cleanly against `regex =1.12.3` (current pin).
  The hex-range syntax `\x00-\x08` IS accepted inside a `\p{}`-bearing
  character class; no fallback to `\u{0000}` notation was needed.
- A full sweep of all 1,112,064 non-surrogate codepoints (`notes/m10-strip-probe/`)
  shows ZERO disagreements between this regex and Python's
  `not (chr(cp).isprintable() or chr(cp).isspace())` predicate. The
  byte-exact equivalence holds across the entire Unicode codepoint range
  for the vendored Python interpreter
  (`benchmark/oracles/trafilatura/.venv/Scripts/python.exe`).

A1 (hand-coded predicate) and A2 (vendored bitmap) are explicitly
rejected for Phase 1: A1 cannot enumerate Cn from std alone, and A2 was
proposed as a perf hedge against A3's regex compile cost — the full-sweep
probe shows A3 is correct and the perf budget is comfortable (see §6e
for the timing check). C (new general-category crate) is rejected for
the same dep-posture reason as the previous revision.

### Pseudocode

```rust
// At the top of src/trafilatura/output.rs (or in a sibling internal module
// `output_strip` if review prefers separation).

use regex::Regex;
use std::sync::OnceLock;

/// Faithful port of Python's `remove_control_characters`
/// (`utils.py:266-274`).
///
/// One `Regex::replace_all` over the input. The character class covers
/// every codepoint Python's `isprintable() or isspace()` predicate
/// rejects:
///
///   - `\p{Cf}` — Format (all stripped; e.g. ZWSP U+200B, INVISIBLE
///     SEPARATOR U+2063, BOM U+FEFF, SOFT HYPHEN U+00AD).
///   - `\p{Co}` — Private Use Area (all stripped).
///   - `\p{Cn}` — Unassigned (all stripped).
///   - `\x00-\x08`, `\x0E-\x1B`, `\x7F-\x84`, `\x86-\x9F` — the Cc
///     codepoints Python rejects (i.e. `\p{Cc}` minus the ten
///     `isspace()` Cc kept-set: 0x09-0x0D, 0x1C-0x1F, 0x85 NEL).
///   - Cs (surrogates) is structurally unreachable in `&str` and is
///     omitted from the class.
///
/// Full-Unicode equivalence with Python verified by an offline sweep
/// (`notes/m10-strip-probe/`) — zero disagreements across all
/// 1,112,064 non-surrogate codepoints.
pub(crate) fn strip_control_chars(s: &str) -> String {
    static STRIP_RE: OnceLock<Regex> = OnceLock::new();
    let re = STRIP_RE.get_or_init(|| {
        Regex::new(r"[\p{Cf}\p{Co}\p{Cn}\x00-\x08\x0E-\x1B\x7F-\x84\x86-\x9F]")
            .expect("static regex")
    });
    re.replace_all(s, "").into_owned()
}
```

The pseudocode is six lines. `Regex::replace_all` returns a
`Cow<str>` that is borrowed when no match is found (the common case
on clean inputs) and owned when at least one replacement happens;
`into_owned` realises the `Cow` to a `String` to match Python's
`str -> str` signature regardless of branch. The static regex is built
on first call (~µs) and reused for the lifetime of the process.

### Integration into `metadata.rs::extract_metadata` (new, per O1)

A new internal helper `fn clean_and_trim_metadata(m: &mut Metadata)`
lives in `src/trafilatura/metadata.rs`, in scope of the module. It is
called once at the end of `extract_metadata`, immediately before the
final `metadata` return at `src/trafilatura/metadata.rs:1080`. Shape:

```rust
/// Phase-1 port of Python's `metadata.clean_and_trim()`
/// (`metadata.py:587` → `settings.py:288-298`). Strips control
/// characters from every str-typed metadata slot. The length cap and
/// `html.unescape` portions of Python's `clean_and_trim` are deferred
/// (see HLD §5 "Phase 1 scope clarification").
fn clean_and_trim_metadata(m: &mut Metadata) {
    fn strip_in_place(slot: &mut Option<String>) {
        if let Some(v) = slot.take() {
            let cleaned = strip_control_chars(&v);
            // Preserve `Some("")` vs `None` semantics: only re-store
            // the cleaned string if it would survive `is_empty()`
            // when the caller's truthy guard runs. mdrcel uses
            // `if !v.is_empty()` everywhere as the truthy proxy
            // (matching Python's `if value:` semantic), so an empty
            // post-strip is functionally equivalent to None — but we
            // keep it as `Some("")` here to avoid changing the
            // None/Some shape silently. The formatter's `is_empty()`
            // guards do the right thing.
            *slot = Some(cleaned);
        }
    }
    strip_in_place(&mut m.title);
    strip_in_place(&mut m.author);
    strip_in_place(&mut m.url);
    strip_in_place(&mut m.hostname);
    strip_in_place(&mut m.description);
    strip_in_place(&mut m.site_name);
    strip_in_place(&mut m.date);
    strip_in_place(&mut m.language);
    strip_in_place(&mut m.image);
    strip_in_place(&mut m.pagetype);
    strip_in_place(&mut m.license);
    strip_in_place(&mut m.filedate);
    for c in m.categories.iter_mut() {
        *c = strip_control_chars(c);
    }
    for t in m.tags.iter_mut() {
        *t = strip_control_chars(t);
    }
}

// ... at the end of extract_metadata, just before `metadata`:
clean_and_trim_metadata(&mut metadata);
metadata
```

The helper is private (no `pub`/`pub(crate)`) — it's an internal
extension point for the future full `clean_and_trim` port. Re-uses the
`strip_control_chars` helper via `pub(crate) use
crate::trafilatura::output::strip_control_chars;`.

### O1 resolution — metadata-slot strip placement

**Investigation:** `metadata.py:587` ends `extract_metadata` with
`metadata.clean_and_trim()`, which (per `settings.py:288-298`) iterates
every `__slot__` of the `Document` and, when the slot is a `str`, runs
`value = line_processing(unescape(value))` (with a prior 10,000-char
truncation step). This is the **single centralised cleanup** that Python
applies to title / author / url / hostname / description / sitename /
date / language / image / pagetype / license / filedate. The three
formatters (`build_yaml_header`, `build_json_output`, `xmltocsv`) and
the XML/TEI `add_xml_meta` consumer all read the *post-clean* values
from the `Document` — no per-formatter sanitisation happens (verified
by reading Python's `core.py:75-90` (YAML header), `xml.py:115-134`
(`build_json_output`), `xml.py:366-390` (`xmltocsv`),
`xml.py:178-198` (`add_xml_meta`); none of them apply
`line_processing` or `sanitize`).

**mdrcel today:** `extract_metadata` (`src/trafilatura/metadata.rs:919-1081`)
returns `Metadata` slots as harvested. The three formatters
(`build_yaml_header` `output.rs:1239`, `build_json_output` `output.rs:1348`,
`xmltocsv` `output.rs:1503`) and `add_xml_meta` (`output.rs:1641`) all
read those slots directly and emit them through `format!` /
`json_optional_str` / `csv_or_null` / `set_attribute` with NO
sanitisation step. A Cf in a `<meta name="description">` therefore
leaks into JSON `excerpt`, CSV (no description column, but other
columns are affected — e.g. `title`), and the markdown YAML
`description:` line, plus the `<doc>` element's attributes in XML/TEI.

**Patch decision (per RC2): patch `extract_metadata` once, not each
formatter individually.** A single new internal helper
`fn clean_and_trim_metadata(m: &mut Metadata)` lives in
`src/trafilatura/metadata.rs`, immediately above `extract_metadata`; it
runs `strip_control_chars` over every `Option<String>` slot's value
(when `Some` and non-empty) and over every entry in `categories` / `tags`
(belt-and-braces — `extract_catstags`'s per-element `line_processing`
will already have stripped them, but JSON-LD-sourced category/tag
entries (`metadata_jsonld.rs`) bypass that path). The helper is called
once, at the end of `extract_metadata` (just before the final
`metadata` return at `src/trafilatura/metadata.rs:1080`).

This single patch covers all four metadata-bearing emitters
(YAML header, JSON, CSV, XML/TEI `<doc>` attributes) AND mirrors
Python's architecture: centralised post-extraction cleanup, no
per-formatter sanitisation. The smallest-change preference applies —
one new call site in `extract_metadata` is strictly less work than
four formatter patches and avoids the "did I cover them all" failure
mode.

**Phase 1 scope clarification: strip only, not full `clean_and_trim`.**
Python's `clean_and_trim` does three things: (a) 10,000-char length
truncation with `…`, (b) `html.unescape(value)`, (c) the
`line_processing` cascade (entity-sub + strip + LINES_TRIMMING +
end-trim). Phase 1 ports ONLY the control-char strip portion (c-strip),
which is the M10 Phase 1 charter. The length cap (a) and HTML unescape
(b) are tracked separately:

- The 10,000-char cap rarely fires on real article metadata and is
  outside this milestone's KPI yield (no fixture in the 51-corpus
  trips it).
- HTML unescape on metadata slots is a separate fidelity gap (Python
  unescapes; mdrcel may or may not, depending on slot). If it surfaces
  in Phase 2's long-tail triage we patch it then; it is not within
  Phase 1's scope.

`clean_and_trim_metadata` is named to telegraph its eventual extension
to Python's full `clean_and_trim`. Phase 1 lands the strip line only;
Phase 2 (or a tracked follow-up) can grow the helper to include
unescape and truncation when the next bytewise-equivalence audit
demands them.

### Slots covered

For reference, `clean_and_trim_metadata` operates on the following
`Metadata` fields (`src/trafilatura/metadata.rs:95-133`):

- **Strings (apply strip when Some + non-empty):** `title`, `author`,
  `url`, `hostname`, `description`, `site_name`, `date`, `language`,
  `image`, `pagetype`, `license`, `filedate`.
- **String lists (apply strip per entry):** `categories`, `tags`.

`date` and `filedate` are produced internally (today's date or
htmldate's output) and contain only ASCII digits/`-` — the strip is a
no-op on them, but applying it uniformly keeps the helper simple. The
12 string slots + 2 list slots = 14 strip sites, all inside one
function body.

## 7. Risk assessment

| Risk | Likelihood | Mitigation |
|---|---|---|
| **Cc kept-set mismatch.** Off-by-one in the strip-regex's Cc hex ranges (e.g. accidentally including 0x09 TAB by writing `\x00-\x09`, or missing 0x85 NEL by writing `\x7F-\x85` instead of `\x7F-\x84`). | LOW | The full-Unicode equivalence probe (§4) catches any off-by-one across all 1,112,064 codepoints. Unit tests in §6a cover each kept codepoint explicitly as belt-and-braces. The empirical-table source in §1 is the oracle. |
| **NFC reorder bug.** Implementation accidentally runs NFC before strip, causing a Cf that NFC would have left to become a different (still Cf) codepoint and re-arrive on the strip side. | LOW | NFC does not introduce or transform Cc/Cf (§3c, verified empirically). The order is enforced by the existing pipeline shape — NFC is the LAST step in `extract_to_*`; strip is in `line_processing`, which is upstream. Just don't reorder. |
| **Mask removal causes gate regressions.** A fixture that was masked as "deferred" for one reason actually had multiple divergences, and removing the mask exposes a second divergence. | MEDIUM | The five 507b9cdb gate entries cite the U+2063 / Cf leak SPECIFICALLY. The ADR `wrk_docs/m7-deferred/507b9cdb.md` documents the byte counts (`Rust = 121,842 chars, Python = 121,769 chars`; net 73 chars of U+2063 pairs) — strip will close exactly those. If a residual divergence appears, it's a separate bug surfaced by the strip and the implementation must NOT bury it back into the allowlist. Report and triage. |
| **Cc consumers break.** Some downstream caller of mdrcel relies on Cc characters in extracted text. | LOW-NEGLIGIBLE | mdrcel is an extractor for HTML article text. Cc consumers (e.g. binary protocols) would not use it. The behaviour matches Python — any Python user of trafilatura's text output has been getting the strip for years. |
| **Cn regex compile cost / DFA size.** `\p{Cn}` is the largest Unicode general category (>800k codepoints). `regex` compiles this once at startup; the compiled DFA carries the full Cn class. | LOW | `OnceLock<Regex>` lazy-init amortises the compile across the program lifetime. The compile is <10 ms on a cold cache (probe-compile measured). The DFA is <100KB. §6e's perf check guards against any user-visible regression. |
| **Cs reachability.** Rust `&str` is UTF-8-validated, so Cs codepoints (U+D800-U+DFFF) are unreachable. The strip regex deliberately omits the Cs range from its character class for this reason. | LOW | Rust's type system enforces UTF-8; the only way to get a Cs codepoint into the function is `unsafe`. Omitting Cs from the regex keeps the DFA marginally smaller and reflects the actual reachable input domain. |
| **`metadata_url.rs` deferral propagation.** Adding the strip to the metadata_url mirror might shift YAML header / category-tag bytes in a way the existing fixtures didn't tolerate. | LOW | The metadata_url path is exercised by the categories/tags allowlist; if behaviour shifts, the 51-fixture gates surface it on the markdown gate's YAML-header path. Validate by re-running the gates after the metadata_url patch. The new `clean_and_trim_metadata` helper in `extract_metadata` (§5 O1 resolution) runs after `extract_catstags`, so categories/tags are stripped twice — second pass is a no-op but provides belt-and-braces over JSON-LD-sourced entries that bypass `extract_catstags`. |
| **Metadata-slot strip changes byte output unexpectedly.** Adding `clean_and_trim_metadata` to `extract_metadata` flips bytes for some 51-fixture corpus entries whose metadata slots happened to contain Cf/Cc chars. | MEDIUM | This IS the intended fix for 507b9cdb's metadata-slot leak — but other fixtures might surface latent leaks too. The 51-fixture gates will catch any unexpected byte shift; if a gate fails because a slot's bytes changed for a different reason (e.g. our extractor was already producing different metadata bytes vs Python and the strip closes a residual gap), the implementation phase treats it as a bug-finding signal, not a regression to mask. |
