## 1. The design in one paragraph

`mdrcel::extract` keeps its frozen public surface and grows a real implementation: an idiomatic Rust transcription of Mozilla Readability v0.6.0, organised so that every line that adds or removes a descendant text node is ported faithfully against a cited `Readability.js` line, and every line that only affects score-invisible cosmetics is simplified or skipped on purpose. Fidelity is judged by two instruments and two only: the **gold-subset Coverage/Precision bands** and the **the named acceptance cases**, both frozen ground truth owned outside this milestone. Agreement with the Readability-JS oracle is a *reported diagnostic, never a gate* — copying the oracle's output is the explicit anti-goal this whole document is structured to prevent. The work lands as small independently-committable stages, each gated by a named field in the harness's `results.json`, fronted by a hard parser-equivalence blocker that proves our DOM produces the same post-tokenizer token stream as the oracle's before any extraction logic is written. If it doesn't fit in your head, it's wrong.

```
   html bytes ─► html5ever + markup5ever_rcdom ─► dom.rs facade ─► Readability port ─► Extracted
                          │                                              │
            BLOCKER: post-tokenizer token identity            faithfulness = cited Readability.js lines
            vs jsdom 29.1.1 body.textContent                  acceptance = gold bands + the consumer cases
            (7 gold + table-heavy non-gold)                   oracle Jaccard = non-gating diagnostic only
```

## 2. Fidelity anchor — what the harness actually scores

The harness scores `article.textContent` of Readability's final `articleContent` DIV (`run.mjs:212`), tokenized by `metrics.rs::tokens` (whitespace-split → drop-empty → lowercase → NFC), token-set Jaccard = Coverage (sibling §8). `textContent` is a **pure depth-first concatenation of every descendant `#text` node's `data`, with ZERO synthetic inter-element whitespace** — this is the WHATWG DOM `Node.textContent` definition (§2.1). Because the tokenizer collapses whitespace runs, the dominant risk is not losing text but **inserting or omitting a separator that fuses or splits tokens differently from the oracle**.

This partitions the port:

- **Score-invisible (may simplify or skip):** HTML serialization, attribute fixups, class cleaning, `_fixRelativeUris`, `_postProcessContent` cosmetics; title / byline / excerpt / siteName / lang **metadata that is not scored**; whitespace-run normalization (the tokenizer collapses runs anyway).
- **Score-critical (port faithfully, cite the line):** everything that adds or removes a descendant text node — node prepping; unlikely-candidate strip; scoring; top-candidate selection; sibling append; `_prepArticle` (especially `_cleanConditionally`, `_markDataTables`, `_clean`, `_cleanHeaders`, single-cell-table unwrap, empty-`<p>` removal); the `_charThreshold` retry / flag-sieve loop; the final fallback.

### 2.1 The `textContent` semantic basis — jsdom, not JSDOMParser

The architect plan originally anchored `textContent` semantics on `@mozilla/readability`'s bundled `JSDOMParser.js`. **That file is dead code in this harness.** The oracle (`run.mjs`) constructs the DOM with `new jsdom.JSDOM(...)`; under jsdom, `Readability.js:754`'s `this._docJSDOMParser` (set from the `doc` having a JSDOMParser-stamped marker) is **`undefined`**. Readability therefore runs entirely against jsdom's WHATWG-compliant DOM, and `article.textContent` is jsdom's implementation of the standard `Node.textContent` getter — recursive concatenation of descendant `Text` node data, no separators.

**The normative `textContent` definition for this port is the WHATWG DOM `Node.textContent`, as implemented by jsdom 29.1.1** (the version the oracle adapter pins; sibling Oracle HLD §6). `dom.rs::text_content` (§5) MUST reproduce that and only that: raw DFS concat of `#text` data, no inter-element whitespace, no normalization. This is not a theoretical match — Stage 0 carries an **empirical probe** diffing `doc.body.textContent` produced by jsdom 29.1.1 against `html5ever + rcdom` on real snapshots, and the result is a hard gate (§4, M-1).

The fact that `_docJSDOMParser` is falsy under jsdom is load-bearing again in §2.2.

### 2.2 `_setNodeTag` — only the slow branch exists in this harness

`_setNodeTag` (`Readability.js:752-773`) has two branches gated on `this._docJSDOMParser`:

```
752  _setNodeTag(node, tag) {
753    this.log("_setNodeTag", node, tag);
754    if (this._docJSDOMParser) {            // ← FALSY under jsdom → never taken
755      node.localName = tag.toLowerCase();   // in-place "fast" branch: DEAD CODE here
756      node.tagName = tag.toUpperCase();
757      return node;
758    }
759
760    var replacement = node.ownerDocument.createElement(tag);   // SLOW branch — always taken
761    while (node.firstChild) {
762      replacement.appendChild(node.firstChild);                // move every child, in order
763    }
764    node.parentNode.replaceChild(replacement, node);
765    if (node.readability) {
766      replacement.readability = node.readability;              // carry the score-table entry
767    }
768    for (var i = 0; i < node.attributes.length; i++) {
769      replacement.setAttributeNode(node.attributes[i].cloneNode());
770    }
771    return replacement;
772  },
```

Because `_docJSDOMParser` is falsy under jsdom, the oracle **always** executes the slow branch (760-772): create a fresh element, move every child in document order, replace the old node in its parent, **carry the `readability` score-table entry (765-767)**, then clone every attribute, and return the *new* element.

**Design ruling (supervisor B-2):** the port implements **only** the slow branch. The in-place fast branch (754-758) is **explicitly forbidden** — porting it would be an optimization that diverges from observed oracle behaviour. `dom.rs::set_node_tag` therefore: allocates a new element node of `tag`; moves all children preserving order; splices it in place of the old node in the parent's child list; **transfers the score side-table entry and the `_readabilityDataTable` flag from old pointer to new pointer (the Rust analogue of 765-767)**; clones all attributes; returns the new handle. Every caller must use the returned handle (the old one is detached), exactly as the JS does. This retires architect risk #6 ("`_setNodeTag` slow/fast branch"): there is no branch — there is one path, and it is the slow one.

## 3. Dependency decision

**Chosen:** `html5ever` + `markup5ever_rcdom` (rcdom), versions pinned exactly, `Cargo.lock` committed.

Readability needs, at minimum: HTML5-spec parsing of hostile real-world markup; a *mutable* tree (remove / replace / append / `createElement`; attribute get / set / remove); ordered traversal (`firstElementChild` / `nextElementSibling` / `parentNode`; the `_getNextNode` DFS); a `getElementsByTagName` / `querySelectorAll`-equivalent subtree collection; `textContent`; and a `children` (element-only) vs `childNodes` (all nodes) distinction. rcdom is `Rc<Node>` + `RefCell<{NodeData, children}>` + a `Weak` parent link — precisely the mutable, parent-linked, ordered tree the algorithm requires. We build a **thin facade of ~15 operations** over it (§5) and never leak rcdom types past the facade.

Rejected alternatives, with reasons (not future-proofing — the actual constraint each fails):

- **scraper** — read-only query API; cannot mutate the tree Readability rewrites in place.
- **tl** — flat, immutable; no parent links, no mutation.
- **custom arena DOM** — premature; the M8-ring-road antipattern (build the road where it goes, not where it might).
- **kuchikiki** — *viable fallback*, same algorithm, **zero plan change** (mutable DOM + selectors over the same html5ever). Adopted only if rcdom borrow-management blocks Stage 0 (criterion in §6 / M-2).

This is justified against the workspace minimal-dependency ethos using the same reasoning applied to `sha2` and `unicode-normalization`: a correct HTML5 parser + mutable DOM is irreducible for this algorithm; writing our own is the speculative path, not the conservative one. Our code is sync, does no I/O, contains no `unsafe`, is deterministic, and pins exact versions.

### 3.1 Honest dependency-cost record (supervisor M-2)

`markup5ever_rcdom` is published as `+unofficial` and self-describes as *"Basic, unsupported DOM structure for use by tests."* This is recorded here deliberately rather than glossed. It was **verified to build**: `html5ever 0.39` + `markup5ever_rcdom 0.39` + `tendril 0.5`, on `rustc 1.95.0`, edition 2024, pulling **~36 transitive crates** (including `xml5ever` and `web_atoms`, which this port does **not** use). The architect plan's "minimal transitive graph" phrasing is **withdrawn**; the honest figure is ~36 crates, several unused. The rationale stands regardless (the alternative is hand-writing a spec-compliant HTML5 parser), but the cost is stated, not hidden. `Cargo.lock` is pinned and committed. **Version-bump fence:** any bump of `html5ever` / `markup5ever_rcdom` / `tendril` re-runs the §4 parser-equivalence BLOCKER gate before merge, and the bump commit must state that the gate was re-run green (this is the dependency analogue of the sibling Oracle HLD's pin-bump checklist).

## 4. Success metric & anti-inversion (read this before any stage)

This section is normative and overrides any looser phrasing elsewhere in this document. It exists because the single largest design risk in porting a heuristic extractor against a reference implementation is **oracle-mimicry**: tuning the port until it reproduces another extractor's output rather than implementing the specified algorithm.

### 4.1 What "correct" is measured against (supervisor B-3)

The harness (`benchmark/src/score.rs:267-279`) resolves the scoring reference as **gold if present, else Trafilatura**. *Readability-JS is never the reference* (sibling Harness HLD §2). Therefore the **M2 acceptance gates are exactly two things, and nothing else**:

1. **The gold-subset Coverage / Precision bands** — the only frozen hand-decided ground truth, authoritative on the ~6–10 the consumer-critical / SEC-EDGAR gold URLs (sibling §7; gold frozen at `7afaabe`).
2. **The the named acceptance cases from the parent brief** — the named real-world outcomes the library must satisfy.

The brief's "≈95 % vs Readability-JS" and "≈70–80 % vs Trafilatura" figures are **descriptive expectations** of where a faithful port of a faithful algorithm naturally lands. They are **explicitly NOT acceptance gates.** They describe the expected neighbourhood; they do not define pass/fail.

### 4.2 Readability-JS Jaccard is a non-gating diagnostic

Per-URL `Jaccard(crate.text, readability_js.text)` over the `Ok ∩ Ok` set is computed and reported every run as a **diagnostic to localise divergence** — it tells us *where* to go read `Readability.js` next. It is **never** an exit criterion at **any** stage. A high number is not success; a faithful port that disagrees with the oracle because jsdom and html5ever parsed hostile markup differently is *correct* and that residual is documented, not chased.

### 4.3 The mechanised anti-inversion guard (supervisor B-4)

Binding rules, enforced per stage:

- **(a) Cited-line discipline.** Every commit that changes scoring-relevant behaviour cites the exact `Readability.js` line(s) it implements **and** states, in one line, the JS behaviour reproduced. A scoring change with no line citation does not merge.
- **(b) No similarity threshold, ever.** Readability-JS Jaccard is never an exit threshold at any stage. No stage's exit condition may read it.
- **(c) Divergence is closed by a behavioural test, not a number.** A crate-vs-oracle mismatch is resolved **only** by adding a test that asserts the *JS-specified behaviour at the cited line*. "The number went up" is never the fix and never the evidence.
- **(d) Oracle-Jaccard-as-acceptance is a declared ANTI-GOAL.** Anyone proposing it as a gate is, by this document, wrong.
- **(e) Devil's-advocate audit per stage.** Each stage's pre-merge review explicitly audits for *output-matching-without-line-citation* — any change whose justification is similarity rather than a cited line is rejected at review.

This is consistent with the harness's own anti-laundering posture (sibling §2 honest-ceiling; gold-set freeze; no I/O in the crate; scoring lives in the harness). The yardstick in §10 (corpus agreement) **measures** faithfulness; it never **manufactures** it.

### 5.1 Side-table invariant (supervisor m-1)

Per-node Readability state lives in two side structures: a score side-table `HashMap<NodeKey, f64>` (the `node.readability` analogue) and a `HashSet<NodeKey>` for `_readabilityDataTable`. **Invariant: both are POINT-QUERY-ONLY.** Candidate *ordering* never comes from iterating a `HashMap` (non-deterministic order would silently diverge from JS `candidates` array order); ordering comes from a `Vec<NodeRef>` that mirrors the JS `candidates` array exactly. As-built, this is a **structural** guarantee, not a runtime check: both maps are private to `dom.rs`; the only methods that touch them are the point-query getters/setters (`content_score`, `set_content_score`, `has_content_score`, `is_readability_data_table`, `set_readability_data_table`) and the `set_node_tag` transfer — none of which iterate; and there is deliberately no `pub fn` returning an iterator over, or the keys of, either map. (An earlier draft of this section called for a `debug_assert`-based enforcement; that was removed as tautological during implementation — a `HashMap`'s `capacity() >= len()` is always true and would not detect iteration — and replaced with the stronger structural guarantee documented at `dom.rs:649-664`. This neutralises the determinism foot-gun the architect plan implied but did not pin.)

## 6. Stage 0 — DOM facade + parser-equivalence BLOCKER (infrastructure)

**Goal:** stand up `dom.rs` and *prove the DOM substrate is faithful* before one line of extraction logic exists. `lib.rs` still returns `NotImplemented`; the harness is honestly **vacuous** (every corpus URL `not_implemented`) and that is the correct Stage-0 floor.

**Work:** add the pinned deps (§3); implement `dom.rs`; exhaustively unit-test the score-critical primitives. The load-bearing ones: `text_content` produces **no inter-element separator** (§2.1); `inner_text` trims-then-normalizes; `children` vs `child_nodes`; the `_getNextNode` DFS order; `set_node_tag` preserves children + attributes + carries the score-table/data-table entry (§2.2); snapshot lists are true snapshots (later mutation does not retroactively change a returned `Vec`).

### 6.1 Parser-equivalence BLOCKER gate (supervisor M-1, m-4)

This is promoted from "a Stage-0 check" to a **hard blocker on starting Stage 1a**:

> **Require post-tokenizer (`metrics.rs::tokens`) token-sequence identity between jsdom-29.1.1 `body.textContent` and `html5ever + rcdom` for ALL 7 gold snapshots AND the table-heavy non-gold EDGAR / HMRC snapshots, BEFORE Stage 1a may begin.**

Mechanism: a Stage-0 harness-adjacent probe runs jsdom 29.1.1 (the pinned oracle parser) to emit `body.textContent` for each named snapshot, runs the `html5ever + rcdom` facade's `text_content` on the same bytes, pushes both through `metrics.rs::tokens`, and asserts **token-sequence identity** (not merely set or Jaccard — sequence). Raw `textContent` char-level diff is reported for diagnosis when sequences differ.

**Divergence policy:** a failure means **the parser / DOM choice changes (rcdom → kuchikiki, §3), NOT a Stage-1 workaround.** A token mismatch at the substrate level is unfixable downstream and would silently corrupt every later Coverage number; it is therefore a Stage-0 design-decision trigger, never a tuning task. (This is the concrete realisation of architect risk #1 — the highest risk — and the kuchikiki fallback criterion of §3/M-2.)

### 6.2 Stage 0 exit criteria (explicit — supervisor m-4)

All four must hold:
1. New modules compile **warning-clean** (`clippy` + `fmt`).
2. The §6.1 parser-equivalence **BLOCKER gate passes** (token-sequence identity on all 7 gold + the table-heavy non-gold EDGAR/HMRC snapshots).
3. `cargo test` green.
4. **Every corpus URL still `not_implemented`** in `results.json` — the baseline floor is unchanged and the harness is honestly vacuous (no premature "progress").

**Risks:** rcdom `Rc<RefCell>` borrow conflicts (mitigation: the facade returns owned `Handle` clones and never holds a `RefCell` borrow across a mutation; hard fallback = kuchikiki, §3); a wrong `text_content` separator = silent corpus miss — which is exactly what §6.1 exists to catch *before* it can do damage.

## 7. Ordered stages

Common contract for every stage below: compiles warning-clean (`clippy` + `fmt`); `cargo test` green; **never red, independently committable**; the headline metric is the harness mean Coverage over Scored URLs against the §4.1 reference (gold → else Trafilatura) plus per-gold-URL Coverage / Precision in `results.json`; until the crate returns `Ok` for a URL it is `NotScored{CrateNotImplemented}` (`score.rs:258`). Each stage names the **exact `results.json` field** its gate reads (supervisor m-2). No stage's exit reads Readability-JS Jaccard (§4.3(b)).

### 7.1 Stage 1a — parse spine + scoring + single top-candidate (first `Ok`)

`_prepDocument` (strip `style`; `font`→`span`; `_replaceBrs`), `_removeScripts`, the `_grabArticle` prepping walk including unlikely-candidate strip + div→p, `elementsToScore` scoring + ancestor score propagation, `topCandidates` selection, **single top-candidate only — NO sibling append, NO retry loop**. `_prepArticle` is a near-noop here (only the safe `_clean` of object/embed/footer/link/aside + empty-`<p>` removal; **NOT** `_cleanConditionally` / `_markDataTables`). Returns a real `Ok(Extracted { text: text_content(articleContent), ... })`.

**Title pulled forward (supervisor M-4).** `_getArticleTitle` (`Readability.js:572-651`) **and the title half of `_getArticleMetadata`** are implemented in **Stage 1a, not deferred.** Reason, with citations: `_grabArticle` calls `_headerDuplicatesTitle` (`Readability.js:1105`), which computes `_textSimilarity(this._articleTitle, heading) > 0.75` and removes matching headings — so the article title **directly changes the scored body text**. `_articleTitle` is set at `Readability.js:2745`, *before* `_grabArticle` runs at `2747`. Deferring it would make Stage-1a body text wrong by construction. This is a supervisor refinement prompted by the devil's-advocate; metadata is **not** descoped (see §7.4).

TDD: unit fixtures per predicate with hand-traced expected scores.

**Gate (m-2):** the `example.com` gold URL shows `ScoreOutcome::Scored` with high `.precision` in `results.json` (its entry in the per-status `BTreeMap` moves from the `not_implemented`/`NotScored` bucket to `Scored`, and `Scored.coverage` is non-trivial ≈ 14–20). The gov.uk hub gold URL is governed by the revised gate immediately below — its `Scored.coverage`/`.precision` are expected LOW by anti-inversion design, **not** the earlier "high precision / ≈ 6–12" band. Wikipedia is `Scored` but over-includes — **record, do not tune**.

> **§7.1 gov.uk-hub gate (revised — supervisor adjudication 2026-05-18, DA-validated):** `https://www.gov.uk/` moves from `not_implemented`/`NotScored` to `ScoreOutcome::Scored` in `results.json`. Its `Scored.coverage`/`.precision` against the gold are expected to be **LOW by anti-inversion design**: the gov.uk hub was curated (gold.tsv) as a case where Readability **fundamentally fails** — Readability-JS itself yields ~234 nav-link words, none of which is the gold's `<h1>` lead line. The Stage-1a requirement is therefore **faithful Readability behaviour, NOT high precision**: the crate's extracted token-multiset MUST be a **subset of (or equal to) the Readability-JS token-multiset on this URL** (faithful single-top-candidate with sibling-append/`_cleanConditionally` deliberately deferred — the crate may legitimately retain *more* nav chrome than RJS at Stage 1a, never *different invented* content). Convergence toward RJS parity is expected by Stage 2 (`_cleanConditionally`) and Stage 1b (sibling-append). **It is an explicit ANTI-GOAL (§4) for the port to reach the 6–12-word gold band on gov.uk-hub by out-cleaning Readability-JS; any change so motivated is rejected at the §4.3(e) audit.** The 6–12-word gold band remains the frozen *human* ground truth and is reported as the (expected, documented) Coverage residual; it is **not** a Stage-1a pass/fail threshold.

**Exit:** mean Coverage is no longer n/a; `example.com` recorded non-trivial; the gov.uk hub is `Scored` with the (expected-LOW, RJS-subset) Coverage/Precision the revised gate above describes; no URL regresses from `Scored` back to `NotScored`.

**Risks:** scoring float divergence (mitigation: pin float behaviour, hand-traced fixtures — but note §9: the *real* numeric risk is integer coercion, not float); sibling-loop index mutation (deferred to 1b but pre-empted by design: explicit index walk over a refetched children `Vec`, never a live iterator).

### 7.2 Stage 1b — sibling append

Add the sibling-append pass (`Readability.js:1415-1535`): siblings of the top candidate whose score clears the threshold (or which are content-y `<p>`/text) are appended to `articleContent`. Implemented as an **explicit index walk over a refetched children `Vec`** (the JS mutates the sibling list while iterating; a live Rust iterator would be UB-adjacent and divergent).

TDD: fixtures exercising the score-threshold branch, the `<p>`-with-link-density branch, and the node-name/`content`-class branch of 1415-1535.

**Gate (m-2):** the Apple and Wm Morrison English-Wikipedia gold URLs reach their **prose-only Coverage band** — read as `ScoreOutcome::Scored.coverage` for those two gold URLs in `results.json`, within the gold band recorded in `gold/gold.tsv`.

**Risk:** sibling-list mutation during iteration — closed by the refetched-`Vec` index walk above and a test that mutates mid-walk and asserts JS-identical traversal.

### 7.3 Stage 1c — `FLAG_*` retry / flag-sieve loop + fallback

The `_charThreshold` retry loop, the `FLAG_*` flag sieve (strip-unlikelys → weight-classes → clean-conditionally progressively disabled across attempts), `_attempts` bookkeeping, and the fake-`<div class="page">` body fallback when no candidate qualifies.

**Retry-loop semantics — decided (supervisor m-3).** Each attempt **re-parses from the ORIGINAL HTML bytes**, not a cloned tree. ## 8. Regex dialect fidelity (supervisor M-5)

JS `RegExp` and Rust `regex` are **not** the same dialect; a silent class mismatch corrupts scoring invisibly. Binding rules for every ported regex:

- **ASCII `\W`** is written as the explicit literal class `[^A-Za-z0-9_]` (the ECMA-262 `\w` complement). The original plan was `(?-u:\W)` but Rust `regex` rejects `(?-u:\W)` at compile time on `&str` matchers ("pattern can match invalid UTF-8"); the literal-class substitution is dialect-equivalent and pinned by the conformance-table row `tokenize("café") → true` at `regexps.rs:129-146`. Never bare Rust `\W` (which is Unicode-aware and would match differently on non-ASCII).
- **JS `\s`** is written as the explicit class `[ \t\n\r\f\x0B   -     　﻿]` — never bare Rust `\s`. Notes baked into the port's comments: `\v` = U+000B (`\x0B`); **JS `\s` includes U+FEFF, which Rust `regex`'s `\s` excludes** — this is the exact trap the explicit class closes.
- A **per-regex conformance test table** (`input → expected JS-match vs Rust-match`) is committed as a **Stage-0 / Stage-1a oracle** for `REGEXPS` (137-175) and every other ported pattern. Divergence in the table fails the build.

**Verified facts (state as confirmed):** there are **no backreferences and no lookaround** anywhere in `REGEXPS` (verified — so Rust `regex` is expressively sufficient and no `fancy-regex` dependency is needed); and there is **no `Math.round`** anywhere in the scored path (verified — so no JS round-half-to-even vs Rust round-half-away divergence exists to handle).

## 9. Integer coercion fidelity — the real numeric risk (supervisor M-6)

The architect plan flagged f64 scoring divergence as a top risk. The adjudicated position: **float arithmetic is the lesser risk; JS `parseInt` semantics are the real one.** `_getRowAndColumnCount` (`Readability.js:2240-2263`) does:

```
2245   var rowspan = trs[i].getAttribute("rowspan") || 0;
2246   if (rowspan) { rowspan = parseInt(rowspan, 10); }
2249   rows += rowspan || 1;
...
2255   var colspan = cells[j].getAttribute("colspan") || 0;
2256   if (colspan) { colspan = parseInt(colspan, 10); }
2259   columnsInThisRow += colspan || 1;
```

This drives data-table classification (≥10 rows / >4 cols branch), which decides whether a table's text is **kept or stripped** — directly score-affecting on the EDGAR gold.

**Design:** a `parse_int_js(s, radix)` helper faithful to ECMAScript `parseInt`: skip leading ASCII whitespace; optional leading sign; consume the **longest valid digit prefix** for the radix; **trailing garbage is ignored, not an error** (`"12abc"` → `12`); empty / no valid digits → "NaN", which the caller treats exactly as JS does via the `|| 1` / `|| 0` fallbacks at lines 2249 / 2259 / 2245 / 2255. Rust's `str::parse::<i64>()` is **wrong here** (it rejects `"12abc"` and surrounding space) and must not be used for these attribute reads.

**Pinned by tests:** rowspan/colspan classification fixtures on the **FY2019 and FY2025 EDGAR gold** documents (the table-classification boundary cases), asserting the same keep/strip decision the cited JS lines produce.

## 10. Faithfulness verification & anti-inversion (the yardstick)

The port is a **transcription reviewed against named JS lines** (§4.3(a)). The corpus-level `Jaccard(crate, RJS)` over `Ok ∩ Ok` is the **fidelity yardstick** — it *measures* agreement to surface where to read next; it is categorically distinct from inversion, which would *copy* the oracle. The distinction is enforced structurally, not by good intentions:

- Gold is frozen (`7afaabe`); the crate has **no I/O**; all scoring lives in the harness — the crate cannot observe the oracle at runtime even if it wanted to.
- Determinism + single-host (sibling §2.9) — no hidden nondeterministic tie-breaks (§5.1).
- No laundering: divergence-from-Trafilatura is reported as improvement **only** on the gold set (sibling §2 Bug-E2 lesson; `score.rs:247` gate).
- Per §4.3: no similarity number is ever an exit threshold; every divergence is closed by a cited-line behavioural test; the per-stage devil's-advocate audits for output-matching-without-citation.
