## 1. Goal

Two additions to `tests/fuzz_diff.rs`:

1. **Per-page timeout in `safe_extract`.** Wrap the existing
   `catch_unwind` call in a spawned thread with a 30-second deadline.
   Pages that exceed the deadline are recorded as `ExtractOutcome::Timeout`
   — same treatment as panics (recorded, not propagated). This protects
   both `fuzz_diff()` and `broad_sweep()` from pathological inputs
   (catastrophic backtracking, exponential tree walks) hanging the harness
   silently during unattended runs.

2. **`broad_sweep()` test function.** A new `#[test] #[ignore]` entry
   point that processes the 50K broad corpus (`benchmark/broad/...`),
   filtering only outliers into `report.jsonl` and printing a Dice
   histogram + worst-50 summary instead of the M9-era KPI table.

No new crate dependencies. All threading uses `std::thread`,
`std::sync::mpsc`, `std::time::Duration`.

### 2.2 Threading approach

The standard pattern for "run closure with a wall-clock deadline" in Rust
without async or external crates is `std::thread::spawn` +
`std::sync::mpsc::channel` + `mpsc::Receiver::recv_timeout`. Pseudocode:

```rust
use std::sync::mpsc;
use std::thread;
use std::time::Duration;

/// Per-page extraction timeout. Generous: typical pages process in
/// < 100ms; anything > 30s is pathological.
const EXTRACT_TIMEOUT: Duration = Duration::from_secs(30);

fn safe_extract(
    f: ExtractFn,
    html: &str,
    base_url: Option<&str>,
    opts: &mdrcel::Options,
) -> ExtractOutcome {
    let html_owned = html.to_string();
    let base_owned = base_url.map(|s| s.to_string());
    let opts_clone = opts.clone();

    let (tx, rx) = mpsc::channel();

    // Spawn the extraction on a dedicated thread so we can enforce a
    // wall-clock timeout independently of what the extraction does
    // internally (CPU-bound spin, regex backtrack, etc.).
    let handle = thread::spawn(move || {
        let result = panic::catch_unwind(panic::AssertUnwindSafe(move || {
            f(&html_owned, base_owned.as_deref(), &opts_clone)
        }));
        let outcome = match result {
            Ok(Ok(s)) => ExtractOutcome::Ok(s),
            Ok(Err(e)) => ExtractOutcome::Err(format!("{e}")),
            Err(payload) => {
                let msg = if let Some(s) = payload.downcast_ref::<&'static str>() {
                    (*s).to_string()
                } else if let Some(s) = payload.downcast_ref::<String>() {
                    s.clone()
                } else {
                    "<non-string panic payload>".to_string()
                };
                ExtractOutcome::Panic(msg)
            }
        };
        // Ignore send error — receiver may have been dropped on timeout.
        let _ = tx.send(outcome);
    });

    match rx.recv_timeout(EXTRACT_TIMEOUT) {
        Ok(outcome) => {
            // Thread completed within deadline. Join to reclaim resources.
            // The join cannot fail here (the thread already sent its result).
            let _ = handle.join();
            outcome
        }
        Err(mpsc::RecvTimeoutError::Timeout) => {
            // Thread is still running. We cannot kill it (Rust has no
            // thread cancellation), but we detach it — the JoinHandle
            // drops without joining, the thread runs to completion in
            // the background and its resources are reclaimed by the OS
            // when it eventually finishes or the process exits.
            //
            // This is acceptable for a test harness: a handful of
            // timed-out threads dribbling to completion in the background
            // do not affect correctness of subsequent pages (each page
            // gets its own thread + fresh owned data). If many pages
            // time out simultaneously the process accumulates background
            // threads, but at 50K pages the expected timeout count is
            // single-digit — negligible resource pressure.
            ExtractOutcome::Timeout
        }
        Err(mpsc::RecvTimeoutError::Disconnected) => {
            // Sender dropped without sending — thread panicked before
            // reaching the send. Shouldn't happen (catch_unwind covers
            // it), but handle defensively.
            ExtractOutcome::Panic("<thread disconnected without result>".to_string())
        }
    }
}
```

### 2.3 Integration into `process_format`

The existing `process_format` already pattern-matches on `ExtractOutcome`
variants (line 788). The new `Timeout` variant slots in alongside `Panic`:

```rust
let (mdrcel_text_raw, force_shape): (String, Option<ShapeClass>) = match mdrcel_out {
    ExtractOutcome::Ok(s) => (nfc(&s), None),
    ExtractOutcome::Err(msg) => (msg, Some(ShapeClass::MdrcelError)),
    ExtractOutcome::Panic(msg) => (msg, Some(ShapeClass::Panic)),
    ExtractOutcome::Timeout => (
        "<extraction timed out>".to_string(),
        Some(ShapeClass::Timeout),
    ),
};
```

The similarity-metric shortcircuit (line 817-819) already handles
`Panic | MdrcelError` with hardcoded `(0.0, 0.0)`. Extend the pattern:

```rust
Some(ShapeClass::Panic | ShapeClass::MdrcelError | ShapeClass::Timeout) => (0.0, 0.0),
```

Timeout pages land in the `truly_divergent` bucket (Dice 0.0 < 0.80)
and are always recorded as outliers in both harnesses.

### 2.4 Thread-leak analysis

When a page times out, the spawned thread continues running in the
background. This is safe because:

- Each thread owns its data (`html_owned`, `opts_clone`). No shared
  mutable state.
- The `tx` send on a dropped `rx` returns `Err(SendError)`, which is
  handled by `let _ = tx.send(...)`.
- Expected timeout count is single-digit across 50K pages. Each leaked
  thread consumes ~8 MB stack + the owned HTML string. At <10 timeouts,
  that is <100 MB — negligible on a 16+ GB machine.
- The process exits after the test completes, reclaiming all threads.

If the timeout count is unexpectedly high (>50), the summary will make
this visible and the operator can investigate.

## 3. `broad_sweep()` Design

### 3.1 Code reuse — Option A (recommended)

**Recommendation: Option A — extract shared logic into helper functions.**

Rationale:

- **Option B** (parameterised config struct) would require threading a
  config struct through `process_format` and `print_summary`, which
  currently take direct arguments. The config surface is small (paths +
  slicing flag + outlier threshold + summary style) but the two tests
  differ in their *output logic* (what to write, how to summarise), not
  their *processing logic*. A config struct parameterises the wrong axis.

- **Option C** (full duplication) is tempting for a one-off sweep but
  violates the codebase's quality philosophy ("reduce code duplication,
  even if refactoring takes extra effort" per CLAUDE.md). The shared
  processing loop is ~60 lines; duplicating it means two diverging copies
  to maintain if `process_format` or `safe_extract` evolves.

- **Option A** keeps the natural factoring: the per-page processing loop
  (read HTML, look up cache, call `process_format` for each format,
  update stats, print progress) becomes a shared helper. Each test
  function provides its own paths, output sink, and post-loop summary.
  The refactoring is surgical — move ~60 lines from `fuzz_diff()` into
  a helper, call it from both.

**Shared helper signature (sketch):**

```rust
struct PageProcessingConfig {
    corpus_dir: PathBuf,
    progress_every: usize,
}

/// Process all manifest entries against the oracle cache, calling
/// `process_format` for each (sha, format) pair. Returns the count of
/// processed / missing-html / missing-cache pages.
///
/// `on_page` is called after all three formats for a manifest entry have
/// been processed, with the entry index and the per-format stats map
/// (mutable reference). This is the extension point where `fuzz_diff()`
/// decides working-vs-held-out slicing and `broad_sweep()` decides
/// outlier filtering.
fn process_corpus(
    manifest: &[ManifestEntry],
    cache_entries: &BTreeMap<String, CacheEntry>,
    config: &PageProcessingConfig,
    opts: &mdrcel::Options,
    per_format: &mut BTreeMap<&'static str, PerFormatStats>,
    out: &mut dyn Write,
    divergence_buf: &mut Vec<Divergence>,
    record_divergence_fn: impl Fn(usize) -> bool,
) -> (usize, usize, usize) // (processed, missing_html, missing_cache)
```

For `fuzz_diff()`, `record_divergence_fn` is `|idx| idx < WORKING_SLICE_SIZE`.
For `broad_sweep()`, it is `|_| true` (all pages recorded uniformly; filtering
to outliers happens at write time).

**Alternative simpler approach:** rather than the callback, `fuzz_diff()`
can keep its two-map (working/held-out) split by calling the shared helper
twice — once for `manifest[..WORKING_SLICE_SIZE]` with recording on, once
for `manifest[WORKING_SLICE_SIZE..]` with recording off. This avoids the
callback but means two passes over the cache lookup, which is negligible.
Either approach works; the implementer should pick whichever reads most
naturally against the existing code.

### 3.2 Paths

```rust
let cache_path = workspace_path("benchmark/broad/oracle_cache.jsonl");
let manifest_path = workspace_path("benchmark/broad/manifest.jsonl");
let corpus_dir = workspace_path("benchmark/broad_corpus");

```

### 3.3 No working/held-out split

All 50K pages are processed uniformly. There is a single
`BTreeMap<&'static str, PerFormatStats>` (not a working + held-out pair).

### 3.4 Cache integrity

The broad corpus has its own `oracle_cache.jsonl` with the same header
schema. The existing `read_cache_header` / `verify_cache_integrity`
functions work unchanged — they take a `cache_path` argument. The only
difference is that the manifest SHA check must point at the broad
manifest path. This requires a small refactor: `verify_cache_integrity`
currently hardcodes `workspace_path("benchmark/fuzz/manifest.jsonl")`.
Change it to accept a `manifest_path: &PathBuf` parameter. This is a
backward-compatible signature change (the single existing call site in
`fuzz_diff()` passes the fuzz manifest path explicitly).

### 3.5 Outlier filtering

`broad_sweep()` records ALL divergences into `divergence_buf` (same as
the working-slice in `fuzz_diff()`), but writes to `report.jsonl` only
the **outliers**:

- Dice < 0.80 on any format, OR
- empty-vs-nonempty (one side empty, other not), OR
- panic, OR
- timeout.

The filtering happens at write time. The `Divergence` struct already
carries `dice_similarity` and `shape_class`, so the filter is:

```rust
fn is_outlier(d: &Divergence) -> bool {
    d.dice_similarity < 0.80
        || matches!(d.shape_class, ShapeClass::EmptyVsNonempty)
        || matches!(d.shape_class, ShapeClass::Panic)
        || matches!(d.shape_class, ShapeClass::Timeout)
}
```

### 3.6 Outlier report JSONL format

Each outlier line in `report.jsonl`:

```json
{
  "sha": "a1b2c3...",
  "source_url": "https://example.com/page",
  "format": "markdown",
  "dice_similarity": 0.42,
  "jaccard_similarity": 0.35,
  "shape_class": "text-content",
  "mdrcel_len": 1234,
  "oracle_len": 5678,
  "mdrcel_empty": false,
  "oracle_empty": false,
  "panic": false,
  "timeout": false
}
```

This is a DIFFERENT schema from `divergences.jsonl` (simpler — no
diff windows, no diff_shape_hash, no near_tag). Implemented as a
dedicated `OutlierRecord` struct with its own `to_jsonl()` method,
constructed from a `Divergence` + the oracle/mdrcel text lengths.

To compute `mdrcel_len` and `oracle_len`, `process_format` needs to
expose these. Two approaches:

**Approach 1:** Add `mdrcel_len` and `oracle_len` fields to `Divergence`.
These are the NFC + mask-applied string lengths. This is natural — the
information is already computed inside `process_format`; just store it.

**Approach 2:** Compute them at outlier-write time from the raw
`Divergence` fields. But the divergence doesn't currently carry the
full texts — only diff windows.

**Recommendation: Approach 1.** Add two `usize` fields to `Divergence`:

```rust
struct Divergence {
    // ... existing fields ...
    mdrcel_len: usize,   // NEW
    oracle_len: usize,   // NEW
}
```

These are populated in `process_format` from `mdrcel_text.len()` and
`oracle_text_n.len()` (post-NFC, post-mask). The existing
`Divergence::to_jsonl()` can optionally include them (or not — they are
only consumed by the outlier report writer). For `fuzz_diff()`, the
extra fields are harmless dead data in memory.

### 3.8 Histogram bins

The histogram uses seven hardcoded bins. These match the existing
`DICE_NEAR_EQUIVALENT` (0.95) and `DICE_CONTENT_SIMILAR` (0.80)
thresholds from `fuzz_diff.rs`, extended downward:

| Bin | Range | Label |
|-----|-------|-------|
| 0 | `= 1.00` | `byte-equal` |
| 1 | `[0.95, 1.00)` | `near-eq` |
| 2 | `[0.80, 0.95)` | `similar` |
| 3 | `[0.50, 0.80)` | `weak` |
| 4 | `[0.10, 0.50)` | `poor` |
| 5 | `[0.00, 0.10)` | `broken` |
| 6 | panic | `panics` |
| 7 | timeout | `timeouts` |

Bins 0-5 count page-format pairs across all three formats combined.
Bins 6-7 count distinct SHAs (a page that panics on txt + markdown
counts once). Byte-equal pages (bin 0) are those that skip the
divergence path entirely (Dice implicitly 1.0).

Implementation: a `[usize; 6]` array for bins 0-5, plus separate
`panic_count` and `timeout_count` counters. Populated during the
processing loop using the existing `PerFormatStats` accumulators
(byte-equal = `stats.equal`; near-eq/similar from the existing
bucket counters; weak/poor/broken require splitting the existing
`truly_divergent` counter into three sub-buckets).

To avoid changing the `PerFormatStats` struct (which `fuzz_diff()` also
uses), `broad_sweep()` maintains its own histogram counters alongside
the shared stats. Alternative: extend `PerFormatStats` with the three
sub-buckets — `fuzz_diff()` can ignore them. This is cleaner and avoids
drift. **Recommendation: extend `PerFormatStats` with three additional
fields (`weak`, `poor`, `broken`) replacing the single
`truly_divergent` count.** Then `truly_divergent` becomes a derived
value (`weak + poor + broken`). The existing `fuzz_diff()` summary
continues to print `truly_divergent` by summing the three.

### 3.9 Worst-50 list

After the histogram, print the 50 divergences with the lowest Dice
score (across all formats). Sort the full `divergence_buf` by
`dice_similarity` ascending, take the first 50, print a table.

## 5. Test Plan

### 5.1 Unit test for timeout

Add a self-test alongside the existing `elide_literals_collapses_*` tests:

```rust
#[test]
fn safe_extract_timeout_returns_timeout_variant() {
    // A deliberately slow "extraction" function that sleeps for 5 seconds.
    // With a short test-only timeout override, this should return Timeout.
    //
    // Implementation note: the production EXTRACT_TIMEOUT is 30s. For the
    // unit test, either:
    //   (a) Lower the constant to 1s for this test only (not possible with
    //       a const — would need a parameter or thread-local).
    //   (b) Accept a 30s test runtime (too slow for unit tests).
    //   (c) Test the channel/thread mechanism directly with a short timeout.
    //
    // Recommendation: (c). Factor the timeout mechanism into a helper
    // `run_with_timeout<F, T>(f: F, timeout: Duration) -> Option<T>` and
    // test that helper with a 1-second timeout + a 3-second sleep closure.
    // Then `safe_extract` calls `run_with_timeout` with EXTRACT_TIMEOUT.
}
```

**Concrete approach:** extract a generic `run_with_timeout` helper:

```rust
fn run_with_timeout<T: Send + 'static>(
    f: impl FnOnce() -> T + Send + 'static,
    timeout: Duration,
) -> Option<T> {
    let (tx, rx) = mpsc::channel();
    let _handle = thread::spawn(move || {
        let _ = tx.send(f());
    });
    rx.recv_timeout(timeout).ok()
}
```

Then test it:

```rust
#[test]
fn run_with_timeout_returns_none_on_slow_closure() {
    let result = run_with_timeout(
        || { thread::sleep(Duration::from_secs(3)); 42 },
        Duration::from_secs(1),
    );
    assert!(result.is_none());
}

#[test]
fn run_with_timeout_returns_value_on_fast_closure() {
    let result = run_with_timeout(
        || 42,
        Duration::from_secs(5),
    );
    assert_eq!(result, Some(42));
}
```

## 6. Open Questions

### 6.1 Should `EXTRACT_TIMEOUT` be configurable?

Currently proposed as a hardcoded `const` (30 seconds). An environment
variable (`MDRCEL_EXTRACT_TIMEOUT_SECS`) would let operators adjust
without recompilation. Cost: ~5 lines of env-parse boilerplate.

**Recommendation:** hardcode for now. The 30s value is generous
(100x the typical page). If a future run needs adjustment, a const
change + recompile is trivial for a test harness. Environment-variable
configuration adds a knob that nobody has asked for.

### 6.2 Should the histogram bins be configurable?

No. The bins align with the existing `DICE_NEAR_EQUIVALENT` (0.95) and
`DICE_CONTENT_SIMILAR` (0.80) thresholds plus natural downward
extensions. If a future analysis needs different bins, the const arrays
are trivial to edit.

### 6.3 Detached-thread accumulation on mass timeouts

If >50 pages time out, the process accumulates background threads. This
is a theoretical concern — at 30s timeout, 50 stuck threads consume
~400 MB stack space and continue burning CPU. The summary will report
the timeout count, making this visible. If it happens in practice, the
mitigation is to investigate and fix the pathological input class, not
to add thread-kill infrastructure.

### 6.4 `verify_cache_integrity` manifest path parameter

The current implementation hardcodes the fuzz manifest path. The
refactor to accept a parameter is small and backward-compatible. The
alternative (duplicate `verify_cache_integrity` for broad) is worse.
No question here — just flagging the refactor as a required precursor.
