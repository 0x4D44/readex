# HLD — M12: Broad Sweep Confidence Test

**Date:** 2026-05-24
**Author:** Claude (team lead)
**Status:** DRAFT — awaiting review

---

## 1. Goal

Run mdrcel and Python trafilatura on 50,000 diverse real-world HTML pages.
Compare every page-pair. Surface any nasty corner cases: panics, content
extraction failures, or genuinely broken output. This is a confidence sweep,
not a KPI exercise — the output is a ranked list of surprises, not a
percentage to optimise.

### Non-Goals

- Improving the mean KPI. M10/M11 already brought byte-equivalence to 85.8%
  and effective parity to 94.8%. This milestone is about the tail, not the mean.
- Building permanent CI infrastructure. This is a one-off sweep. If it reveals
  systematic patterns, those become fixes in a future milestone.
- Replacing the existing 1500-page M9 corpus. The M9 corpus and harness are
  preserved untouched. The broad sweep has its own isolated data.

## 2. Layout

All broad-sweep data lives under `benchmark/broad/`, separate from the

```

    oracle_cache.jsonl      # Python trafilatura output for all 50K
```

## 3. Corpus Sourcing

### 3.1 Harvester changes

The existing `benchmark/fuzz/harvest.py` already streams Common Crawl WARCs
over HTTPS and applies the M9 content filter. Two additions:

**a) Append mode (`--append`).** When set, the harvester does NOT wipe the
output directory or truncate the manifest. Instead it appends new entries to
the existing manifest and skips SHAs already present on disk. This lets us
run the harvester multiple times with different `--warc-index` values to
accumulate pages from multiple WARC segments.

**b) Domain diversity cap (`--max-per-domain N`).** Track the registered
domain (simple heuristic: last two dot-separated labels of the hostname, or
last three if the second-level is a country-code TLD like `.co.uk`) for each
kept page. Skip pages once a domain has N entries. Default: 50.

### 3.2 Adversarial subset (~2K pages, loose filter)

In addition to the main 50K harvest, harvest a separate ~2,000-page
"weird inputs" batch with a much looser content filter. The purpose is to
stress-test mdrcel on inputs the M9 filter deliberately excludes — exactly
the kind of structural edge cases a confidence sweep should probe.

**Loose filter (applies ONLY to the adversarial subset):**
- Size: 1 KB – 2 MB (same as M9).
- Control-char ratio < 2% (same as M9).
- NO `<p>` tag or visible-text-length requirement. Pages with zero `<p>`
  tags, content entirely in `<div>` or `<table>` structures, near-empty
  pages with a tiny content island — all kept.

Implementation: add a `--loose-filter` flag to `harvest.py` that disables
the `<p>` tag / visible-text check. The adversarial batch is harvested into
the same `benchmark/broad_corpus/` directory and appended to the same
manifest. It is NOT a separate corpus — the sweep processes everything
together. The adversarial pages are simply pages the main filter would have
rejected.

```bash
$PYTHON $HARVEST \
    --warc-index 15 \
    --target-count 52000 \
    --output-dir benchmark/broad_corpus \
    --manifest benchmark/broad/manifest.jsonl \
    --max-per-domain 50 \
    --loose-filter \
    --append \
    --verbose
```

The `--target-count 52000` here means "stop when the TOTAL manifest reaches
52K entries" — the adversarial batch adds ~2K pages on top of the ~50K main
harvest.

### 3.3 Harvesting recipe

Common Crawl publishes ~90,000 WARC segments per crawl. Each segment yields
roughly 10,000-20,000 HTML pages after the content filter. Five segments
should comfortably yield 50,000 pages with the domain cap active.

```bash
PYTHON="benchmark/oracles/trafilatura/.venv/Scripts/python.exe"
HARVEST="benchmark/fuzz/harvest.py"

# Main harvest: 50K pages with the standard M9 content filter.
for i in 10 11 12 13 14; do
    $PYTHON $HARVEST \
        --warc-index $i \
        --target-count 50000 \
        --output-dir benchmark/broad_corpus \
        --manifest benchmark/broad/manifest.jsonl \
        --max-per-domain 50 \
        --append \
        --verbose
done

# Adversarial subset: ~2K extra pages with loose filter.
$PYTHON $HARVEST \
    --warc-index 15 \
    --target-count 52000 \
    --output-dir benchmark/broad_corpus \
    --manifest benchmark/broad/manifest.jsonl \
    --max-per-domain 50 \
    --loose-filter \
    --append \
    --verbose
```

WARC indices 10-14 are chosen to avoid overlap with the existing M9 corpus
(which used index 0). Index 15 is for the adversarial batch. The harvester
stops early when `--target-count` is reached across cumulative runs. Total
network transfer: ~5-6 GB compressed. Total disk: ~2.5 GB HTML.

The existing `benchmark/oracles/trafilatura/run.py` already supports batch
mode. Point it at the broad corpus:

```bash
$PYTHON benchmark/oracles/trafilatura/run.py \
    --batch benchmark/broad_corpus benchmark/broad/manifest.jsonl
```

This produces `benchmark/broad/oracle_cache.jsonl` with the same schema as

## 5. Sweep Harness

### 5.1 New test function

Add `#[test] #[ignore] fn broad_sweep()` in `tests/fuzz_diff.rs`, alongside
the existing `fuzz_diff()`. It reuses the same core functions (`process_format`,
`safe_extract`, `classify_shape`, similarity computation) but with different
paths and different output focus:

```rust
#[test]
#[ignore]
fn broad_sweep() {
    let cache_path = workspace_path("benchmark/broad/oracle_cache.jsonl");
    let manifest_path = workspace_path("benchmark/broad/manifest.jsonl");
    let corpus_dir = workspace_path("benchmark/broad_corpus");
    let report_path = workspace_path("benchmark/broad/report.jsonl");
    // ...
}
```

### 5.2 Differences from `fuzz_diff()`

| Aspect | `fuzz_diff()` | `broad_sweep()` |
|--------|--------------|-----------------|
| Corpus size | 1,500 | 50,000 |
| Working/held-out split | Yes (1000/500) | No — all pages processed uniformly |
| Divergence JSONL | Writes all working-slice divergences | Writes ONLY outliers (Dice < 0.80 or empty-vs-nonempty or panic) |
| Summary output | KPI table + top-30 groups | Outlier report: panic count, empty count, Dice distribution histogram, worst-50 pages by Dice |
| Per-format stats | Same `PerFormatStats` with Dice accumulators | Same struct, reused |

### 5.3 Outlier report

The sweep writes `benchmark/broad/report.jsonl` with one line per outlier
(Dice < 0.80 on any format). Each line includes:

```json
{
  "sha": "...",
  "source_url": "...",
  "format": "markdown",
  "dice_similarity": 0.42,
  "jaccard_similarity": 0.35,
  "shape_class": "text-content",
  "mdrcel_len": 1234,
  "oracle_len": 5678,
  "mdrcel_empty": false,
  "oracle_empty": false,
  "panic": false
}
```

The summary (printed to stderr) includes:

1. **Total pages processed** and per-format byte-equality rate.
2. **Panic count** — any mdrcel panic is a bug. List the SHAs.
3. **Empty-vs-nonempty count** — pages where one side extracted content and
   the other didn't. List the SHAs and directions.
4. **Dice distribution histogram** — bucket counts at 0.0-0.10, 0.10-0.20,
   ..., 0.90-0.95, 0.95-1.00, 1.00 (byte-equal). This shows the shape of
   the quality distribution, not just the mean.
5. **Worst 50 pages by Dice** — the triage target.

### 5.4 Per-page timeout

At 50K pages running unattended, a single pathological input (regex
catastrophic backtracking, exponential tree traversal) could hang the
entire sweep silently.

`safe_extract` (`fuzz_diff.rs:693`) already catches panics via
`catch_unwind` but has **no timeout**. The broad sweep adds one:

Wrap the `catch_unwind` call in a spawned thread with a 30-second
deadline. If the thread doesn't return in time, record a new
`ExtractOutcome::Timeout` variant — same treatment as `Panic` (recorded
as an outlier, not propagated). The timeout is generous: typical pages
process in < 100ms; anything taking > 30s is pathological.

```rust
enum ExtractOutcome {
    Ok(String),
    Err(String),
    Panic(String),
    Timeout,          // NEW
}
```

The timeout variant is added to the existing `safe_extract` function (not
a broad-sweep-only change) so the regular `fuzz_diff()` test benefits too.
Timeout pages are classified as `ShapeClass::Timeout` (new variant) and
always land in the `truly_divergent` Dice bucket with score 0.0.

### 5.5 Crash resilience

At 50K pages, a single panic could abort the entire run. `safe_extract`
already catches panics via `catch_unwind`. The broad sweep reuses this —
panics are recorded as outliers, not propagated.

The Python oracle runner (`run.py:428-444`) already has per-page
try/except, recording crashes as `__TRAFILATURA_ORACLE_ERROR__` sentinels.
No changes needed on the oracle side.

## 6. Success Criteria

This is a confidence sweep. "Pass" means:

1. **Zero panics** on the full 50K corpus.
2. **Empty-vs-nonempty rate** (mdrcel empty, Python non-empty) under 2%.
   Both extractors legitimately return empty on some pages (boilerplate-only,
   login walls, etc.); a small asymmetry is expected.
3. **Dice < 0.50 on fewer than 1%** of page-pairs — truly broken output.
4. **No systematic pattern** in the low-Dice tail that suggests a fixable
   class of bugs. (Individual snowflake pages with Dice 0.3 are acceptable;
   200 pages with the same failure mode are not.)

Any criterion that fails triggers a triage-and-fix round, not a milestone
closure.

---

### 5.6 Oracle cache parsing at scale

The oracle cache at 50K entries will be ~500–800 MB on disk. The Rust
harness's `read_cache_entries()` parses it line-by-line into a `BTreeMap`.
This will take ~30–60 seconds and ~500 MB RAM at startup. Acceptable for a
one-off sweep on a machine with ≥ 16 GB RAM. Not suitable for a laptop with
8 GB without increasing swap.

No code change needed — just a note for the operator.

## 8. Execution Sequence

```
1. Implement harvester additions           (~30 min)
2. Implement broad_sweep() test            (~30 min)
3. Harvest 50K pages                       (~1-2 hours, network-bound)
4. Run Python oracle batch                 (~2-3 hours, CPU-bound)
5. Run broad_sweep()                       (~15-30 min)
6. Triage outliers                         (depends on findings)
```

Steps 3-4 are long-running but unattended. Step 5 produces the outlier
report. Step 6 is where the value is.
