## TL;DR

`mdrcel` is a Rust content-extraction crate whose correctness is defined by **differential testing against two reference implementations**, because "main content" has no formal spec. This brief covers the first concrete build deliverable: two subprocess **oracle adapters** that wrap **Trafilatura** (Python) and **Mozilla Readability** (`@mozilla/readability`, JS) behind one frozen I/O contract, pinned for reproducibility, so the Rust harness can shell out to them uniformly. Once the contract is frozen, the two adapters are independent and can be built in parallel.

---

## Locked decisions (from the 2026-05-16 design discussion)

These are settled. Do not relitigate them in the adapter build; relitigate in the methodology note if needed.

1. **Oracle hierarchy is asymmetric, not co-equal.** The parent brief's "ground truth = union of agreement" framing is **superseded**. The ordering is:

   ```
   gold set  >  Trafilatura (oracle #1)  >  Readability-JS (oracle #2)
   ```

2. **Trafilatura is oracle #1 — the one we follow.** State of the art per published benchmarks (Barbaresi 2021, ACL). On a disagreement with Readability and no gold-set entry, the target output is Trafilatura's.

3. **Readability-JS is oracle #2 — a guardrail, not a co-equal voter.** Its role is to *raise a flag*, specifically: when it extracts substantially **more** than Trafilatura on a well-structured page, that is a signal Trafilatura may have truncated (a known Trafilatura failure mode). Flagged divergences get a human/gold-set look — they do not silently resolve to Trafilatura.

4. **The gold set is NOT an oracle.** It is hand-decided ground truth for ~6–10 the consumer-critical / SEC-EDGAR URLs, authoritative *only* on those URLs — because that is precisely where Trafilatura is **not** benchmark-validated (the published benchmarks are news/blog-heavy; EDGAR 10-Ks are not represented). The gold set is defined in the methodology note, not here.

5. **Honest ceiling — state it, don't hide it.** Outside the gold set, mdrcel's *measurable* quality is capped at Trafilatura's. By construction we are building "Trafilatura, in safe/fast Rust." If mdrcel is genuinely better than Trafilatura somewhere, the harness scores that as *error*, not improvement. "Exceeds the state of the art" is only provable on the gold set. v1.0 ambition, stated honestly: *match Trafilatura's algorithm in Rust, plus a small set of cases we've proven we're better.*

## Why the oracles come first

Per the parent brief: the first deliverable is the harness, before any extraction code. The oracles **are** the harness's definition of "correct." For a heuristic problem with no spec, two well-validated reference implementations are the only thing that encodes "what correct looks like" in a form a machine can check at scale across 50+ URLs. The crate initially outputs "not implemented" for every URL; the oracle baseline is the floor everything else is measured against. Get the oracles wrong and every downstream number is meaningless.

## The I/O contract (FREEZE THIS BEFORE PARALLEL WORK)

This is the seam. Both adapters MUST conform exactly. This is the single most important section.

### Invocation

```
<oracle-entrypoint> <path-to-snapshot.html> [--base-url <URL>]
```

- HTML is read from the file path, **bytes as given** — encoding detection is explicitly the caller's responsibility (matches the parent brief's scope boundary). The adapter does not fetch anything; **no network access**.
- `--base-url` is optional; used only for canonical-URL/relative-link resolution where the underlying tool supports it.

### Output

A **single JSON object on stdout**, UTF-8, no other stdout noise (logging goes to stderr):

```json
{
  "oracle": "trafilatura | readability-js",
  "oracle_version": "<exact upstream version string>",
  "title": "string | null",
  "text": "string",
  "html": "string | null",
  "word_count": 0,
  "canonical_url": "string | null",
  "language": "string | null",
  "ok": true,
  "error": "string | null"
}
```

- `text` is the extracted plain text (the primary comparison surface).
- `html` is sanitised content-only HTML if the tool produces it cheaply; `null` otherwise (not v1-gating).
- `language` is best-effort; `null` is acceptable (matches parent-brief scope).
- `oracle_version` makes every result row self-describing — a version bump shows up as a diff in committed baselines.

### Word-count definition (shared, single source of truth)

`word_count` MUST be computed by **one shared definition applied to `text`**, identical across both oracles and (later) the crate, or coverage numbers are not comparable. Definition: split `text` on Unicode whitespace (`\p{White_Space}`), count non-empty tokens. Specify it once in the methodology note; both adapters compute it the same way (do **not** trust each tool's own internal word count).

### Exit codes & the ok/error split (this encodes Bug E2's lesson)

- Exit `0` + `"ok": true` → the extractor ran and produced output, **even if the output is empty/short**. "Found little content" is a valid result, not a failure.
- Exit non-zero + `"ok": false` + `"error": "<message>"` → the adapter itself failed (broken env, unreadable input, tool crash).
- Never conflate "extractor legitimately found almost nothing" with "adapter blew up." The parent brief's Bug E2 was exactly a catch-all error hiding distinct states; do not reproduce it here.

### Determinism

Same input → byte-identical JSON across runs and machines. No network, no wall-clock or locale-dependent values in emitted fields.

## Reproducibility / pinning (non-negotiable)

Unpinned subprocess oracles make the entire differential signal non-reproducible across machines and time. Requirements:

- **Trafilatura adapter:** exact pinned version (e.g. `requirements.txt` with `==`, or a `uv`/lockfile). Commit the lock. Record the Python version in the adapter README and emit the Trafilatura version as `oracle_version`.
- **Readability-JS adapter:** pinned `@mozilla/readability` **and** `jsdom` in `package.json`; commit `package-lock.json`. Record the Node version in the README and emit the Readability version as `oracle_version`.
- Each adapter's README states the **exact upstream version it represents** and how to reproduce the environment from the committed lockfiles.
- Containerisation (Docker per oracle) is the most robust option and is **recommended**, but **not mandated** for v1 — document-and-lock is the floor; container is the stretch. Decision deferred to build time (see Open Questions).

## What "done" means for this deliverable

Both adapters:

1. **Honour the contract exactly** — verified by a tiny contract test: feed a known snapshot (start with `example.com`), assert the JSON schema, the `ok` semantics, and a non-empty `text` for a content page.
2. **Pinned + locks committed + runtime versions recorded** in the adapter README.
3. **Deterministic** — two runs on the same snapshot produce byte-identical JSON.
4. **One-command smoke test** per oracle (`run.py <sample>` / `node run.mjs <sample>`), documented in the README.
5. **README** documents setup, the exact upstream version represented, and reproduce-from-lock steps.

**Stretch (recommended): a contract-conformance harness both adapters must pass** — a small "oracle of the oracles" that feeds a fixed set of snapshots and validates schema + determinism + exit-code semantics for either adapter. This is itself a differential-testing pattern applied one level down; cheap, high-leverage, and it makes "the contract is frozen" a checkable claim rather than a hope.

## Parallelization plan

Dependency graph:

```
[methodology note + this brief: FREEZE CONTRACT]   <-- the seam, gates everything
        |
        ├──> Trafilatura adapter      ─┐  independent;
        ├──> Readability-JS adapter   ─┤  can be built concurrently
        └──> corpus snapshot tool +   ─┘  (loosely coupled; feeds the adapters)
             URL list
        |
        v
   Rust runner (needs contract + ≥ stub oracles)
        |
        v
   compare/report (needs runner output schema)
```

- The two oracle adapters are the cleanest parallel pair — different runtimes, zero shared code. They may be built by separate sessions/subagents once the contract is frozen.
- Corpus snapshot tooling is a third, loosely-coupled stream; it feeds the adapters' tests but doesn't block their construction (build adapters against 2–3 sample snapshots, backfill the full corpus).
- **Honest note:** the adapters are ~1-day units; the parallelism win is modest and carries coordination cost. The real prerequisite and bottleneck is **freezing the contract**, not running two builds at once. Do not over-engineer the concurrency.
