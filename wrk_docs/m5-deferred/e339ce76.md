# DEFERRED: `e339ce76eb1cba73.html` (FRED — St. Louis Fed economic data page)

## Status

Deferred to **M6** (or later). Listed in
`tests/trafilatura_markdown_gate.rs::DEFERRED_KNOWN_DEFECT`.

This is **not** an allowlist entry — mdrcel is the buggy side. Python's
output is the faithful body; Rust emits a `<noscript>` JS-disabled stub
in its place. Documenting the diagnosis here so M6 can pick up the work
without re-discovering the root cause.

## Why mdrcel is wrong

mdrcel's `extract_to_markdown` emits ~5187 chars beginning with
"Please enable JavaScript..." (a `<noscript>` fallback). Python's
`trafilatura.extract(output_format='markdown')` emits 2907 chars of
substantive economic-data narrative beginning with "An official website
of the United States Government".

Both engines agree the body has more `<table>` than `<p>` (16 > 4 in
Rust, 16 > 10 in Python), so both route through cascade Branch 6 and
both produce a readability-arm winner of ~5128 chars that INCLUDES the
noscript stub. The divergence happens at the **jusText override gate**:

| Side    | jusText output | jusText override gate (`len_text > 4 * jt_len`) | Outcome |
|---------|---------------:|-------------------------------------------------|---------|
| Python  | 2907 chars     | `5128 > 4 * 2907` = `5128 > 11628` = **FALSE**  | jusText REPLACES winning body → 2907-char clean output |
| mdrcel  | **487 chars**  | `5128 > 4 * 487`  = `5128 > 1948`  = **TRUE**   | jusText does NOT override → winning body (with noscript stub) stays → 5187 chars |

Root cause: **mdrcel's jusText classifier returns 487 chars on the FRED
body where Python's returns 2907.** The override gate then correctly
implements its spec on both sides; the divergence is upstream in the
paragraph-classification arm itself.

## What deferred work is needed

Trace Python's `justext.core.classify_paragraphs` vs mdrcel's
`justext_core::classify_paragraphs` on the FRED body and reconcile the
classifier disagreement. The candidate divergence surfaces (any of these
could be the root cause individually or in combination):

1. **Stopword scoring** — Python's `justext` ships English stopword
   lists (`justext.utils.get_stoplist`); mdrcel's port may use a smaller
   list or a different lookup path. Stopword density is a key signal in
   `classify_paragraphs` for "good" vs "bad" verdicts.
2. **Boilerplate-vs-text threshold** — `justext` classifies each
   paragraph as `good` / `bad` / `neartext` / `short` based on length
   and link-density heuristics with specific magic numbers (length 80,
   link density 0.2, etc.). Mismatched constants would shift the cut.
3. **Multi-pass refinement** — `justext.core` runs a second pass that
   promotes `neartext` paragraphs adjacent to confirmed `good`
   paragraphs. A bug in the neighbour-window walk would suppress legit
   text.
4. **Link-density computation** — the FRED body has heavy nav chrome
   surrounding the data narrative; if mdrcel's link-density counts text
   inside `<a>` tags differently from Python, the cut moves.

Suggested investigative recipe (M6 Stage X):

1. Dump both engines' per-paragraph classification verdicts on the FRED
   body (paragraph index → text length → link density → stopword
   density → final class). Diff the verdicts. The first diverging
   paragraph identifies the misclassified heuristic.
2. Once the heuristic is named, port-trace Python's exact code path for
   that heuristic and the corresponding mdrcel function. Apply the
   smallest fix that converges classifier output on the FRED body.
3. Re-run the full markdown gate. Expected outcome: 1 fixture flips
   green (FRED), 0 regressions (the jusText fix should not move any
   already-passing fixture if the classifier was already correct on
   them, but verify).

Suggested milestone: **M6** (or a follow-on M5 patch if Arthur wants to
extend M5's scope rather than wait).

## What this ADR deliberately does NOT claim

This ADR does NOT claim mdrcel's jusText port is generally defective —
the port is byte-equivalent to Python on most corpus fixtures.
Classifier divergences are body-specific (different stopword density,
different link-density patterns produce different verdicts). FRED is
the first corpus fixture where the classifier disagreement is severe
enough to flip the override-gate decision. There may be others lurking
in larger corpora; the M6 fix should generalise rather than special-case
FRED.
