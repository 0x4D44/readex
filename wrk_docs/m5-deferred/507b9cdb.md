# DEFERRED: `507b9cdbe036bf58.html` (Apple FR — French Wikipedia Apple Inc. article)

## Status

Deferred to **M6** (or later). Listed in
`tests/trafilatura_markdown_gate.rs::DEFERRED_KNOWN_DEFECT`.

This is **not** an allowlist entry — mdrcel emits `<sup
class="reference">` inline-citation markers in a structurally different
shape from Python's `xmltotxt` and the fix is straightforward port work.

## Why mdrcel is wrong

The French Wikipedia Apple Inc. article uses inline numbered references
of the form:

```html
... text <sup class="reference">[19]</sup> more text ...
```

Python's `xmltotxt`, when rendering `<sup class="reference">[N]</sup>`
references, emits each one as its **own paragraph break**:

```
... text

[19]

 more text ...
```

mdrcel's `xmltotxt` renders them **inline**:

```
... text [19]  more text ...
```

The harness's coarse char-count classification reports the divergence
as a 1-byte net diff inside the 122,740-char fixture, but that masks
~145 bytes of structural divergence — local diffs (Python's extra
`\n\n` paragraph breaks vs Rust's inline single space) **cancel out**
at the char-count level. There are ~30 reference markers in the
article; each one contributes a +/-5-byte local diff that nets to
near-zero in aggregate.

This is the same brief-misclassification class as Stage 6h documented
("Brief misclassification: the harness reports the NET diff; local
divergences can compensate to a near-zero net"). The Stage 6h doctrinal
note flagged 507b9cdb specifically as "structurally divergent despite
the 1-byte net" and deferred it pending a real `<sup
class="reference">` fix.

## What deferred work is needed

Align mdrcel's `<sup class="reference">` rendering in `xmltotxt`
(`src/trafilatura/output.rs`) to Python's paragraph-break treatment.
Specifically:

1. **Identify Python's render path.** Trace `xml.py`'s `process_element`
   for `<sup>` elements: when `class` contains `reference`, Python
   emits the `<sup>` as a standalone paragraph
   (`element.text + '\n\n'`) rather than inline. The class-substring
   check is the discriminant.
2. **Mirror in Rust.** mdrcel's `replace_element_text` (or equivalent
   `<sup>` branch in `output.rs`) needs the same class-substring
   discriminant. When `<sup class="reference">` is the match, prepend
   `\n\n` to the element's text and append `\n\n` to its tail; when
   `<sup>` lacks the `reference` class, keep the existing inline
   rendering (M5 Stage 4 bold-marker behaviour for non-reference
   `<sup>` is correct and should not regress).
3. **Verify on the Apple FR fixture.** Re-run the markdown gate; the
   507b9cdb fixture should flip green, and no other fixture should
   regress (only fixtures with `<sup class="reference">` are affected,
   and the change is additive — non-reference `<sup>` is unchanged).
4. **Re-check the entity / reference class on other Wikipedia
   fixtures.** The 86df4d2e ADR (M5 allowlist) is about a DIFFERENT
   `<sup>` quirk (markdown-path structural fracture); this fix should
   not affect it. Confirm post-fix.

Suggested milestone: **M6** (small targeted fix, expect ~30 LOC + 1-2
new lib tests; could land as a single Stage in M6's opening atom).

## What this ADR deliberately does NOT claim

This ADR does NOT claim the existing `<sup>` inline rendering is wrong
for all use cases — only specifically for `<sup class="reference">` on
Wikipedia-style citation markers. The Stage 4 bold-marker work
(`<hi rend="#sup">` → markdown) is correct and orthogonal. The M6 fix
should add the `class="reference"` discriminant as an ADDITIVE check,
not replace the existing rendering.
