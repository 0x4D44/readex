# M5 Allowlist ADR — `9c64e8e3fcd844d4.html` (Rust blog index, Python under-extracts to description-only)

**Bucket pre-allowlist:** `content-mismatch` (Rust = 17,031 chars, Python = 162 chars; first-diff at byte 0).
**Verdict:** anti-inversion-clean — Python under-extracts a blog **index/listing page** to just the meta description sentence, dropping the 31 KB post-listing table that is the page's actual content.

## Why Python is wrong

The fixture is `https://blog.rust-lang.org/` — the official Rust blog's landing page. Its substantive content is a `<table class="post-list collapse w-100 ...">` containing ~year-by-year listings of post titles + dates (the table renders as rows of `Date | <a href="...">Post title</a>`). The page also has, in a `<header>` block, two short `<p>` paragraphs of description ("This is the main Rust blog. Rust teams use this blog to announce major developments in the world of Rust." + a "See also: ..." callout — 162 chars total).

Python `trafilatura.extract` returns **exactly those 162 description chars**, dropping the entire post-listing.

### Trace through Python's cascade

1. **`_extract` (`main_extractor.py:567-617`)** — `BODY_XPATH` (`xpaths.py:13-54`, 5 expressions) misses the page entirely: none of the `article`/`section`/`div`/`main` selectors match (the `<table>` is wrapped only in `<main class="container ...">` without any of the `content-main`/`article-body`/`post-content`/... class names the XPaths key on). `_extract` returns an empty `result_body`.

2. **`extract_content` (`main_extractor.py:620-640`)** — sees `len(result_body) == 0` (line 633) and calls `recover_wild_text` (`main_extractor.py:512-530`).

3. **`recover_wild_text`** — searches `.//blockquote|.//code|.//p|.//pre|.//q|.//quote|.//table|...` and pulls every wild `<p>` and `<table>` from the tree. The 2 description `<p>` elements come through, AND the `<table class="post-list">` is found.

4. **The link-density filter (`htmlprocessing.py`, called from `prune_unwanted_sections` -> `delete_by_link_density` AND from the table-prune at `main_extractor.py:552-556`)** — calls `link_density_test_tables` (in `htmlprocessing`) on the `<table>`. The post-listing table has link text 11,279 chars of total 14,689 chars — **76.8% link density**. Debug log: `DEBUG:trafilatura.htmlprocessing:table link text: 11279 / total: 14689`. Above the threshold, so the table is deleted.

5. **`compare_extraction` + readability fallback (`external.py`, called from `core.py:112-120`)** — readability_lxml's algorithm then picks the same `<table>` as its top candidate (`DEBUG:trafilatura.readability_lxml:Top 5: table 59.22869834570082`) but ALSO drops it via its OWN link-density filter (`DEBUG:trafilatura.readability_lxml:Removed 0.000 table with weight 25 cause it has too many links 0.768 for its weight 25`).

6. **The baseline rescue (`core.py:122-125`)** — `len_text=161 < min_extracted_size=250 (settings.cfg:26)` SHOULD trip the baseline rescue. It does — `baseline(deepcopy(tree_backup))` is called — but `baseline` (in `baseline.py`) finds **the same 161 chars** (the description `<p>` elements) because the table is consistently rejected by every algorithm in the cascade.

The result: 162 chars (161 + final newline), missing every post link.

### What Python is "right" about

The 76.8% link density is, in isolation, a perfectly reasonable signal for "this is a navigation widget, not article content". Trafilatura is tuned for **article extraction** — long-form prose with the occasional inline link, NOT listing/index pages where the links ARE the content. For its target use case (extract the article from a news/blog page), Trafilatura's filter is correct.

But on a blog **index** page, the listings ARE the content — there is no other article. Python's output (description sentence only) is a silent failure: anyone using `trafilatura.extract` to scrape "what's on the Rust blog homepage" loses every post link and every post title.

## What mdrcel does instead

mdrcel's pipeline (Stage 1b cleaning + Stage 2 `_extract`/`recover_wild_text` ports) DOES preserve the post-listing table. It emits 17,031 chars of markdown — the description paragraphs plus the full listing of every post visible on the page (year headers + dates + post titles + per-post URLs).

The structural-token gate (`trafilatura_extract_content_gate`, BLOCKER #3) does not flag this fixture either: the post-listing IS the structural body, faithfully extracted.

## What evidence supports this verdict

1. **Manual inspection of the rendered page** — every link in the post-listing is a real Rust blog post URL (`/2026/05/04/outreachy-2026-may/` etc.). The listing is the page's primary value.

2. **Python's own debug log shows the table being deleted twice** (once at `main_extractor.py:554` via `link_density_test_tables`, once again by `readability_lxml` as the chosen candidate). Two independent algorithms both reject it for being "too link-heavy" — the rejection is structural, not accidental.

3. **Baseline rescue confirms the under-extraction is consistent** — `baseline.baseline(tree)` independently returns 161 chars (the description). Python's full cascade has no path to recover the listing.

4. **Rust output structure is honest**: post listings rendered as markdown tables/lists; URLs preserved as `[Title](url)` markdown links; year headers preserved as `## Posts in YYYY`. No hallucination, no over-extraction — just the content the page actually displays.

5. **`min_extracted_size` rescue does not fire helpfully** — the description IS just barely below threshold (161 < 250), the rescue runs, but `baseline` finds the same description. The Stage 6c "baseline-rescue" fix in mdrcel that recovered other under-extraction cases does NOT help here because Python's baseline is also stuck on the description.

## What this allowlist deliberately does NOT claim

This ADR does NOT claim mdrcel is *better at article extraction* than Python — only that on listing/index pages, Python's link-density filter is over-aggressive enough to silently drop the page's actual content. The allowlist is one fixture, not a class.

## Anti-inversion check

HLD §4 forbids out-cleaning Trafilatura — i.e. mdrcel must not silently OVER-clean content to ape Python's heuristics. The inverse — deliberately deleting the post-listing table to match Python's empty result — would be inversion in the worst sense: dropping 16.8 KB of valid, structured content (post titles + URLs) to match a known oracle blind spot.

Allowlisting `9c64e8e3` as Python-under-extract is the honest verdict: mdrcel preserves the listing, the corpus records why the byte-for-byte gate cannot pass here, and the gate's substantive-pass count stays honest.
