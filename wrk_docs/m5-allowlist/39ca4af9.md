# M5 Allowlist ADR — `39ca4af9befa0524.html` (Rust blog post, html5ever HTML5-spec `<pre>` leading-newline strip)

**Bucket pre-allowlist:** `content-mismatch` (Rust = 20,595 chars, Python = 20,602 chars; first-diff at byte 1106).
**Verdict:** parser-level spec divergence — html5ever follows HTML5 §13.2.5.51 (start-tag dispatch for `<pre>` / `<listing>` / `<textarea>`) and ignores a `U+000A LINE FEED` token immediately following `<pre>`. lxml's `HTMLParser` does NOT implement that spec rule and preserves the leading newline as `<pre>.text`. mdrcel should not undo html5ever's spec compliance to bug-for-bug-match lxml's non-compliance; that would be anti-inversion-violating against the W3C HTML5 spec.

## Why Python is (per the HTML5 spec) wrong

The fixture's source HTML at the divergence point is:

```html
<p>For example, here's what you could see for <code>cargo info cc</code>:</p>
<pre>
<code class="language-text"><b><span style="color:#0A0">cc</span></b> <b><span style="color:#0AA">#build-dependencies</span></b>
...
</code></pre>
```

Note the literal `\n` between `<pre>` and `<code>` (the source HTML is pretty-printed across multiple lines).

### HTML5 spec — §13.2.5.51, In body insertion mode

The spec states (verbatim):

> A start tag whose tag name is one of: "pre", "listing"
>   If the stack of open elements has a p element in button scope, then close a p element.
>   Insert an HTML element for the token.
>   **If the next token is a U+000A LINE FEED (LF) character token, then ignore that token and move on to the next one.** (Newlines at the start of pre blocks are ignored as an authoring convenience.)

So per the spec: `<pre>\n<code>...</code></pre>` MUST be parsed as if it were `<pre><code>...</code></pre>` — the leading `\n` is dropped at tokenisation time.

### Empirical parser behaviour

Probed with lxml 6.0.2 (Python 3.13) and html5ever (via mdrcel's `Dom::parse`):

```python
from lxml import html
tree = html.fromstring("<html><body><pre>\n<code>X</code></pre></body></html>")
pre = tree.xpath('//pre')[0]
print(repr(pre.text))   # '\n'  <-- lxml KEEPS the leading newline (spec-non-compliant)
print(len(pre))         # 1
```

html5ever (mdrcel side) strips the leading `\n` per spec — `<pre>.text == None` and the `<code>` element is the only child.

### Downstream effect on the markdown gate

Trafilatura's `convert_tags` renames `<code>` (via `is_code_block_element` → `handle_code_blocks`), and `xmltotxt`'s `<code>` formatting branch (xml.py:270-274) fences the content with ```` ``` ```` when `elem_text.contains('\n')`. With lxml's preserved leading-`\n`, an empty wrapper-only `<code>\n</code>` slips through the cascade and surfaces as ```` ```\n``` ```` in Python's markdown output — a 7-byte empty code fence with nothing between the open and close markers, immediately preceding the bolded content emitted from the `<b>` formatting children. mdrcel's html5ever-parsed tree has no such leading `\n`, so no empty fence is emitted; the bolded content immediately follows the colon.

This is NOT an extraction defect — it's a downstream artefact of lxml's spec non-compliance. The empty fence carries no information; it's an artefact of the leading whitespace bubbling through `convert_tags` + `xmltotxt`.

### What mdrcel does instead

mdrcel parses per HTML5 spec, so `<pre>` does NOT carry the leading `\n` as text, and `xmltotxt` does not emit an empty open/close fence pair. The bolded `<b>` formatting children render directly:

```
:\n\n**cc** **#build-dependencies**\n...
```

vs Python's:

```
:\n\n```\n```**cc** **#build-dependencies**\n...
```

The 7-byte difference at byte 1106 is the empty ```` ```\n``` ```` fence pair — no semantic content gained or lost.

## What this allowlist deliberately does NOT claim

This ADR does NOT claim mdrcel is generically better than Python at code-block rendering. It claims that:

1. html5ever follows HTML5 §13.2.5.51's leading-newline-strip for `<pre>` (spec-compliant).
2. lxml's HTMLParser does NOT (spec-non-compliant — likely inherited from libxml2's HTML mode predating HTML5 parsing rules).
3. The downstream `xmltotxt` emits a content-free `` ``` `` fence pair when the spec-non-compliant leading `\n` survives the cascade.
4. Replicating Python's empty-fence emission requires either patching html5ever (invasive — html5ever is a spec implementation, not Trafilatura-specific) OR injecting a leading `\n` into every `<pre>` (destructive — would mis-render `<pre>` whose source genuinely has no leading newline).

Neither approach is anti-inversion-clean. The cleanest mitigation is to allowlist this specific fixture and document the parser-level root cause.

## Recommendation pinned for Stage Final

If a future stage wants to flip this fixture green:
1. **Best**: detect `<pre>` elements whose source HTML had a leading `\n` (would require keeping the raw input bytes alongside the parsed tree, then a byte-position scan). Complex.
2. **Worse**: always inject a leading `\n` into every `<pre>`'s `.text`. Destructive.
3. **Best long-term**: switch parser to one that supports the leading-newline-strip-OFF mode, OR fork html5ever to add a non-spec option. Invasive.

Default: allowlist. The 7-byte divergence is a parser quirk, not an extraction defect.
