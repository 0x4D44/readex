# M5 Allowlist ADR — `86df4d2e654952e4.html` (Wikipedia article, Python markdown-path `<sup>` fracture)

**Bucket pre-allowlist:** `content-mismatch` (Rust = 65,903 chars, Python = 65,906 chars; first-diff at byte 30291).
**Verdict:** anti-inversion-clean — Python's `extract(output_format="markdown")` re-runs `handle_formatting` during `xmltotxt(body, formatting=True)` and STRUCTURALLY FRACTURES the source's `<sup>[<i><a><span>citation needed</span></a></i>]</sup>` into multiple sibling `<hi>` elements at different tree positions. mdrcel preserves Python's own XML-path shape (`extract(output_format="xml")` keeps `[citation needed]` intact), which is the faithful Trafilatura body. Replicating Python's markdown-path mangling would be anti-inversion-violating bug-for-bug replication of an internal Python pipeline quirk.

## Why Python is wrong (or at least, internally inconsistent)

The fixture's source HTML at the divergence point contains an inline citation marker:

```html
... and price comparison websites may be coming close<sup id="cite_ref-..." class="reference"><a href="#cite_note-..."><span class="cite-bracket">[</span><i>citation needed</i><span class="cite-bracket">]</span></a></sup>. The main difference between ...
```

i.e. a paragraph with an inline `<sup>` containing the bracketed-italic "citation needed" link.

### Empirical proof — Python's XML and markdown outputs DIVERGE on the same body

Running Python's `trafilatura.extract` twice on the same fixture with different `output_format`:

```python
# output_format='xml'  (faithful body representation)
"...coming close[citation needed]. The main difference..."

# output_format='markdown'  (the gate's compare target)
"...coming close\n\n[. The main difference..."
```

XML keeps `[citation needed]` intact; markdown emits `\n\n[.` and drops `citation needed]` outright at this position.

### Trace through Python's `xmltotxt` quirk

Python's `trafilatura.extract(output_format='markdown')` flows:
1. `bare_extraction(tree, options)` runs the BODY arm: `extract_content` (`main_extractor.py:625-638`) produces `result_body` — at this point `<sup>...</sup>` is still a single inline span inside the parent `<p>`.
2. `determine_returnstring` (`core.py`) routes through `xmltotxt(document.body, formatting=True)` (`xml.py:194-244`).
3. `xmltotxt`'s formatting=True branch invokes `handle_formatting` for `<sup>` / `<hi>` / `<i>` / `<a>` descendants — this is the SECOND time `handle_formatting` runs on the body (the first was during `_extract`'s dispatch).
4. The second `handle_formatting` pass on the inline `<sup>` AND its nested `<i>` AND nested `<span>` produces multiple `<hi>` wrappers, each at a different tree position:

```xml
<p>...coming close</p>
<p><hi rend="#sup">[</hi>. The main difference...combine.</p>
<hi rend="#i">citation needed</hi>]
<p>An important type of web agent...</p>
```

The leading `[` becomes the text of a `<hi rend="#sup">` lifted into a fresh `<p>`; the rest of the original `<p>`'s content becomes that `<hi>`'s tail (`.`); the `<i>citation needed</i>` survives as a BARE `<hi rend="#i">` sibling at body level (no `<p>` parent) with `]` as ITS tail; the next paragraph starts cleanly.

### What mdrcel does instead

mdrcel emits the natural lxml-parsed shape:

```
...coming close[ citation needed]. The main difference...
```

— a single inline run, matching Python's XML output (the faithful body), with the bracketed citation marker rendered verbatim.

## Anti-inversion check

HLD §4 anti-inversion forbids out-cleaning Trafilatura. The inverse here would be deliberately fracturing the inline `<sup>` into multiple sibling `<hi>` elements at different tree positions to match Python's markdown output. That is precisely bug-for-bug replication of a Python-side pipeline quirk that:

1. Is INTERNALLY INCONSISTENT — Python's XML and markdown outputs disagree on the same body for the same fixture.
2. Has NO source-side justification — the fracture serves no extraction goal; it's an artefact of `handle_formatting` running twice (once in `_extract`, once in `xmltotxt`'s formatting branch) on inline formatting that doesn't deserve `<p>` lifting.
3. Mangles user-visible text — `[citation needed]` (a single semantic unit) is split across three tree positions with the closing `]` orphaned as a tail.

Replicating this requires re-running `handle_formatting` on already-processed inline elements DURING markdown rendering, with full understanding of which descendant patterns trip the fracture. That's multi-function refactoring with no faithfulness payoff — the Python XML output, which mdrcel matches, is the authoritative body.

## What evidence

- Stage 6i investigation (commit `adb2a04`, Cargo 0.15.17): probed Python's behaviour empirically, confirmed XML ≠ markdown for this fixture, traced the fracture to `xmltotxt(body, formatting=True)` re-invoking `handle_formatting`.
- Stage 6h-a (commit `5f44153`) and 6h-b (commit `74b37c7`): closed the `<code>`-cell-tail rendering and `&ddagger;` entity decode bugs; first-diff-byte moved 26846 → 27230 → 30291. Total diff shrank from 5 bytes to 3 bytes (the residual `[citation needed]` arm).
- Journal: `wrk_journals/2026.05.21 - JRN - M5 forward push goal loop.md`, Stage 6i section ("86df4d2e re-triage").

## What this allowlist deliberately does NOT claim

This ADR does NOT claim mdrcel is generically better than Python on Wikipedia or on inline citation markers. It claims that on THIS specific `<sup>[<i><a>...</a></i>]</sup>` fracture pattern, Python's markdown output is internally inconsistent with Python's XML output, and mdrcel's choice to match Python's XML (the faithful body) is the anti-inversion-correct decision. The 3-byte gap is a Python pipeline quirk, not an extraction defect on either side.
