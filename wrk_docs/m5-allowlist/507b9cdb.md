# M5 Allowlist ADR ŌĆö `507b9cdbe036bf58.html` (Apple FR ŌĆö French Wikipedia Apple Inc., Python markdown-path `<sup class="reference">` fracture)

**Bucket pre-allowlist:** `content-mismatch` masquerading as 1-byte net diff (Rust ~122,740 chars vs Python ~122,741; ~145 bytes of structural divergence cancel out at the char-count level over ~30 reference markers).
**Verdict:** anti-inversion-clean ŌĆö Python's `extract(output_format="markdown")` re-runs `handle_formatting` on inline `<hi rend="#sup">` markers inside `xmltotxt(body, formatting=True)`, structurally lifting each `<sup class="reference">[N]</sup>` into its own paragraph. The `class="reference"` discriminant is destroyed by `htmlprocessing.convert_tags` BEFORE `xmltotxt` ever sees the tree; the paragraph break is a side-effect of `strip_tags` mutating the live `iter("*")` during `handle_paragraphs`, not a class-aware rendering choice. mdrcel emits the inline reference, matching Python's XML output (the faithful body) ŌĆö replicating the markdown-path break would be doctrinally wrong bug-for-bug replication.

## Why Python is wrong (or at least, internally inconsistent)

The French Apple Inc. fixture uses ~30 inline numbered references of the form:

```html
... text <sup class="reference">[19]</sup> more text ...
```

The M5-deferred ADR (`wrk_docs/m5-deferred/507b9cdb.md`, now superseded) claimed Python's `xmltotxt` has a `<sup class="reference">` branch that emits the marker as its own paragraph. **That diagnosis is wrong.** M6 Stage 1 verified Python's source directly.

### Trace through Python's pipeline ŌĆö there is NO `<sup class="reference">` branch

1. `htmlprocessing.convert_tags` (`htmlprocessing.py:402-407`) walks the parsed tree and renames `<sup>` ŌåÆ `<hi rend="#sup">`, calling `elem.attrib.clear()` FIRST. The `class="reference"` attribute is gone before any downstream code sees the element.

2. `xml.py:process_element` (`xml.py:253-351`) ŌĆö the renderer used by `xmltotxt` ŌĆö has NO `<sup>`-class branch. There is NOTHING in `process_element` that could distinguish a `<sup class="reference">` from any other `<sup>` even if the attribute had survived.

3. The paragraph break that the M5 deferred ADR attributed to `<sup class="reference">` rendering is in fact a side-effect of `main_extractor.handle_paragraphs` / `handle_formatting` (`main_extractor.py:108-115, 272-351`) running on inline `<hi rend="#sup">` markers when `formatting=True`. `strip_tags` calls inside `handle_paragraphs` mutate the live tree during `iter("*")`, leaving some `<hi>` descendants un-marked-"done" with their parent renamed. Subsequent `_extract` iteration dispatches them to `handle_formatting`, which wraps each in a fresh `<p>` whose tail becomes the new paragraph's body text.

### Empirical proof ŌĆö Python's XML and markdown outputs DIVERGE on the same body

Running Python's `trafilatura.extract` twice on the same fixture with different `output_format`:

```python
# output_format='txt'  (formatting=False)
"...texte [19] suite..."           # inline, matches mdrcel exactly

# output_format='markdown'  (formatting=True ŌĆö the gate's compare target)
"...texte\n\n[19]\n\nsuite..."     # lifted into its own paragraph
```

The XML/txt path preserves the inline reference; the markdown path fractures it. Same fixture, same body, two disagreeing Python outputs. The fracture is triggered by `formatting=True`, not by anything specific to `class="reference"`.

### What mdrcel does instead

mdrcel emits the natural lxml-parsed inline shape:

```
...texte [19] suite...
```

ŌĆö matching Python's XML and `output_format='txt'` outputs (the faithful body), with each reference marker rendered verbatim where it lives.

## Anti-inversion check

HLD ┬¦4 anti-inversion forbids out-cleaning Trafilatura. The inverse here would be deliberately fracturing each inline `<sup>` (now `<hi rend="#sup">`) into its own paragraph to match Python's markdown output. That is precisely bug-for-bug replication of a Python-side pipeline quirk that:

1. Is INTERNALLY INCONSISTENT ŌĆö Python's XML, txt, and markdown outputs disagree on the same body for the same fixture.
2. Has NO source-side justification ŌĆö the fracture serves no extraction goal; it's an artefact of `handle_formatting` running on inline `<hi>` markers when `formatting=True`, identical to the already-allowlisted 86df4d2e pattern.
3. Mangles document flow ŌĆö ~30 inline references on a single Wikipedia article each break the sentence containing them into three pieces.

The M5 deferred ADR's "~30 LOC fix in `output.rs`" recipe was predicated on a `<sup class="reference">` discriminant that does not exist in Python. Implementing it would mean inventing a class-aware code path that Python itself does not have, then defending it to future supervisors as "matching Python" ŌĆö the worst kind of anti-inversion drift.

## What this allowlist deliberately does NOT claim

This ADR does NOT claim mdrcel is generically better than Python on Wikipedia or inline reference markers. It claims that on THIS specific `<sup class="reference">[N]</sup>` inline pattern, Python's markdown output is internally inconsistent with Python's XML and txt outputs (Python's own non-markdown paths), and mdrcel's inline rendering ŌĆö which matches those non-markdown Python paths ŌĆö is the anti-inversion-correct decision. The ~145 bytes of structural divergence (net ~1) are a Python pipeline quirk, not an mdrcel extraction defect.
