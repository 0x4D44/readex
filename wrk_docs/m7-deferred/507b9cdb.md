# M7 Deferred ADR — `507b9cdbe036bf58.html` (Apple FR, mdrcel leaks non-printable U+2063 on the txt path)

**Gate:** `tests/trafilatura_txt_gate.rs` (`--txt` / `output_format="txt"`).
**Bucket pre-defer:** `content-mismatch` (Rust = 121,842 chars, Python = 121,769 chars; first-diff at byte ~31,000).
**Verdict:** **mdrcel is the buggy side.** This is a genuine mdrcel defect, NOT an anti-inversion-clean Python bug. Pinned to a future milestone pending a Unicode-category facility.

## The divergence

At byte ~31,000 the two outputs are:

```
rust:   …iPadOS 15⁣⁣, il arrive sur les iPad.…
python: …iPadOS 15, il arrive sur les iPad.…
```

The two extra chars in mdrcel's output are **U+2063 INVISIBLE SEPARATOR**
(Unicode category `Cf`, "Format"). The source HTML literally contains them:

```html
<a … title="IPadOS 15">iPadOS 15⁣⁣</a>, il arrive sur les …
```

The fixture sprinkles such invisible separators throughout (around link text /
reference markers). The net ~73-char delta over the whole document is U+2063
pairs that mdrcel preserves and Python strips.

## Why mdrcel is wrong (not Python)

Python's txt path routes body text through `sanitize` → `line_processing` →
`remove_control_characters` (`utils.py:272-300`). `remove_control_characters`
maps `return_printables_and_spaces` over every char:

```python
def return_printables_and_spaces(char):
    return char if char.isprintable() or char.isspace() else ''
```

Python's `str.isprintable()` returns **False** for category `Cf` chars (and the
rest of the `Other`/`Separator` categories except U+0020 space), and U+2063 is
not whitespace, so Python correctly **strips** it. Verified empirically:
Python's txt output for this fixture contains ZERO non-printable, non-space
characters.

mdrcel's `output::line_processing` (`src/trafilatura/output.rs:787`) **omits**
`remove_control_characters` — the code comment there explicitly says it was
deferred "until a future test surfaces a control-character leak." This txt gate
is exactly that test. mdrcel passes U+2063 through verbatim.

(Note: this is a DIFFERENT and INDEPENDENT issue from the markdown-path
`<sup>` fracture documented in `wrk_docs/m5-allowlist/507b9cdb.md`. That ADR
correctly noted Python's txt path matches mdrcel on the reference-marker
*structure*; it did not examine the invisible-separator characters, which are
the divergence here. The two findings do not conflict — markdown diverges on
`<sup>` paragraph fracture under `formatting=True`; txt diverges on U+2063
control-char stripping. Both are present in this same fixture.)

## Why the fix is deferred (not applied now)

A faithful port of `remove_control_characters` requires replicating Python's
`str.isprintable()`, i.e. testing each char against the Unicode `Other` (Cc,
Cf, Cs, Co, Cn) and `Separator` (Zl, Zp, Zs) general categories (keeping U+0020
and whitespace). The Rust standard library has no `is_printable` /
general-category accessor (`char::is_control()` only covers `Cc`). A correct
implementation therefore needs either:

1. a new pinned dependency (e.g. `unicode-general-category`), or
2. a vendored Unicode general-category table.

Both are dependency / data-table decisions that warrant supervisor sign-off
under the project's DEC-3 new-dependency discipline (CLAUDE.md: discuss
architectural/dependency decisions before implementation). A hand-rolled
approximate `isprintable` would be a fresh correctness-divergence surface —
exactly the kind of thing DEC-3 says to avoid by using an audited dependency.
So the fix is real port work scoped to a future milestone, not a small/safe
in-stage change.

## Remediation (future milestone, e.g. M8)

Implement `remove_control_characters` faithfully in
`src/trafilatura/output.rs::line_processing` (and the sibling
`src/trafilatura/metadata_url.rs::line_processing`, which omits it too), using a
Unicode general-category facility to mirror `char.isprintable() or
char.isspace()`. Expected effect: 507b9cdb moves from deferred to substantive on
the txt gate (and the markdown gate's existing 507b9cdb allowlist entry is
unaffected — that one is the separate `<sup>` fracture).

## Also affects the json gate (M7 Stage 2)

`tests/trafilatura_json_gate.rs` (`--json` / `output_format="json"`) hits the
SAME defect for the SAME reason. Python's `build_json_output` (`xml.py:125/129`)
sets the `text` field to `xmltotxt(body, include_formatting=False)` — the exact
string the txt path emits — and mdrcel's `extract_to_json` does likewise via
`build_json_output` (`src/trafilatura/output.rs:1190`). So the leaked U+2063
pairs surface verbatim inside json's `"text"` value (Rust = 122,622 chars,
Python = 122,547 chars; first-diff at byte ~31,141, the same `iPadOS 15⁣⁣` vs
`iPadOS 15` window). The json gate therefore lists `507b9cdbe036bf58.html` in
its `DEFERRED_KNOWN_DEFECT` array, cross-referencing this ADR. The remediation
(implementing `remove_control_characters` in `line_processing`) fixes both the
txt and json gates in one move — no json-specific work is needed.

## Also affects the csv gate (M7 Stage 3)

`tests/trafilatura_csv_gate.rs` (`--csv` / `output_format="csv"`) hits the SAME
defect for the SAME reason: Python's `xmltocsv` (`xml.py:369`) fills the `text`
column with `xmltotxt(body, include_formatting=False)` — the identical string
the txt / json paths emit — so the leaked U+2063 pairs surface verbatim inside
the csv `text` cell. The csv gate therefore lists `507b9cdbe036bf58.html` in its
`DEFERRED_KNOWN_DEFECT` array, cross-referencing this ADR. The remediation
(`remove_control_characters` in `line_processing`) fixes the txt, json, and csv
gates in one move — no csv-specific work is needed.

## Anti-inversion check

This is NOT an allowlist case: Python's behaviour here is correct (stripping a
non-printable format character from extracted text is the right thing), and
matching it is faithful port work, not bug-for-bug replication. mdrcel is simply
missing a pipeline step that Python has. Deferring documents the defect and the
remediation milestone so the gate does not silently regress on other fixtures
while we live with this one.
