## Why branch is so much lower than line / region

Two structural reasons, both already documented in the source modules:

1. **Faithful, exhaustive `match` ladders.** Many ports of Python `if/elif` cascades (e.g. `metadata_jsonld::walk_article`, `metadata::examine_meta`, `htmldate::core::search_page`, `output::process_element`) compile down to `match` ladders where each pattern is its own branch pair. The "happy" branch fires on every corpus snapshot; the negative side fires only on inputs that don't carry that signal. The Trafilatura/htmldate fixtures the gates already drive are deliberately article-shaped, so dozens of negative arms never see traffic.
2. **`Option`/`Result` early returns from helper primitives.** Functions like `htmldate::extractors::try_date_expr`, `validators::date_validator`, `metadata_url::extract_catstags`, and `xpath_engine::*` use `?`/`if let Some(..) else return None` ladders for fail-fast on malformed input. The `Err(..)` / `None` half of each `?` is its own branch; almost none of the fail paths are exercised by article-shaped fixtures.

Both classes are **non-malicious low-hanging fruit**: faithful negative-shape unit tests in the same `mod tests {}` blocks where the function lives drive the missing arms without any architectural risk.

## Cold-spot taxonomy

Walking the 25 files in order reveals **five repeating shapes** the new tests will need to drive:

1. **Deep-fallback cascades in `htmldate`.** `search_page` runs 10 regex catalogues in source order, returning early on the first match — only the first 2-3 of those arms are hit by the parity corpus. Same pattern at `examine_header` (meta-tag walker), `examine_time_elements`, and `examine_abbr_elements`. **Fix shape:** craft synthetic minimal-HTML strings that survive the earlier arms and force the later arms to fire.
2. **`Option`/`Result` early-return helpers.** `htmldate::extractors::custom_parse`, `try_date_expr`, `validators::date_validator`, `metadata_url::extract_catstags`. **Fix shape:** unit tests that feed each known-bad input shape — empty string, malformed regex match groups, out-of-range numerics — and assert the negative side returns `None`/`Err`.
3. **`metadata_jsonld` walker + JSON-LD shape catalog.** `walk_article` and friends handle `dict`/`list`/`str`/`null` polymorphism inline with `match` on the carrier enum; the live corpus is article-shaped JSON-LD only. **Fix shape:** unit tests that feed each non-article JSON-LD shape (`@graph`, `WebPage`, `WebSite`, `Organization`, malformed nested arrays, nested null breadcrumbs).
4. **`grab_article` retry / flag-sieve.** `_grabArticle` is one attempt; the outer retry loop (FLAG_STRIP_UNLIKELYS, FLAG_WEIGHT_CLASSES, FLAG_CLEAN_CONDITIONALLY) survives in the function body. The retry loop only fires when `textLength < _charThreshold`, which the corpus does not trigger. **Fix shape:** short-article synthetic HTML below the 500-char threshold that walks each retry arm.
5. **Format-emitter side branches in `output.rs`.** `add_xml_meta`, `build_yaml_header`, `process_element`, `replace_element_text` each carry per-attribute conditional emit logic. The metadata gates use article-shaped fixtures so half the attribute conditionals never fire. **Fix shape:** YAML/XML emitter tests with metadata structs that have ONE field populated at a time (and `None` for everything else), then with each field missing exactly to walk both halves of each conditional.

Each shape has a clear, low-risk fix path: pure in-`mod tests {}` unit tests on the existing data structures, no new dependencies, no Python oracle round-trips, no parity-gate risk.

## What the existing corpus does well

The 51-fixture snapshot corpus + 14-fixture TEI/XML/JSON/CSV/MD/TXT/equivalence parity ladder + 10-fixture htmldate parity gate gives us **93.96 %** region coverage and **96.58 %** function coverage — the algorithmic shape is exhaustively tested at the macro level. The 6 % gap is the branch-pair coverage of negative input shapes the corpus does not need to carry.

In other words: **the corpus tests algorithmic correctness; the new tests will test defensive correctness.** Both are real-world important; only one is suitable for a parity oracle.

## Risk register — anti-Goodhart fences

The coverage push must NOT:

- Introduce tests that **assert nothing** (no `assert!(true)`, no `let _ = f(x);`).
- Use **mocks** for behaviour the production code computes directly.
- Add tests that **lock in current (possibly-wrong) output** without a citation back to the Python oracle or to an explicit `// faithful-divergence:` doc comment.
- Use names like `cov_branch_47` that signal "I exist for the metric, not the contract".
- Touch existing parity-gate assertions or fixtures (M2/M3/M4 BLOCKER status is sacrosanct).

Every new test must carry an `// rationale:` comment naming **the contract it pins**, not the line it covers.
