## 1. Goal and Non-Goals

### Goal

Add a pre-processing step inside `Dom::parse()` (`D:\language\mdrcel\src\readability\dom.rs:127`) that transforms the raw HTML string **before** it reaches html5ever's `parse_document()`. The pre-processor rewrites three specific HTML patterns where html5ever's HTML5-spec-strict behaviour produces materially different DOM trees than lxml's permissive parser, causing downstream content selection and scoring divergences.

The pre-processor targets the 5.4% "truly divergent" pages (Dice < 0.80) identified by Phase B's content-similarity oracle, estimated to resolve ~18-25 of the ~81 truly-divergent page-records across the three shapes.

### This is explicitly section-7-deviating

Python trafilatura does NOT have any equivalent pre-processing step. Python achieves the equivalent result through lxml's parser permissiveness, not through any algorithmic filter. The M10 Phase 2F, 2G, and 2H escalation reports each independently confirmed this: no Python code path filters, rewrites, or compensates for any of the three shapes documented here. The deviation is justified by Phase B's similarity metric — we measure improvement against Dice similarity (content quality), not byte-mimicry of Python's pipeline.

### Non-Goals

- **Full HTML parser replacement.** This is a bounded, targeted pre-processor for three known shapes, not a general-purpose lxml-compatibility layer. Patterns beyond the three shapes are out of scope.
- **Byte-equivalence to Python.** The pre-processor aims to improve content quality (Dice similarity), not to produce byte-identical output to Python's pipeline. Some divergences will remain.
- **Spec compliance.** The pre-processor deliberately undoes html5ever's spec-correct behaviour for these three cases. This is intentional.
- **New dependencies.** No new crate dependencies. The pre-processor is a pure `fn(&str) -> Cow<'_, str>` operating on the raw HTML byte stream.

## 2. The Three Shapes

### 2.1 Shape 1: Stray `</br>` end-tags

**HTML5 spec:** section 13.2.6.4.7 ("An end tag whose tag name is 'br'") — "Parse error. Drop the attributes from the token, and act as described in the next entry; i.e. act as if this was a 'br' start tag token with no attributes, rather than the end tag token that it actually is." html5ever implements this literally: `</br>` becomes a `<br>` start tag.

**lxml behaviour:** Drops `</br>` silently. Surrounding text is concatenated as a single text run.

**html5ever behaviour:** Creates a `<br>` element node. Text following `</br>` becomes the `<br>` element's tail (a following Text sibling in rcdom). Downstream, `convert_tags` renames `<br>` to `<lb>`, and the serialiser emits a newline boundary at each `<lb>`, producing extra blank lines that Python's output does not have.

**Evidence (Phase 2H report, section 3d):**

```python
# lxml
>>> snippet = '<div><p>before</br>after</p></div>'
>>> tree = html.fromstring(snippet)
>>> etree.tostring(tree)
b'<div><p>beforeafter</p></div>'
```

```
# html5ever
<div><p>before</br>after</p></div>
  -> <p> text="before"
       <br> text=None tail="after"
```

**Pre-processor transformation:** Strip every `</br>` token (case-insensitive, with optional whitespace around the slash: `</ br>`, `</BR>`, `</Br >`, etc.) from the HTML before parsing.

**Before:** `<p>Warning text.</br>Legal disclaimer text.</p>`
**After:** `<p>Warning text.Legal disclaimer text.</p>`

**Population estimate:** 10 of 300 corpus files contain stray `</br>`. Estimated 3-6 whitespace-class divergence records + 10-15 text-content records where the spurious `<lb>` cascades into selection shifts. Total: ~13-21 records.

**Escalation report:** `D:\language\mdrcel\wrk_docs\2026.05.24 - REPORT - M10 Phase 2H escalation.md`

### 2.2 Shape 2: Stray table-cell tags outside `<table>` context

**HTML5 spec:** section 13.2.6.4.7 ("in body" insertion mode) — start tags `<td>`, `<th>`, `<tr>`, `<tbody>`, `<tfoot>`, `<thead>` encountered outside a `<table>` insertion-mode context are parse errors. The spec says "Ignore the token." html5ever implements this literally: the element is silently dropped. The text content that was between the tags survives as flat body text (not wrapped in an element).

**lxml behaviour:** Permissively keeps these tags as normal elements, regardless of whether they appear inside a `<table>`. Content stays wrapped in elements, which affects downstream scoring.

**html5ever behaviour:** Drops the start tags. Text content between `<td>foo</td>` becomes a bare Text node directly under the parent element (not wrapped in a `<td>` element). This changes the tree structure, which cascades through readability scoring (`score_paragraphs` scores `<td>` descendants differently than bare text), causing different best-candidate selection.

**Evidence (Phase 2G report, section 4.2):** On fixture `523a23495e14`, lxml's tree preserves `<td>`, `<tr>`, etc. inside the SEO-spam div, giving the spam div many descendant elements. html5ever's tree drops those tags, leaving the spam div with mostly text nodes. The scoring math (paragraph text length + comma count) then picks the spam div in mdrcel (because its raw text, including `<xmp>` literal text, is longer) but picks the legitimate `<section>` in Python (because the `<section>` has more scored `<p>`/`<td>` descendants).

**Pre-processor transformation:** Rewrite stray table-cell start/end tags to `<div>`/`</div>` ONLY when they appear outside any `<table>...</table>` nesting. Inside `<table>` nesting, leave them untouched.

Tags rewritten: `<td>`, `<th>`, `<tr>`, `<tbody>`, `<tfoot>`, `<thead>` (start tags) and their corresponding end tags `</td>`, `</th>`, `</tr>`, `</tbody>`, `</tfoot>`, `</thead>`.

Attributes are preserved during rewrite: `<td class="foo" id="bar">` becomes `<div class="foo" id="bar">`.

**Before:** `<div><td class="spam">content</td></div>`
**After:** `<div><div class="spam">content</div></div>`

**Before (inside table, NOT rewritten):** `<table><tr><td>cell</td></tr></table>`
**After (unchanged):** `<table><tr><td>cell</td></tr></table>`

**Population estimate:** ~14 records (Phase 2F corpus analysis: 4/5 representatives showed substantial table-cell disagreement; extrapolation from the 18-record strict stray-tag pattern to the full working slice).

**Escalation reports:** `D:\language\mdrcel\wrk_docs\2026.05.23 - REPORT - M10 Phase 2F parser probe.md`, `D:\language\mdrcel\wrk_docs\2026.05.24 - REPORT - M10 Phase 2G escalation.md`

### 2.3 Shape 3: `<xmp>` raw-text elements

**HTML5 spec:** section 13.2.5.2 — `<xmp>` is a "raw text element". The tokenizer switches to the "RAWTEXT" state, meaning everything inside `<xmp>...</xmp>` is treated as literal character data (HTML tags are NOT parsed as elements; they become text nodes).

**lxml behaviour:** Treats `<xmp>` as a normal container element. Inner HTML tags are parsed as element children, exactly as if they were inside a `<div>`.

**html5ever behaviour:** Treats inner content as a single Text node. E.g., `<xmp><p>hello</p></xmp>` produces a Text node with data `<p>hello</p>`, not a `<p>` child element. This means that if SEO-spam pages embed tag-soup HTML inside `<xmp>`, the literal tag text (`<td id="...">`, `<rt id="...">`, etc.) bleeds into mdrcel's text output as visible content, while Python's lxml-parsed tree had those as silent empty elements that downstream filters removed.

**Evidence (Phase 2F report, section 3a, mechanism (ii)):**

```
# html5ever
<xmp> [id="woq6h"]
  #text "<rt id=\"woq6h\"><legend id=\"woq6h\"></legend></rt>"

# lxml
<xmp> [id="woq6h"]
  <rt> [id="woq6h"]
    <legend> [id="woq6h"]
```

**Pre-processor transformation:** Rewrite `<xmp>` to `<div>` and `</xmp>` to `</div>` (case-insensitive, preserving attributes). This causes html5ever to parse inner content as normal HTML elements rather than raw text.

**Before:** `<xmp id="foo"><p>hello</p></xmp>`
**After:** `<div id="foo"><p>hello</p></div>`

**Population estimate:** ~4 records (Phase 2F: `<xmp>` appeared in SEO-spam fixtures; most are also covered by Shape 2's table-cell rewrite, so the independent incremental yield is smaller).

**Escalation report:** `D:\language\mdrcel\wrk_docs\2026.05.23 - REPORT - M10 Phase 2F parser probe.md`

## 3. Implementation Sketch

### 3.1 Function signature

```
fn preprocess_html(html: &str) -> Cow<'_, str>
```

- **Common case (no transformations needed):** returns `Cow::Borrowed(html)` — zero allocation.
- **Uncommon case (transformations needed):** returns `Cow::Owned(modified_string)`.

### 3.2 Integration point

Inside `Dom::parse()` at `D:\language\mdrcel\src\readability\dom.rs:127-162`. The call is inserted immediately before `parse_document(...).one(html)`:

```
pub fn parse(html: &str) -> Self {
    let html = preprocess_html(html);         // <-- NEW
    let opts = ParseOpts { ... };
    let dom = parse_document(RcDom::default(), opts).one(&*html);
    strip_comments_and_pis(&dom.document);
    Dom { dom, content_score: HashMap::new(), readability_data_table: HashSet::new() }
}
```

The `&*html` dereference works for both `Cow::Borrowed` (returns the original `&str`) and `Cow::Owned` (returns `&str` into the owned `String`). The `let html = ...` shadowing is intentional — the original `html: &str` parameter is not needed after pre-processing.

### 3.3 Fast-path (quick-scan)

Before allocating or scanning character-by-character, do a fast check for whether the HTML could possibly contain any of the three trigger patterns. This is a case-insensitive substring search for a small set of marker substrings.

**Quick-scan markers:**

- `</br` — Shape 1 trigger (stray `</br>` end-tag). The `<` + `/` + `br` sequence is specific enough to avoid false positives on legitimate `<br>` or `<br/>`.
- `<xmp` — Shape 3 trigger. The `<xmp` prefix (case-insensitive) is the entry into a raw-text element.
- For Shape 2 (stray table cells), the quick-scan cannot cheaply distinguish stray-outside-table from legitimate-inside-table. Two approaches:
  - **(Option A)** Always do the full scan if `<td` or `<th` or `<tr` or `<tbody` or `<tfoot` or `<thead` appears — but this means nearly every HTML page triggers the full scan (most pages have `<table>`).
  - **(Option B)** Only trigger the full scan if one of the cell tags appears AND the document also lacks `<table` — but this misses the stray-outside-table case on pages that ALSO have legitimate tables elsewhere.
  - **(Option C, recommended)** Accept that the Shape 2 scan runs on any page containing cell-tag substrings. The full scan is still O(n) and cheap (a single forward pass). The Shape 2 state machine's table-nesting tracker ensures correctness: cell tags inside `<table>` are left untouched.

**Quick-scan pseudocode:**

```
let lower = html.as_bytes();  // for fast scanning
let has_br_end = contains_case_insensitive(lower, b"</br");
let has_xmp = contains_case_insensitive(lower, b"<xmp");
let has_stray_cell_candidate = contains_case_insensitive(lower, b"<td")
    || contains_case_insensitive(lower, b"<th")
    || contains_case_insensitive(lower, b"<tr")
    || contains_case_insensitive(lower, b"<tbody")
    || contains_case_insensitive(lower, b"<tfoot")
    || contains_case_insensitive(lower, b"<thead");

if !has_br_end && !has_xmp && !has_stray_cell_candidate {
    return Cow::Borrowed(html);  // fast path, zero allocation
}
```

The `contains_case_insensitive` helper does a single O(n) pass over the bytes, checking for any of the marker prefixes. For efficiency, short-circuit on the first match found (we need to know "any trigger present?" not "which triggers?").

**Pushback on quick-scan granularity:** The quick-scan could be a single pass that checks all markers simultaneously (like an Aho-Corasick automaton), rather than multiple independent `contains_case_insensitive` calls. However, the markers are short (3-6 bytes each) and HTML pages are typically 10-100KB. Seven sequential `memchr`-style scans over 50KB take ~1 microsecond total on modern hardware. The simplicity of independent checks outweighs the constant-factor improvement of a merged automaton. If profiling shows this matters, a merged scan can be retrofitted without API change.

### 3.4 Transformation pass

When the quick-scan finds at least one trigger, allocate a `String` and do a single forward scan of the entire HTML input, building the output character-by-character (or chunk-by-chunk for non-trigger regions). The scan is a simple state machine with the following states:

**State variables:**

- `table_depth: usize` — tracks `<table>` nesting depth. Starts at 0.
- `pos: usize` — current byte position in the input.
- `output: String` — pre-allocated with `String::with_capacity(html.len())`.

**At each position**, the scanner checks (in priority order):

1. **Is this the start of a `<` character?** If not, copy the byte to output and advance.

2. **Is this `</br` (case-insensitive)?** If yes, scan forward to find the closing `>` (or end of input). Drop the entire `</br ... >` token (do NOT copy to output). Advance past the `>`.

3. **Is this `<xmp` (case-insensitive) followed by whitespace or `>`?** If yes, replace `<xmp` with `<div` in output, copy any attributes and the `>` verbatim. Advance.

4. **Is this `</xmp` (case-insensitive) followed by whitespace or `>`?** If yes, replace `</xmp` with `</div` in output, copy the rest verbatim. Advance.

5. **Is this `<table` (case-insensitive) followed by whitespace or `>`?** If yes, increment `table_depth`. Copy the tag verbatim. Advance.

6. **Is this `</table` (case-insensitive) followed by whitespace or `>`?** If yes, decrement `table_depth` (saturating at 0 to handle malformed HTML with more `</table>` than `<table>`). Copy the tag verbatim. Advance.

7. **Is this a stray-cell start tag (`<td`, `<th`, `<tr`, `<tbody`, `<tfoot`, `<thead`, case-insensitive) followed by whitespace or `>`?** If yes AND `table_depth == 0`, replace the tag name with `div` in output (preserving any attributes and the `>`). If `table_depth > 0`, copy verbatim. Advance.

8. **Is this a stray-cell end tag (`</td`, `</th`, `</tr`, `</tbody`, `</tfoot`, `</thead`, case-insensitive) followed by whitespace or `>`?** If yes AND `table_depth == 0`, replace with `</div>` in output. If `table_depth > 0`, copy verbatim. Advance.

9. **Otherwise:** Copy the `<` to output and advance by one byte.

**Tag-name matching** is case-insensitive byte comparison. The scanner does NOT need to be a full HTML tokenizer: it only recognises a fixed set of tag-name prefixes. Anything that doesn't match a known prefix is copied through verbatim.

**"Followed by whitespace or `>`" guard:** This prevents false matches on tag names that are prefixes of longer tags. E.g., `<thread>` should NOT match the `<thead` rule. The guard checks that the byte immediately after the tag name is one of: `>`, ` `, `\t`, `\n`, `\r`, `\x0C` (form feed), or `/` (for self-closing tags like `<br/>`). For end tags, the guard is after the tag name and before the `>`.

### 3.5 Pseudocode for `preprocess_html`

```
fn preprocess_html(html: &str) -> Cow<'_, str> {
    // Quick-scan: does the HTML contain any trigger pattern?
    if !has_any_trigger(html) {
        return Cow::Borrowed(html);
    }

    let bytes = html.as_bytes();
    let mut out = String::with_capacity(bytes.len());
    let mut pos = 0;
    let mut table_depth: usize = 0;

    while pos < bytes.len() {
        if bytes[pos] != b'<' {
            // Fast bulk copy: find next '<' and copy everything before it
            let next_lt = memchr(b'<', &bytes[pos..]).unwrap_or(bytes.len() - pos);
            out.push_str(&html[pos..pos + next_lt]);
            pos += next_lt;
            continue;
        }

        // We're at a '<'. Try to match a trigger pattern.

        if matches_closing_br(bytes, pos) {
            // Shape 1: </br...> — strip entirely
            pos = skip_to_closing_angle(bytes, pos);
            continue;
        }

        if let Some(end) = matches_xmp_open(bytes, pos) {
            // Shape 3: <xmp...> -> <div...>
            out.push_str("<div");
            out.push_str(&html[pos + 4..end]);  // attributes + '>'
            pos = end;
            continue;
        }

        if let Some(end) = matches_xmp_close(bytes, pos) {
            // Shape 3: </xmp...> -> </div...>
            out.push_str("</div");
            out.push_str(&html[pos + 5..end]);  // whitespace + '>'
            pos = end;
            continue;
        }

        if let Some(end) = matches_table_open(bytes, pos) {
            // Track table nesting
            table_depth += 1;
            out.push_str(&html[pos..end]);
            pos = end;
            continue;
        }

        if let Some(end) = matches_table_close(bytes, pos) {
            table_depth = table_depth.saturating_sub(1);
            out.push_str(&html[pos..end]);
            pos = end;
            continue;
        }

        if table_depth == 0 {
            if let Some((tag_end, name_len, end)) = matches_cell_open(bytes, pos) {
                // Shape 2: <td...> -> <div...> (outside table)
                out.push_str("<div");
                out.push_str(&html[pos + 1 + name_len..end]); // attrs + '>'
                pos = end;
                continue;
            }
            if let Some((name_len, end)) = matches_cell_close(bytes, pos) {
                // Shape 2: </td...> -> </div...> (outside table)
                out.push_str("</div");
                out.push_str(&html[pos + 2 + name_len..end]); // ws + '>'
                pos = end;
                continue;
            }
        }

        // No match — copy the '<' and advance
        out.push('<');
        pos += 1;
    }

    Cow::Owned(out)
}
```

### 3.6 Complexity

- **Quick-scan:** O(n) where n = input length. Seven sequential substring searches, each O(n). Total: 7 * O(n) = O(n).
- **Transformation pass:** O(n). Single forward scan with O(1) work per byte (tag-name matching is bounded by the longest tag name, 6 bytes for "thead"/"tbody"/"tfoot").
- **Space:** O(n) for the output `String` in the uncommon case. Zero allocation in the common case.

## 4. Edge Cases

### 4.1 Trigger patterns inside attribute values

Example: `<div title="Click </br> here">text</div>`

The pre-processor's forward scanner does NOT track whether it is inside an attribute value. It will match `</br>` inside the attribute and strip it, producing `<div title="Click  here">text</div>`.

**Assessment:** This is acceptable. Attribute values containing literal `</br>` are vanishingly rare in real-world HTML. The pre-processor is a best-effort heuristic targeting SEO-spam pages and malformed HTML, not a spec-compliant tokenizer. If a legitimate page has `</br>` inside an attribute value, the attribute value will be slightly mangled, but the page's main content (which is what mdrcel extracts) will be unaffected. The same reasoning applies to `<xmp` and cell-tag patterns inside attributes.

**Mitigation (if needed in future):** A quote-tracking state variable (`in_single_quote`, `in_double_quote`) could be added to suppress matching inside attribute values. This is NOT proposed for the initial implementation because (a) it adds complexity, (b) the risk is negligible on the corpus, and (c) it would need to handle unmatched quotes and nested quotes, which adds further complexity. If a corpus fixture surfaces that triggers this edge case, the mitigation can be added as a targeted follow-up.

### 4.2 Trigger patterns inside `<script>` and `<style>` blocks

Example: `<script>var html = "</br>";</script>`

The pre-processor does NOT track `<script>` or `<style>` blocks. It will match `</br>` inside a `<script>` and strip it.

**Assessment:** Acceptable. html5ever treats `<script>` and `<style>` as raw-text/raw-script elements; their text content is NOT parsed as HTML. The pre-processor stripping a `</br>` from inside `<script>` will not change any parsed element or text node — it will change the raw text content of the `<script>` node, which mdrcel (and trafilatura) already strip entirely during `tree_cleaning` (`MANUALLY_CLEANED` includes `script` at `settings.py:357`). No content extraction is affected.

### 4.3 Trigger patterns inside HTML comments

Example: `<!-- </br> -->`

The pre-processor does NOT track HTML comments. It will match `</br>` inside a comment and strip it.

**Assessment:** Acceptable. `Dom::parse()` already calls `strip_comments_and_pis()` immediately after `parse_document()` (`D:\language\mdrcel\src\readability\dom.rs:156`). Comment content is discarded before any extraction logic runs. Stripping a `</br>` from inside a comment has no effect on the final output.

### 4.4 Trigger patterns inside CDATA sections

CDATA sections (`<![CDATA[...]]>`) are an XML construct, not valid HTML5. html5ever does not recognise CDATA in HTML mode (it treats `<![CDATA[` as a bogus comment). The pre-processor does not need to handle CDATA.

### 4.5 Nested tables (Shape 2)

Example:
```html
<div>
  <td>stray cell 1</td>           <!-- outside table: rewritten to <div> -->
  <table>
    <tr>
      <td>legit cell</td>         <!-- inside table: NOT rewritten -->
      <table>
        <tr>
          <td>inner cell</td>     <!-- inside nested table: NOT rewritten -->
        </tr>
      </table>
    </tr>
  </table>
  <td>stray cell 2</td>           <!-- outside table: rewritten to <div> -->
</div>
```

The `table_depth` counter handles this correctly:
- Before first `<table>`: depth=0, `<td>` rewritten.
- After first `<table>`: depth=1, `<td>` NOT rewritten.
- After inner `<table>`: depth=2, `<td>` NOT rewritten.
- After inner `</table>`: depth=1, still NOT rewritten.
- After outer `</table>`: depth=0, `<td>` rewritten.

### 4.6 Self-closing variants

- `<br/>` — NOT matched by the Shape 1 rule (which only targets `</br`). Self-closing `<br/>` is a legitimate void element that html5ever and lxml both handle correctly.
- `<xmp/>` — Matched by the Shape 3 rule, rewritten to `<div/>`. html5ever would treat `<xmp/>` as `<xmp>` (it ignores the `/` on non-void elements per spec). Rewriting to `<div/>` is harmless — html5ever will parse it as `<div>`.
- `<td/>`, `<tr/>`, etc. — Matched by the Shape 2 rule when outside `<table>`. Rewritten to `<div/>`. This is correct: a self-closing cell tag outside a table is malformed regardless.

### 4.7 Uppercase and mixed case

All tag-name matching is case-insensitive. `</BR>`, `</Br>`, `<XMP>`, `<Xmp>`, `<TD>`, `<tD>`, `<TBODY>`, etc. are all matched.

Implementation: compare each byte of the tag name with both its uppercase and lowercase ASCII variant. This is a simple `byte | 0x20` ASCII-lowering trick for the alphabetic characters in the tag names (all tag names in the trigger set are pure ASCII alphabetic).

### 4.8 Whitespace inside closing `</br>` tag

HTML allows whitespace between the tag name and `>` in end tags: `</ br>`, `</br >`, `</ br >`. The pre-processor's `</br` match should scan forward past optional whitespace to find the closing `>`.

Whitespace between `<` and `/` is also possible in malformed HTML: `< /br>`. This is NOT valid HTML and is unlikely in practice. The pre-processor does NOT handle this variant (the quick-scan for `</br` would not match `< /br`). If a corpus fixture surfaces this, a targeted extension can be added.

### 4.9 Malformed nesting: more `</table>` than `<table>`

The `table_depth` counter uses `saturating_sub(1)` on `</table>`, so it clamps to 0 rather than underflowing. After an unmatched `</table>`, the scanner is back at depth=0 and stray-cell rewriting is active. This is the conservative choice: if the HTML is malformed enough to have unmatched `</table>`, cell tags after it are likely stray.

### 4.10 Tags that are prefixes of longer tag names

`<thread>` must NOT match the `<thead` rule. `<trace>` must NOT match the `<tr` rule. `<tdc>` must NOT match the `<td` rule.

The "followed by whitespace or `>`" guard (section 3.4, step 7) prevents this: after matching the tag-name bytes, the scanner checks that the next byte is a tag-boundary character (`>`, ` `, `\t`, `\n`, `\r`, `\x0C`, `/`). Alphabetic or digit characters following the tag name cause the match to fail, and the `<` is copied through verbatim.

## 5. Performance

### 5.1 Common case (no triggers)

The quick-scan runs seven case-insensitive substring searches over the HTML input. Each search is a linear scan over the bytes. For a typical 50KB page:

- Seven scans * 50KB = 350KB of byte comparisons.
- On modern hardware, byte scanning runs at ~10GB/s (memchr-style). 350KB / 10GB/s = ~35 microseconds.
- This is negligible compared to html5ever's parse time (~0.5-2ms for a 50KB page).

**Zero allocation** in the common case: `Cow::Borrowed` is returned.

### 5.2 Uncommon case (triggers present)

When triggers are found, the transformation pass allocates a `String` with `capacity = html.len()` and does a single forward scan. The scan copies non-trigger regions in bulk (via `push_str` on slices, not character-by-character), so the amortised cost is a memcpy of the input plus O(k) tag-name comparisons at each `<` character, where k is the number of trigger tag names (7 for the cell set + 3 for br/xmp/table = 10).

For a 50KB page with 1000 `<` characters: 1000 * 10 tag-name comparisons * ~6 bytes per comparison = ~60KB of byte comparisons. Total: ~50KB memcpy + ~60KB comparisons = ~110KB of work. At 10GB/s, this is ~11 microseconds.

**One allocation** in the uncommon case: a `String` of size equal to the input.

### 5.3 Profiling recommendation

The fast-path (common case) should be profiled during implementation to confirm it does not regress the full-harness runtime. The harness currently processes 1500 pages in ~107s (~71ms/page). A per-page overhead of ~35 microseconds adds ~50ms total to the harness run — undetectable at this scale.

If profiling reveals that the multiple sequential substring searches are measurable (unlikely), they can be merged into a single-pass scan that checks all markers simultaneously.

## 7. Open Questions

### 7.1 Should the transformation pass track "actually modified?" for Cow optimisation?

The current design allocates a `String` whenever the quick-scan finds a trigger, even if the full scan determines no modification is needed (e.g., all `<td>` tags are inside `<table>` nesting). An optimisation: track whether any byte was actually changed, and return `Cow::Borrowed` if not.

**Assessment:** The extra boolean is trivial to add. But the benefit is marginal — pages with cell tags overwhelmingly have them inside tables, and the quick-scan short-circuits when no cell tags are present. The allocation cost is one `String::with_capacity(n)` + one memcpy; for a 50KB page this is ~50KB, which is small compared to html5ever's own allocations during parsing. Defer this optimisation to a follow-up if profiling shows it matters.

### 7.2 Should `<noframes>` receive the same treatment as `<xmp>`?

Phase 2F identified `<noframes>` as another raw-text element where html5ever and lxml disagree. However, `<noframes>` is semantically meaningful (fallback content for frames), and rewriting it to `<div>` would cause html5ever to parse its content as elements, potentially adding noise from frameset-era fallback text.

**Assessment:** Defer. The Phase 2F report did not trace any specific divergence record directly to `<noframes>`. If a corpus fixture surfaces that implicates `<noframes>`, it can be added as a targeted extension.

### 7.3 `<listing>` — another HTML 3.2 raw-text relic

Like `<xmp>`, `<listing>` is deprecated and treated as a raw-text element by html5ever. It is NOT mentioned in any M10 escalation report and does not appear in the corpus to my knowledge. Excluded from this HLD; can be added if needed.

### 7.4 Should the pre-processor apply only for the trafilatura pipeline, not the readability pipeline?

`Dom::parse()` is the shared entry point for BOTH the M2 readability pipeline (`extract`/`extract_with`) and the M3 trafilatura pipeline (`bare_extraction_with_cascade`). The pre-processor will run for both.

**Assessment:** This is correct. The readability pipeline also benefits from the pre-processor (Shape 2 directly affects readability scoring). The M2 parser-equivalence BLOCKER gate (`tests/parser_equivalence_gate.rs`) should remain green because the gate's corpus snapshots do not contain any of the three trigger patterns (the gate explicitly documents this at `D:\language\mdrcel\src\readability\dom.rs:25-27` — "the gate's per-snapshot guard proves contains ZERO non-whitespace stray text in table parts"). If any gate snapshot DOES contain a trigger pattern, the gate will fail and the pre-processor's impact on that snapshot must be evaluated.

### 7.5 Two-pass vs single-pass for the quick-scan + transformation

The current design does a quick-scan (pass 1) followed by a transformation (pass 2) — two passes over the input. An alternative is a single-pass design that scans and transforms simultaneously, returning `Cow::Borrowed` if no trigger was found by the end.

**Assessment:** The two-pass design is clearer and the common case (no triggers) terminates after pass 1 with zero allocation. The single-pass design would need to either (a) speculatively allocate and copy, returning `Cow::Borrowed` if nothing changed (wasting the allocation), or (b) defer allocation until the first trigger, then backfill the output string with everything before the trigger (one extra memcpy). Option (b) is a reasonable optimisation but adds implementation complexity. The two-pass design is recommended for clarity; the single-pass optimisation can be retrofitted if profiling justifies it.

## 8. Out of Scope

- **Parser replacement** (libxml2 FFI, html5ever fork, etc.) — held in reserve per M11 kickoff.
- **`<noframes>` and `<listing>` rewriting** — no corpus evidence of impact; deferred.
- **Full HTML tokenizer** — the pre-processor is a targeted tag-name scanner, not a general tokenizer. It does not track attributes, quotes, or nesting beyond the `table_depth` counter.
- **Attribute-value-aware scanning** — see section 4.1. Deferred unless a corpus fixture triggers the edge case.
- **`<script>`/`<style>`-aware scanning** — see section 4.2. Not needed because those blocks are stripped by downstream `tree_cleaning`.
- **Post-parse tree surgery** (e.g., removing spurious `<br>` nodes after parsing) — the pre-processor approach (transform before parsing) is simpler and handles all three shapes uniformly. Post-parse surgery would require per-shape tree-walking logic and would not help Shape 3 (`<xmp>` content is already a text node after parsing; no element to remove).
- **Performance optimisation** (merged Aho-Corasick scan, single-pass transformation, Cow-avoidance tracking) — deferred to follow-up if profiling shows need.
