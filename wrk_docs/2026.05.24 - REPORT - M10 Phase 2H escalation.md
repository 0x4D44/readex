### 3d. Direct lxml-vs-html5ever parser comparison


For html5ever (via `examples/m10_phase2h_endbr_probe.rs`):

```
=== HTML snippet ===

  <div> text=None tail=None
    <p> text=Some("before") tail=None
      <br> text=None tail=Some("after")
```

html5ever creates a `<br>` element with text-after-`</br>` as its `.tail`. Per
HTML5 В§13.2.6.4.7 "in body" insertion mode, this is spec-correct: any `</br>`
end tag is treated as a `<br>` start tag.

### 3e. Markdown serialiser is algorithmically correct on both sides

Python's `xmltotxt` (xml.py:354-363) вҶ’ `process_element` (xml.py:300-351) вҶ’
`sanitize` (utils.py:303-312):

- `process_element` for `<lb>` with NO text/children but WITH tail:
  - line 309 `if not element.text and not element.tail:` вҖ” false (HAS tail);
  - line 341 `if element.tag in NEWLINE_ELEMS and not ancestor::cell:` вҖ” true;
  - line 343 appends `"\nвҗӨ\n" if include_formatting and element.tag != 'row' else "\n"`.
- `sanitize` (line 310): `'\n'.join(filter(None, ...)).replace('вҗӨ', '')`.

For markdown, `<lb>` produces `\nвҗӨ\nTAIL` which after sanitize becomes
`\n\nTAIL` (blank line). For txt, `<lb>` produces `\nTAIL` (single newline).

**This algorithm is what mdrcel does too** вҖ” verified by inspecting
`src/trafilatura/xml.rs` and `src/output.rs` in prior phases. The
serialisation per-`<lb>` is byte-equivalent.

The divergence does NOT come from the serialiser. It comes from the fact
that mdrcel's TREE has an extra `<lb>` element that Python's tree does
not have. mdrcel correctly serialises an `<lb>` that shouldn't be there.

## 4. В§7 anti-inversion: is there any Python filter that drops spurious `<br>`?

Searched all of `benchmark/oracles/trafilatura/.venv/Lib/site-packages/trafilatura/`
for any code path that could drop or merge a `<br>`/`<lb>` produced by an
upstream stray `</br>`:

- `settings.py` вҖ” `<br>` does NOT appear in `MANUALLY_CLEANED`,
  `MANUALLY_STRIPPED`, `OVERALL_DISCARD_XPATH`, or any pruning list. The only
  `br` substring is `abbr` (settings.py:408) and a comment about `wbr`
  (settings.py:430).
- `htmlprocessing.py:328-330` `convert_line_breaks`:
  ```python
  def convert_line_breaks(elem: HtmlElement) -> None:
      "Convert <br> and <hr> to <lb>"
      elem.tag = "lb"
  ```
  Pure rename, no filtering. mdrcel mirrors this exactly.
- `htmlprocessing.py:248-249` (`handle_textnode`):
  ```python
  if comments_fix and elem.tag == "lb":
      elem.tag = "p"
  ```
  Only renames `<lb>` to `<p>` in a `comments_fix` branch; does not drop.
- `readability_lxml.py:323`:
  ```python
  if child.tag == "br":
  ```
  This is inside the rescue-pass mdrcel ALREADY ports
  (`readability_fork.rs:830-918`, Phase 2C). It drops `<br>` children of
  `<div>` only during the rescue-pass tree restructure. The stray-`</br>`
  victim in rep#0 is inside a `<p>` (originally `<div style="line-height:...">`
  which the cleanup retags to `<p>`), so this `<div>/<br>` xpath doesn't
  match it.
- `readability_lxml.py:488`:
  ```python
  nodes.update(node.getparent() for node in doc.xpath(".//div/br"))
  ```
  Same `<div>/<br>` pattern as above. Same non-match.
- `xml.py:211-214`:
  ```python
  # convert <lb/> when child of <div> to <p>
  for elem in xmldoc.findall(".//text/body//div/lb"):
      if elem.tail and elem.tail.strip():
          elem.tag, elem.text, elem.tail = 'p', elem.tail, None
  ```
  Only fires when `<lb>` is a CHILD of `<div>` (in the final TEI tree).
  Doesn't affect markdown/txt paths; mdrcel mirrors this in `src/trafilatura/xml.rs`.

**No Python code path filters or drops a stray-`</br>`-derived `<br>`/`<lb>`
element**. Python achieves byte parity by relying on lxml's HTML parser having
dropped the `</br>` *before* the tree is built. By the time Python's pipeline
runs, the spurious element does not exist to be filtered.

mdrcel cannot mirror this without either:
1. Replacing html5ever with a parser that drops stray `</br>` end-tags вҖ”
   **out of scope** for this milestone (M11 candidate per Phase 2D В§11).
2. Adding an mdrcel-only filter to drop `<br>` elements that look like they
   came from a stray end-tag вҖ” **В§7-deviating** (no Python equivalent; would
   require Arthur's ratification).
3. Pre-processing the HTML byte stream to strip stray `</br>` вҖ” **В§7-deviating**.

+1.5% KPI. The estimate is hedged because the parser difference cascades
non-locally into selection-phase shifts that any local filter wouldn't catch.
