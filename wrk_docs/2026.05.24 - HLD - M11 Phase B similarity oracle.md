## §1 Goal & Non-Goals

### Goal

Add a **content-similarity oracle** alongside the existing byte-equality metric
in the M9 fuzz harness (`D:\language\mdrcel\tests\fuzz_diff.rs`). The similarity
oracle computes two metrics — character-level Sorensen-Dice on bigrams (primary)
and token-set Jaccard (secondary) — on every divergent page pair, classifies each
divergence into a severity bucket (near-equivalent / content-similar / divergent),
and publishes per-format and aggregate similarity KPIs in the summary table.

This lets us distinguish "different-but-equivalent" extractions (e.g. both
extractors picked valid main content, just different valid paragraphs) from
"truly divergent" ones (e.g. empty-vs-nonempty, fundamentally wrong content).
The M11 journal (`D:\language\mdrcel\wrk_journals\2026.05.24 - JRN - M11 Metric Pivot and Parser Pre-Processor.md`)
documents the strategic rationale: ~58% of the current ~15% residue is
selection-phase divergence that byte-equality scores as failure but is
functionally successful extraction.

### Non-Goals

- **Replace byte-equality as primary KPI.** Byte-equality remains the top-line
  fidelity metric. Similarity is an ALONGSIDE metric that provides depth, not a
  replacement.
- **Semantic similarity or ML-based scoring.** No embeddings, no model calls, no
  external services. All metrics are pure arithmetic on character/token multisets.
- **Modify the 51-fixture curated gates.** The six `trafilatura_*_gate.rs` files
  and the `trafilatura_extract_content_gate.rs` are untouched.
- **Modify library code.** All changes are harness-only (`tests/fuzz_diff.rs`
  and potentially a new `tests/similarity.rs` module). No changes to `src/`.
- **Add external dependencies.** Both metrics are implementable in <50 LOC of
  pure Rust each. No new crate.
- **Change the working-slice / held-out partition.** The `WORKING_SLICE_SIZE =
  1000` constant (line 93 of `fuzz_diff.rs`) is frozen.

## §2 Metric Choice

### Primary: Character-level Sorensen-Dice on bigrams (multiset)

**Formula:**

```
dice(A, B) = 2 * |A ∩ B|_multiset / (|A| + |B|)
```

where A and B are character bigram multisets (each bigram counted with
multiplicity; intersection takes the minimum count per bigram).

**Properties:**

- **O(n) time, O(unique_bigrams) space.** Critical because docs can be 50KB+
  and we run on 1500 pages x 3 formats = 4500 comparisons per harness run.
- **Tracks normalized Levenshtein similarity almost exactly** (r ~= 0.99 across
  all divergence classes in the empirical probe — see §3).
- **Correctly scores whitespace-only diffs as 0.99+** — the bigrams change only
  at the whitespace boundary, so the score is near 1.0.
- **Correctly scores selection-phase overlap as 0.82-0.98** — shared paragraphs
  produce shared bigrams; omitted paragraphs reduce the score proportionally.
- **Correctly scores empty-vs-nonempty as 0.0** — one side has zero bigrams.
- **Deterministic, reproducible, interpretable.** No randomness, no
  floating-point instability beyond standard IEEE 754 division.

### Secondary: Token-set Jaccard (whitespace-tokenized)

**Formula:**

```
jaccard(A, B) = |tokens(A) ∩ tokens(B)| / |tokens(A) ∪ tokens(B)|
```

where `tokens(s)` = set of `\w+` regex matches over `s`, lowercased. SET
semantics (not multiset) — each token counted once.

**Properties:**

- **Even cheaper than Dice** — tokenization is a single pass; set
  intersection/union on a HashSet.
- **Good "any shared content?" signal** — Jaccard 0.0 means zero word overlap.
- **Loses granularity on whitespace-only diffs** — rounds to 1.0 because the
  token SET is identical when only whitespace changes. This is WHY it is
  secondary, not primary.

### Rejected alternatives

**Raw Levenshtein similarity** (`1 - edit_distance / max(len_a, len_b)`):

- O(n*m) time. For a 50KB doc pair that is 2.5 billion cells — impractical
  for a 4500-comparison harness run.
- The probe shows Dice tracks it at r ~= 0.99 anyway. Dice gives the same
  signal at O(n) cost.

**Longest common substring ratio** (`lcsubstr_ratio`):

- Empirically, lcsubstr_ratio diverges significantly from Levenshtein similarity
  on whitespace-only diffs. Example: sha `a2ebaf77` markdown has
  `lev_sim = 0.9941` but `lcsubstr_ratio = 0.5718` — the longest common
  substring misses the repeated structure. Dice scores this correctly at 0.9941.
- lcsubstr_ratio is O(n*m) in the naive implementation; O(n log n) with suffix
  arrays but with high constant factors.
- The probe data (§3) shows lcsubstr_ratio systematically underscores
  whitespace-only and extra-element divergences where the content is actually
  nearly identical. This makes it a poor oracle for the "different-but-equivalent"
  question Phase B is designed to answer.

**ROUGE-L F1:**

- Competitive with Dice on the probe data, but requires LCS computation
  (O(n*m) or O(n log n) with patience sorting).
- Does not add discriminative value beyond Dice (the probe shows near-identical
  rankings).
- More complex to implement correctly (precision/recall/F1 weighting).

### Threshold calibration

The thresholds were derived from the probe data by identifying natural gaps in the
Dice score distribution:

- **Gap at 0.95:** All whitespace-only diffs score above 0.99; all
  text-content-with-minor-changes score 0.92-0.99. The 0.95 boundary cleanly
  separates "near-equivalent" (trivially different) from "content-similar"
  (meaningfully different but substantially overlapping).
- **Gap at 0.80:** Selection-phase with moderate overlap scores 0.82-0.93; truly
  divergent or disjoint content scores below 0.21. The 0.80 boundary cleanly
  separates "content-similar" (valid extraction with partial overlap) from
  "divergent" (fundamentally different or broken).
- **No probe data falls in the 0.21-0.80 gap.** This may populate with
  corpus-scale data, but the gap gives confidence the boundaries are not
  splitting natural clusters.

#### `fn dice_bigram_similarity(a: &str, b: &str) -> f64`

```rust
/// Character-level Sorensen-Dice coefficient on bigram multisets.
///
/// Formula: 2 * |A ∩ B|_multiset / (|A| + |B|)
/// where A, B are character bigram multisets and intersection takes
/// min(count_a, count_b) per bigram.
///
/// Returns 1.0 for identical strings (including both empty).
/// Returns 0.0 when one is empty and the other is not.
/// Returns 0.0 for two single-char strings (no bigrams).
fn dice_bigram_similarity(a: &str, b: &str) -> f64 {
    // Edge cases: both empty => identical => 1.0
    // One empty, other not => 0.0
    // Both single-char => 0 bigrams each => convention: if a == b, 1.0; else 0.0
    if a == b { return 1.0; }
    if a.is_empty() || b.is_empty() { return 0.0; }

    let bigrams_a = char_bigram_counts(a);
    let bigrams_b = char_bigram_counts(b);

    let size_a: usize = bigrams_a.values().sum();
    let size_b: usize = bigrams_b.values().sum();

    if size_a == 0 && size_b == 0 {
        // Both strings are single chars but not equal (handled by a == b above)
        return 0.0;
    }

    let mut intersection = 0usize;
    for (bigram, &count_a) in &bigrams_a {
        if let Some(&count_b) = bigrams_b.get(bigram) {
            intersection += count_a.min(count_b);
        }
    }

    (2.0 * intersection as f64) / (size_a + size_b) as f64
}

fn char_bigram_counts(s: &str) -> HashMap<(char, char), usize> {
    let mut map = HashMap::new();
    let chars: Vec<char> = s.chars().collect();
    for window in chars.windows(2) {
        *map.entry((window[0], window[1])).or_insert(0) += 1;
    }
    map
}
```

**Complexity:** O(n) time for building the bigram map (one pass over chars);
O(unique_bigrams) space. The intersection loop is O(min(|A_unique|, |B_unique|)).

#### `fn jaccard_token_similarity(a: &str, b: &str) -> f64`

```rust
/// Token-set Jaccard similarity.
///
/// Tokens: \w+ matches, lowercased. SET semantics (not multiset).
/// Returns 1.0 when both are empty (no tokens => identical empty sets).
/// Returns 0.0 when one has tokens and the other has none.
fn jaccard_token_similarity(a: &str, b: &str) -> f64 {
    let tokens_a: HashSet<String> = tokenize(a);
    let tokens_b: HashSet<String> = tokenize(b);

    if tokens_a.is_empty() && tokens_b.is_empty() {
        return 1.0;
    }

    let intersection = tokens_a.intersection(&tokens_b).count();
    let union = tokens_a.union(&tokens_b).count();

    intersection as f64 / union as f64
}

fn tokenize(s: &str) -> HashSet<String> {
    // \w+ equivalent: sequences of [a-zA-Z0-9_]
    let mut set = HashSet::new();
    let mut current = String::new();
    for ch in s.chars() {
        if ch.is_alphanumeric() || ch == '_' {
            current.push(ch.to_ascii_lowercase());
        } else if !current.is_empty() {
            set.insert(std::mem::take(&mut current));
        }
    }
    if !current.is_empty() {
        set.insert(current);
    }
    set
}
```

Note: `to_ascii_lowercase()` is used rather than full Unicode lowercasing. This
matches the `\w+` intent (the probe data uses ASCII-dominated content) and avoids
pulling in a Unicode case-folding dependency. If non-ASCII tokens matter in the
future, this can be upgraded to `to_lowercase()` without changing the metric
semantics.

#### `fn strip_xml_tags(s: &str) -> String`

```rust
/// Strip XML/HTML tags from a string for similarity computation.
/// Removes everything between '<' and '>' inclusive.
/// Does NOT handle CDATA, comments, or processing instructions specially —
/// this is a cheap heuristic for the similarity oracle, not a parser.
fn strip_xml_tags(s: &str) -> String {
    let mut out = String::with_capacity(s.len());
    let mut in_tag = false;
    for ch in s.chars() {
        if ch == '<' {
            in_tag = true;
        } else if ch == '>' {
            in_tag = false;
        } else if !in_tag {
            out.push(ch);
        }
    }
    out
}
```

Used for xml format only: the probe data (§3) shows tag-stripped similarity
is more discriminating (e.g. `a2ebaf77 xml raw: 0.9867` vs
`xml-text-only: 0.9903` — the tag noise dilutes the content signal).

The `dice_sum_all` accumulator counts byte-equal pages as 1.0 (they are

(which would be wasteful — we already know they are identical).

### 4.5 Changes to `process_format()` (line 740 of `fuzz_diff.rs`)

The similarity computation happens BETWEEN the byte-equality fast path (line 776)
and the `if record_divergence` gate (line 807). This ensures:

1. Byte-equal pages skip similarity computation (fast path preserved).
2. Similarity is computed for ALL divergent pages regardless of slice membership
   (both working and held-out), so the KPI table has similarity data for the
   held-out slice.
3. The `record_divergence` gate only controls whether the divergence is
   WRITTEN to JSONL and BUFFERED for grouping — not whether similarity is
   computed.

Sketch of the changed flow in `process_format()`:

```rust
// Byte-equal? Fast path.

    // ... existing fields ...
```

**Key design choice:** the similarity computation is placed BEFORE the
`if record_divergence` gate. This means:

- Working-slice divergences: similarity computed, recorded in JSONL, accumulated

- Held-out-slice divergences: similarity computed, accumulated in
  `PerFormatStats`, but NOT recorded in JSONL (preserving the "look but don't
  peek" invariant for triage).

This is necessary for the KPI table to show similarity metrics for the held-out
slice.

## §5 Threshold Proposal

Three thresholds with empirically calibrated names:

| Dice range | Label | Description | Empirical calibration |
|-----------|-------|-------------|----------------------|
| `= 1.0` | `byte-equal` | Existing fast path; no similarity computation | — |
| `>= 0.95` | `near-equivalent` | Whitespace-only or minor structural differences; functionally identical extraction | a2ebaf77 md 0.9941, ff9757c4 md 0.9999, dd860c78 txt 0.9933, ba3180f9 xml 0.9801 (note: 0.9801 is just above but see below) |
| `0.80 .. 0.95` | `content-similar` | Different paragraphs selected but substantial content overlap; both extractions are valid | 52cdce08 md 0.8602, 0a19f71f md 0.9302 |
| `< 0.80` | `divergent` | Fundamentally different output, empty-vs-nonempty, or broken extraction | df285f53 all 0.0, ba3180f9 txt/md 0.0, 52cdce08 txt 0.2083 |

**Constants in code (hardcoded):**

```rust
const DICE_NEAR_EQUIVALENT: f64 = 0.95;
const DICE_CONTENT_SIMILAR: f64 = 0.80;
```

These are hardcoded, not configurable. Rationale: the thresholds are empirically
calibrated against the probe data and tightly coupled to the interpretation in
the summary table. Making them configurable would invite silent redefinition
without re-calibration. If future data suggests different boundaries, the
constants are trivially updatable (two lines of code) and the change should be
accompanied by a re-calibration note in the journal.

## §7 Open Questions

### 7.1 Should the summary table show mean Dice over ALL pages or only DIVERGENT pages?

**Team lead lean:** Both.

**Design agent recommendation:** Agree. Both are useful:

- **Overall mean** (byte-equal scored as 1.0) answers "how similar is our output
  to Python's, on average?" This is the headline number for stakeholder
  reporting.
- **Divergent mean** answers "among the pages we get wrong by byte-equality, how
  wrong are they?" This is the actionable number for triage.

Implementation cost is trivial — two accumulators instead of one. **Recommend:
implement both.**

### 7.2 Should xml similarity use tag-stripped text or raw?

**Team lead lean:** Tag-stripped.

**Design agent recommendation:** Agree. The probe data consistently shows
tag-stripped gives higher (and more intuitive) similarity scores for xml:

- a2ebaf77 xml: raw 0.9867 vs text-only 0.9903
- 0a19f71f xml: raw 0.9269 vs text-only 0.9370
- 52cdce08 xml: raw 0.8255 vs text-only 0.8848

The raw scores are diluted by tag markup differences that are structurally
irrelevant to content similarity (e.g. `<p>` vs `<div>` wrapping the same text).
Tag-stripping removes this noise. **Recommend: tag-stripped for xml.**

### 7.3 Should we compute similarity on the held-out slice?

**Team lead lean:** Yes.

**Design agent recommendation:** Agree. The held-out slice IS the KPI substrate.
Without similarity data on the held-out slice, the new metric is
triage-only — useless for the headline KPI. The "look but don't peek" invariant
(`record_divergence = false`) is about not TARGETING held-out pages for triage,
not about not MEASURING them. The similarity computation is a read-only
measurement; it does not influence which pages are triaged.
**Recommend: compute on held-out, inline, for the KPI table.**

### 7.4 Should threshold boundaries be configurable or hardcoded?

**Team lead lean:** Hardcoded constants.

**Design agent recommendation:** Agree. Configurable thresholds invite silent
redefinition without re-calibration. The thresholds are tightly coupled to the
empirical probe data and the human-readable labels. A future phase can promote
them to configurable if needed — but the promotion should be accompanied by a
calibration protocol, not just an `Option<f64>`. **Recommend: hardcoded
constants, revisable by future work.**

### 7.5 Should `char_bigram_counts` use `HashMap` or `BTreeMap`?

Not raised in the brief, but worth flagging. `HashMap` is the natural choice
(O(1) amortised lookup); `BTreeMap` gives deterministic iteration order but
we never iterate the map in a user-visible way (only the intersection sum
matters). **Recommend: `HashMap` for performance.** The `std::collections::HashMap`
is already available (no import needed beyond the prelude in test code).

### 7.6 Edge case: panic/error divergences

When `force_shape` is `Some(ShapeClass::Panic)` or `Some(ShapeClass::MdrcelError)`,
the `mdrcel_text` is the error/panic message string, not extraction output.
Computing Dice similarity between an error message and the oracle's extracted
text is meaningless.

**Recommendation:** For panic/error divergences, set `dice_similarity = 0.0`
and `jaccard_similarity = 0.0` unconditionally (do not compute). These are
unambiguously "divergent" — the extractor failed. Accumulate them in the
`truly_divergent` bucket. This avoids the possibility of a panic message
accidentally sharing bigrams with the oracle text and producing a misleading
non-zero score.

### 7.7 The `clone()` cost in process_format for xml tag stripping

The sketch in §4.5 shows `(mdrcel_text.clone(), oracle_text_n.clone())` for the
non-xml path. Since these strings can be 50KB+, the clone is not free. Two
alternatives:

1. Compute similarity on references directly (no clone needed for non-xml path;
   only xml needs a new allocation for the stripped string). The function
   signatures take `&str`, so passing `&mdrcel_text` directly works.
2. Accept the clone (two 50KB allocations per divergent page; at ~228 divergent
   pages per format, that is ~23MB total — negligible for a test harness).

**Recommendation:** option 1 — avoid the clone. Pass `&mdrcel_text` and
`&oracle_text_n` directly for non-xml; only allocate for xml via
`strip_xml_tags()`. The function signatures already take `&str`.

## §8 Out of Scope

The following are explicitly out of scope for Phase B:

- **Semantic similarity.** No word embeddings, no sentence transformers, no API
  calls to language models.
- **ML-based scoring.** No trained classifiers, no supervised models, no feature
  engineering beyond the two formulaic metrics.
- **Replacing byte-equality as primary KPI.** Byte-equality remains the top-line
  metric. Similarity is a secondary, alongside metric.
- **Changes to the 51-fixture curated gates.** The six `trafilatura_*_gate.rs`
  files and `trafilatura_extract_content_gate.rs` are frozen.
- **Changes to library code (`src/`).** All modifications are in test harness
  files under `tests/`.
- **New external dependencies.** No new crate in `Cargo.toml` (beyond the
  version bump).
- **Modifying the working-slice / held-out partition.** `WORKING_SLICE_SIZE`
  remains 1000.
- **Auto-classification of divergences.** The similarity score is a measurement;
  it does not automatically reclassify `ShapeClass` or change the shape hash.
  Reclassification is a triage decision, not a harness decision.
- **Phase A work.** The parser pre-processor is a separate phase with its own
  HLD. Phase B provides the metric infrastructure Phase A will be measured
  against, but Phase B does not implement any parser changes.
