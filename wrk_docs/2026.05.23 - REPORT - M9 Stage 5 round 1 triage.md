## Root cause A — "trailing-space before inline tag"

**Manifest:** mdrcel emits `text. <lb/>` (or `text. </p>`) where Python emits `text.<lb/>` / `text.</p>`.

**Affected groups:**
- `xml / text-content / lb` — **16 of 18 members** match this signature.
- `xml / text-content / p` — **~20 of 48 members** match this signature (the remainder split
  between soft-hyphen preservation and CP1252 mojibake — distinct sub-causes).
- **Total: ≈ 36 records, ≈ 4.6% of the working slice.**

**Vendored Python source path:**
- `htmlprocessing.py:222–265` — `handle_textnode`. With `preserve_spaces=False` (default), line
  253 trims `elem.text` via `trim(elem.text) or None`. The `lb`-bypass at lines 235–241 trims
  ONLY the lb's `.tail`, not the parent's `.text`.
- `utils.py:282–300` — `line_processing`. The `trailing_space` re-attach (lines 296–299) gates on
  `line[0].isspace()` / `line[-1].isspace()`, so the trailing whitespace is removed by `trim`
  and only re-attached if the caller explicitly asked for it.
- `xml.py:159–175` — `control_xml_output` → `sanitize_tree` plumbing.

**mdrcel mirror locations (both look algorithmically faithful when read side-by-side):**
- `src/trafilatura/cleaning.rs:1318–1410` — `handle_textnode` + `handle_textnode_finish`. Mirrors
  the Python `lb`-bypass + trim discipline byte-for-byte.
- `src/trafilatura/output.rs:805–957` — `sanitize_tree` / `sanitize` / `line_processing`. Mirrors
  the Python `trailing_space` re-attach gate.

**Anti-inversion verdict:** the *algorithms* match Python; the *behaviour* doesn't. The bug
therefore lives EITHER upstream (in `handle_paragraphs`'s text-copy via
`main_extractor.rs:712–864`, or earlier in tree-build) OR in the DOM-text-storage layer
itself — recall the rcdom tail-trap documented in `Cargo.toml` lines 440-442 / 534-549 (append
/ insert / extend move elements WITH their tail, and a tight clone/mutate/drop loop can drain
descendants while `Rc` handles are held).

**Confidence:** MEDIUM. The pattern is rock-solid (16/18 + ~20/48 reps with the identical
extra-space-before-tag signature). The exact fix locus is NOT confirmed. The triage agent
explicitly stated: "Do not blind-fix. Before any patch, run mdrcel on one of the 18 `xml/lb`
corpus pages, dump the `<p>` element's `.text` after `handle_paragraphs` and again after
`sanitize_tree`, and compare against Python's lxml tree at the same two points."

## Root cause B — "cell / head text-vs-tail allocation"

**Manifest:** oracle emits `Forest |\n` (where `|` is on the same line as the cell text); mdrcel
emits `Forest\n|\n` (cell text and cell-end pipe on separate lines).

**Affected groups:**
- `markdown / whitespace / <none>` — Reps 1 + 3 (≈ 10 of 21 members).
- `txt / whitespace / <none>` — Reps 1 + 2 (≈ 7 of 9 members).
- **Total: ≈ 17 records, ≈ 2.2% of working slice.**

**Vendored Python source path:**
- `xml.py:288–293` — `replace_element_text` cell branch. When the cell has a `<p>`-child, line
  290 appends `" "` to the cell text BEFORE the cell-end `" | "` emit; the leaf-cell branch
  (lines 291–293) does not.
- `xml.py:341–345` — head emission (`\n␤\n` separator) and cell-end emit (`" | "`).

**mdrcel mirror location:** `src/trafilatura/output.rs:585–603` mirrors both branches. The
algorithms look equivalent; the divergence is likely **upstream of xmltotxt** — mdrcel's
tree probably has different cell structure (each `<td>` in the source becoming a `<cell>`
with a `<p>`-child in Python vs a leaf `<cell>` in mdrcel, or vice versa). This is a
`convert_tags` / table-handling layer concern.

**Confidence:** MEDIUM. Similar story to root cause A — the algorithms match but the
tree-build upstream differs.
