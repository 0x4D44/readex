## 4. Diagnosis — outcome (B')

### 4.1 What both pipelines do at the leak boundary

| Boundary                          | Python (lxml)              | mdrcel (html5ever)         |
|-----------------------------------|----------------------------|----------------------------|
| OVERALL_DISCARD before readability | **NO** (Balanced mode)     | **NO** (Balanced mode)     |
| readability_lxml `remove_unlikely_candidates` | runs `unlikelyCandidatesRe` | runs same regex (port verified) |
| readability scoring               | `score_paragraphs` over `<p>`/`<pre>`/`<td>` descendants | **identical port** |

**The PIPELINES are identical at the leak boundary.** mdrcel's port mirrors Python's `compare_extraction` and `try_readability` semantics verbatim. There is **no missing filter in mdrcel** — the filter Phase 2F flagged (`OVERALL_DISCARD` on body) is NOT applied at the readability boundary in EITHER pipeline in default Balanced mode.

### 4.2 Why outputs differ then?

**The trees fed into readability scoring differ in shape, because html5ever and lxml disagree about how to parse the SEO-spam HTML (Phase 2F mechanism (i)/(ii)).** Specifically on fixture 523a23495e14:

- **lxml's tree (Python):** spam div has lots of nested elements (the `<td>`, `<tr>`, etc. inside it are preserved per lxml's permissive parser); legitimate `<section class="container">` ALSO has many elements (including elements lxml foster-parented out of the spam div into surrounding context). The scoring math (sum of paragraph text length + comma count, per `readability_lxml.py:228-249`) gives `<section class="container">` a higher score than the spam div.
- **html5ever's tree (mdrcel):** spam div has nested EMPTY elements (table-cell tokens dropped per HTML5 spec "in body" insertion mode), BUT the spam div contains a `<xmp id="c8na4">` with literal-text inner HTML that html5ever preserves as a TEXT node. The legitimate `<section class="container">` has many elements stripped (because html5ever doesn't foster-parent the spam tags out). Net effect: the spam div's text content (incl. the `<xmp>` raw-text leak) is LONGER than the section's, and **the spam div wins scoring**.

**This is parser-permissiveness divergence cascading through identical scoring math to a different best candidate.** Phase 2F mechanism (i) (`<td>`/`<tr>`/`<tbody>` outside `<table>` ignored by html5ever) AND mechanism (ii) (`<xmp>` raw-text content kept as text by html5ever) BOTH contribute.

### 4.3 Why this is outcome (B') not (B)

- (B) would be "Python doesn't apply OVERALL_DISCARD at this boundary; spam div survives Python's pipeline too but a different filter removes it." That's NOT what's happening — Python's spam div doesn't get removed, it just doesn't get **picked** by `select_best_candidate`. There is no equivalent "filter" applied; Python's clean output is a **byproduct of the scoring math operating on lxml's tree shape**.
- (B') is "Python's parser-permissiveness produces a different tree shape that causes downstream scoring to inadvertently reject the spam div." That's exactly this case.

Both are forms of "the system doesn't have an explicit filter; byte parity is coincidental on Python's side." Per the Phase 2G brief §5 (anti-inversion / §7), mdrcel must not add a filter that Python doesn't have.

## 5. Why §7 forbids the apparent fix

### 5.1 Apparent "fixes" that would close the leak

1. **Add `prune_unwanted_nodes(backup_html_root, OVERALL_DISCARD_XPATH, false)` in default mode before `try_readability`** — closes the leak on 523a23495e14 by removing the spam div BEFORE readability scoring sees it.
2. **Pre-process HTML bytes to strip `<xmp>`/`<noframes>` blocks before html5ever sees them** — closes Phase 2F mechanism (ii) by preventing the raw-text leakage.
3. **Add a `display:none` / `pl_css_ganrao` filter to mdrcel's `remove_unlikely_candidates`** — closes the leak by making readability skip the spam div as a candidate.

### 5.2 Why each violates §7

- **Fix 1** would diverge from Python's `compare_extraction` lines 53-55 (the OVERALL_DISCARD pre-pass is precision-only). Asymmetric trust §7: "every Python claim cites file:line; the port mirrors Python." Adding the pre-pass in Balanced mode would mean mdrcel does something Python doesn't.
- **Fix 2** is byte-level pre-processing that Python doesn't do (Python's `load_html` calls `html.fromstring` and trusts lxml's permissive parse). The Phase 2F report already flagged this as §7-violating.
- **Fix 3** would extend `unlikelyCandidatesRe` (or add a new pass) with patterns Python's `readability_lxml.py:63-66` does not contain. Direct divergence.

**All three are §7-deviating.** Per CLAUDE.md "asymmetric trust" + the Phase 2G brief: the team lead in autonomous mode cannot ratify a §7 deviation overnight.

### Question A: "Authorize §7 deviation for `OVERALL_DISCARD` pre-pass in `compare_extraction` Balanced mode"

> Python's `compare_extraction` (`external.py:54-55`) only runs `prune_unwanted_nodes(backup_tree, OVERALL_DISCARD_XPATH)` when `options.focus == "precision"`. In default Balanced mode, the readability arm sees the un-pruned backup tree.
>

>
> Should mdrcel deviate from Python and **always** run OVERALL_DISCARD before `try_readability`? This would close the leak on `523a23495e14` and likely ~14 of the ~25 SEO-spam working-slice records (the Phase 2F bounded-fix estimate), but it is a behavioural deviation from the oracle that may regress fixtures where Python's "keep more" behaviour was load-bearing (e.g. articles with `display:none` decorations whose containers are real content).

### Question B: "Authorize §7 deviation for HTML byte pre-processing of `<xmp>`/`<noframes>`/`<style>` blocks"

> Python's `load_html` calls `html.fromstring(raw_bytes)` and trusts lxml's permissive parser to handle raw-text elements. lxml parses inner content as element children; html5ever (per HTML5 spec) parses inner content as TEXT. The text-as-text leakage produces visible literal `<td>`/`<rt>` markers in mdrcel output that Python's tree doesn't have.
>
> Should mdrcel pre-process the HTML bytes (regex-strip or rewrite `<xmp>`/`<noframes>`/`<style>` contents) before handing to html5ever? This is fragile (regex over HTML), but it's the only way to close Phase 2F mechanism (ii) without forking html5ever.
